# Metadata as code — OKFの仕様とサンプルから論じる

## 「Metadata as code」とは何か

ひとことで言えば、**カタログのメタデータを専用のレジストリ／サービスに預けるのではなく、ソースコードと同じようにプレーンテキストのファイルとして書き、Gitで管理する考え方**です。OKFの仕様書自身も `SPEC.md` の §10 で、自らを「'Metadata as code' approaches that store catalog metadata alongside source code rather than in a separate registry」の系譜に位置づけています。

ここで効いているのは "as code" という比喩で、これは Infrastructure as Code（Terraform等）の発想をメタデータに持ち込んだものです。本質は「**何を真実の源（source of truth）にするか**」の選択にあります。

| | 従来のカタログ（registry型） | Metadata as code |
|---|---|---|
| 真実の源 | サービス内のDB（Collibra, Dataplex等） | リポジトリ内のファイル |
| アクセス手段 | 専用API/SDK/クエリ言語 | `cat` / `git clone` |
| 変更履歴 | サービス独自の監査ログ | git log / blame / diff |
| レビュー | 管理画面のワークフロー | Pull Request |
| 可搬性 | ベンダーロックイン | ディレクトリごと持ち出せる |

## OKFがそれをどう体現しているか

OKFは "metadata as code" の一実装であり、サンプルを見ると抽象論ではなく具体的な設計判断として現れています。

**1. ファイル＝概念、パス＝ID**
Concept ID は「ファイルパスから `.md` を除いたもの」と定義されます。`tables/events_.md` がそのまま `tables/events_` というIDになる。識別子をDBの主キーではなくファイルシステムが供給している。これがまさに "as code" の核心で、コードでクラスをファイル名で見つけるのと同じ発想です。

**2. 構造化メタデータ（frontmatter）と非構造化本文（markdown）の意図的な分離**
`references/metrics/avg_pageviews.md` を見ると、上部のYAML（`type: Reference`, `tags: [metric]`, `timestamp`）が「機械が索引・フィルタする少数のフィールド」、下のSQLと散文が「人とLLMが実際に読む中身」。クエリしたい列だけを構造化し、残りは本文に逃がす設計です。RDBスキーマに全部押し込まないのは、まさにコードのコメントとシグネチャの関係に似ています。

**3. リレーションを「リンク」で表す**
`tables/events_.md` の joins は `/references/joins/events___ads_clickstats.md` への普通のmarkdownリンクで表現されます。仕様はリンクの「種類（parent/child, joins-with…）」を散文に委ね、リンク自体は無型の有向辺だとする。外部キー制約のような厳格な型付けを捨て、ソースコードの `import` のような緩い参照にしている。だから壊れたリンクも「未執筆の知識」として許容されます。

**4. 寛容な消費モデル（conformance）**
必須は実質「frontmatterがパースでき、`type` が非空」だけ。未知の `type`、未知のキー、欠けたindex、壊れたリンク——どれも拒否してはならない。これは「メタデータがコードのように**漸進的に・部分的に・複数の手（人とエージェント）で**育つ」ことを前提にした割り切りです。

## 論点：何が得で、何が代償か

**得られるもの（OKFの主張）**

- **協働の同一化**：知識のキュレーションが「普通のソフトウェア開発作業」になる。PRレビュー、blame、diffがそのまま使える。
- **ロックインの消滅**：バンドルはただのディレクトリ。tarball、任意のリポジトリ、Obsidian/MkDocsで開ける。
- **人・エージェント両対応**：SDKもクエリ言語も挟まらない。エンジニアは `cat`、LLMはそのままcontextに入れる。実際このリポジトリはエージェント（producer）とビジュアライザ（consumer）の両端でそれを実証している。

**代償・限界（仕様から読み取れる弱点）**

- **整合性保証が弱い**：型レジストリも参照整合性もない（broken link許容）。「壊れていないこと」を保証したければ別途リンタが要る。registry型が標準で持つ制約をトレードオフした形。
- **クエリ性能はファイルスキャン頼み**：タグ別ビューすら「消費時にfrontmatterを走査して合成せよ」とされる。大規模カタログでは索引層を自前で被せる必要がある。
- **スケールでのGit運用問題**：数万概念になればモノレポ同様の課題（巨大diff、マージ衝突、生成物のノイズ）を引き継ぐ。
- **語彙の自由＝相互運用の脆さ**：`type` が中央登録されないので、組織をまたぐと `BigQuery Table` か `Table` か揺れる。可搬性は高いが意味的相互運用は producer の規律頼み。

## まとめ

「Metadata as code」とは、**メタデータの真実の源をサービスからリポジトリへ移し、ソフトウェア開発のツールチェーン（Git・PR・diff・プレーンテキスト）をそのまま知識管理に転用する設計思想**です。OKFはその思想を「YAML frontmatter付きmarkdownのディレクトリ」という最小形式に固めた仕様で、サンプル（`bundles/ga4/` の実バンドル）を見ると、**厳格なスキーマ保証を犠牲にして、可読性・可搬性・人とエージェントの協働可能性を最大化する**という一貫したトレードオフが貫かれています。中央集権的カタログの「正しさの保証」と、コード的アプローチの「自由と可搬性」のどちらを取るか——OKFは明確に後者に賭けた仕様だ、というのが評価の軸になります。
