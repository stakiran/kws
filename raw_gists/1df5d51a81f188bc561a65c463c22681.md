# Tududi に関する Q&A

## Q1. これは何？

**A.**

このディレクトリは **Tududi** という、セルフホスト型のタスク管理システムのソースコードです。

主な特徴:
- **階層構造**: Areas > Projects > Tasks > Subtasks
- **技術スタック**: React 18 + TypeScript（フロント）、Express + Sequelize + SQLite（バック）
- **機能**: 繰り返しタスク、24言語対応、プロジェクト共有、REST API、Telegram連携、タグシステム

開発者向けドキュメントは `CLAUDE.md` と `docs/` 配下に整理されており、ユーザー向けは `README.md` にあります。現在のブランチは `main` でクリーンな状態です。

## Q2. タスク管理としてどんな機能と思想を持つ？

**A.**

### 思想 — 「ユーザーに抗わないシステム」

作者のブログタイトル "Designing a Life Management System That Doesn't Fight Back" が示す通り、**厳格な構造より流れ（フロー）を優先**するのが根幹。タスクを"埋める"作業に追われるのではなく、自然に手が動くツールを目指している。

### 階層と整理

- **Areas > Projects > Tasks > Subtasks**（サブタスクは1階層のみ。深すぎる入れ子を許さない）
- **Tags** はタスク・ノート・プロジェクト横断のフラットな分類軸
- **Inbox** で未整理のキャプチャを受け止め、ハッシュタグやURLを自動解析して提案

### タスクの基本機能

- **7つのステータス**: Not Started / In Progress / Done / Archived / Waiting / Cancelled / Planned
- **3段階のPriority**: Low / Medium / High（ソートのみに影響）
- **Due Date**（締切）と **Defer Until**（指定時刻まで完全非表示）を別概念として明確に分離
  - Defer = 「いつから考え始めるか」
  - Due = 「いつまでに終えるか」
- **Subtasks**: 親と子は独立したライフサイクル。親完了で子は自動完了しない
- **添付ファイル**, **タグ**, **イベント履歴**（不変な監査ログ）

### スマートな仕組み

- **Today ページ**: Overdue / Planned / Suggested / Completed の4セクションで自動振り分け
- **Suggested（おすすめ）**: 期日・優先度・タスク齢・作業時間帯パターンから、今やるべきものを提案
- **Upcoming**: 7日間プレビュー
- **Recurring Tasks**: 親テンプレートと仮想インスタンスの分離。柔軟な繰り返しパターン
- **Habit Mode**: 単なる繰り返しと別に、ストリーク・達成回数・柔軟性モードを持つ習慣トラッカー

### コラボレーションと連携

- プロジェクト単位での共有（read-only / read-write / admin）
- スタンドアロンタスクは所有者のみ
- **Telegram bot** からの作成・日次ダイジェスト
- REST API + 個人アクセストークン
- Email / Push / In-app 通知をタイプ別・チャネル別に細かく制御

### 注目すべき設計判断

- **ソフトデリートなし**: 削除は永久（完了タスクはArchive推奨）
- **イベント履歴は編集不可**: 完全な audit trail
- **Task Intelligence はオプトアウト可能**: AIっぽい提案機能を嫌うユーザーも切れる
- **24言語対応** + **セルフホスト前提**

要するに、GTDやPARA等の方法論を強制せず、**Defer/Due の分離**や**Today の自動振り分け**といった"認知負荷を下げる"細部に思想が現れているシステムです。

## Q3-1. planned って何？

**A.**

**「今日（または近いうち）に取り組むことを自分でスケジュールしたタスク」** を表すステータスです。

- 7つのステータスのうちの1つ（値は `6`）
- 似たステータスとの使い分け:
  - **Not Started** = まだ手付かず
  - **Planned** = 「やる」と決めて予定に入れた状態
  - **In Progress** = 実際に着手中
  - **Waiting** = 他の何かを待っている

### Today ページでの扱い

Todayページの `Planned` セクションには、**status が `In Progress` / `Planned` / `Waiting` のタスク**が集約されます（つまり「今動いている／動かそうとしている」もの全部）。

特徴的なのは:

- **"今日"チェックボックスではなくステータスで判定**する。Plannedにすれば due_date が無くてもTodayに出る
- **Overdue より優先**。期限切れでもステータスがPlannedならPlannedセクションに留まり、Overdueには出ない（重複を避けるため）
- 並び順は Priority → Due date → Project

要するに「今日のTo-doリスト」を作る主役のステータスです。

## Q3-2. inboxの自動解析ってLLMを使うの？

**A.**

**使っていません。** ローカルでのルールベース＋軽量NLPライブラリ（`compromise`）です。

`backend/modules/inbox/inboxProcessingService.js:6` で `require('compromise')` しているのが本体で、`package.json` にも `"compromise": "^14.14.4"` として入っているだけ。OpenAI/Anthropic等の外部API呼び出しはありません。

### 実際の解析内容

| 機能 | 仕組み |
|---|---|
| `#tag` 抽出 | 正規表現 |
| `+project` 抽出 | 正規表現（`+"with spaces"` も対応） |
| URL検出 | `http(s)://` の文字列マッチ |
| **動詞判定**（Task提案の根拠） | `compromise` で品詞解析し、先頭語が動詞かを判定。auxiliary verb（is/has等）は除外 |
| URL のページタイトル取得 | 対象ページを fetch して `<title>` を抜く（3秒タイムアウト） |

`compromise` は辞書ベースの軽量NLPライブラリ（数百KB、推論なし）なので、ドキュメントの "AI-powered suggestions" は誇張表現で、実体は**ヒューリスティックな分類器**です。動詞検出は英語前提なので、日本語の inbox では `Task` サジェストはほぼ機能しないはずです（タグ・プロジェクト・URL抽出は言語に依存しない）。
