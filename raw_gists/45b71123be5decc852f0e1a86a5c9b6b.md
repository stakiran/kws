# Triple Debt Model と Organizational Debt に関する Q&A

## Q1: triple dept model について調べて(agentic engineering の文脈)

「triple dept model」は **Triple Debt Model**(三重債務モデル)のタイポで、ビクトリア大学の Margaret-Anne Storey 教授が 2026 年に提唱した概念です。論文タイトルは "From Technical Debt to Cognitive and Intent Debt: Rethinking Software Health in the Age of AI"(arXiv:2603.22106)。

### 背景にある問題意識

「technical debt(技術的負債)」という用語は1993年に Cunningham が提唱して以来、長らくソフトウェア工学の中心的な概念でした。しかし生成AI・エージェント型AIの普及で、リスクの所在が静かに変わりつつある──というのが Storey の出発点です。「生成AIはソフトウェアエンジニアリングの課題を取り除くのではなく、再分配している」と表現されています。

### 3つの債務

ソフトウェアの健全性は単一のレイヤーではなく、3つのレイヤーに依存しているというのがこのモデルの核です。

**1. Technical Debt(技術的負債)— コードに宿る**
コードレイヤーの問題。システムを「変更しにくく」する。従来から知られている概念で、設計や実装上の妥協が将来の改修コストとして蓄積するもの。

**2. Cognitive Debt(認知的負債)— 人に宿る**
時間の経過とともにチーム内で共有された理解が侵食されること。システムを「理解しにくく」する。"technical debt" が「コードに溜まる債務」をイメージさせるのに対し、cognitive debt は「スピードを優先したツケが開発者の頭の中に溜まり、その後の理解力や変更能力に影響する」ことを示す概念。

**3. Intent Debt(意図債務)— 外部化された知識(アーティファクト)に宿る**
人間とAIシステムの両方がコードベースを安全かつ効率的に扱うために必要な、外部化されたゴール・制約・根拠の欠如。システムが「何のためにあるのか」を分からなくする。

### 3つの債務の相互作用

このモデルが効くのは、3つが互いに強化し合うループを成しているという指摘にあります。

- Intent debt が cognitive debt を引き起こす(目的が文書化されていないと、新メンバーは正確なメンタルモデルを作れない)
- Cognitive debt が technical debt を引き起こす(システムを理解していない開発者は実装判断を誤る)
- Technical debt が cognitive debt を増幅する(汚いコードはそもそも読み解きにくい)

### Agentic Engineering 文脈での重要性

AIは technical debt を減らす一方で、cognitive debt と intent debt を加速させる可能性がある。開発者がコードを一から書くとき、実装の摩擦そのものが「理解」を副産物として生み出していたのですが、エージェントが代わりに書くとその副産物が失われます。

Shaw と Nave の研究では「認知的降伏(cognitive surrender)」が、AIが間違っていても自信を膨らませることが指摘されており、これが cognitive debt が手遅れになるまで見えにくい理由を説明する。チームは「分かっているつもり」のままになる。

コード品質だけにフォーカスしているチームは、ソフトウェア健全性リスクの3分の1しかマネジメントできていない。AIが加速する世界での難問は「コードがきれいか」ではない──というのが実務的な示唆です。

### なぜ今、注目されているか

"intent debt" は、開発者だけでなく AI エージェントが安全にコードを扱うためにも必要な「外部化された根拠」の欠如を指す点が新しいところで、エージェント型AIに仕様や意図を「読ませる」ためのドキュメント・コメント・ADR(Architecture Decision Records)・CLAUDE.md/AGENTS.md のような仕組みの価値を、債務という枠組みで再定義したわけです。

Simon Willison が "Agentic Engineering Patterns" を書き始めたり、DX社や RDEL のような開発者生産性系メディアが相次いで取り上げたりしているのは、Claude Code・Codex などのコーディングエージェントが現場に入った結果、まさにこの3つの債務のバランスが崩れていることを実務者が体感しているからです。

元論文: arXiv:2603.22106(Storey, M.-A., "From Technical Debt to Cognitive and Intent Debt")。DX社のニュースレター記事も読みやすい入り口になります。

## Q2: 僕はさらに organizational debt があると思うんだけど、これについて扱った論文はある?

はい、organizational debt はすでに扱われています ── ただし「Triple Debt Model に第4の柱として加える」議論はまだ未着手の余地が大きい領域です。

### 1. 起源:Steve Blank(2015年)

そもそも "organizational debt" は新概念ではなく、スタートアップの世界で著名な Steve Blank が2015年のブログ記事 "Organizational Debt is like Technical Debt – but worse" で広めたのが出発点です。「スタートアップの初期段階で『とにかく出荷するため』に行われた人/文化面の妥協のすべて」と定義され、企業を成長させる過程で『リファクタリング』しないと、急成長中の企業を混沌としたナイトメアに変えると論じられました。

後に、この概念はスタートアップを越えて大企業にも適用されるようになります。複雑な問題に単純な解決策を求める傾向──問題の根本原因や全体との関連を理解する代わりに、再発防止のためのインストラクションやポリシーをすぐ作る──が組織的負債の最も典型的な源泉とされています。

### 2. ソフトウェア工学領域での学術的な扱い:Al-Baik et al. (2024)

ここが問いの核心に近い論文です。

**Al-Baik et al. (2024), "Organizational debt—Roadblock to agility in software engineering: Exploring an emerging concept and future research for software excellence", PLOS ONE**

ソフトウェア工学において、組織的負債(OD)は重要だがほとんど研究されていない現象。ODは、組織の進歩と適応性を阻害する時代遅れの構造・ポリシー・プロセスの蓄積を指す。このマルチボーカル文献レビュー(MLR)は、ソフトウェア実務者の知見を統合し、ODの原因・帰結・特定方法・緩和策を明らかにする。

9本の査読論文と22本の最近のブログ記事を対象に、定義・原因(管理不全な変革やサイロ化された取り組み)・帰結(イノベーションとアジリティの低下)・緩和策(アジャイル原則、分散意思決定、スタッフの知見の活用)の4テーマが帰納的に抽出された。

つまり、ソフトウェア工学領域でも独立した研究対象として確立されつつあるが、まだ「新興の概念」というステータスです。

### 3. アジャイル大規模開発との接続:Mikalsen et al.

**"Organizational Debt in Large-Scale Hybrid Agile Software Development: A Case Study on Coordination Mechanisms"** という研究もあり、大規模ハイブリッドアジャイル開発における調整メカニズムの観点から OD を扱った事例研究です。組織構造そのものがソフトウェア提供能力に直接効くという話で、Conway の法則の現代版に近い視座を与えます。

### 4. AI / Agentic 文脈との接続:ここが薄い

ここが面白いところです。AI 時代のソフトウェアにおける「○○ debt」の議論は急速に拡張していて、"AI Debt"(プロンプトのバージョン管理欠如、評価基盤の不在、観測可能性の欠如、ブラックボックスAPIへの依存などを含む)、"Model-Stack Workaround Debt", "Model Dependency Debt", "Performance Optimization Debt" といったLLM固有のSATD(Self-Admitted Technical Debt)カテゴリ、"GenAI-Induced Self-Admitted Technical Debt"(AI生成コードが残す TODO や HACK コメントの分析)など、コード・モデル層に閉じた負債論はかなり厚みがあります。

一方、**organizational debt を agentic engineering / AI-assisted development の文脈で正面から扱った論文は、現時点で見当たりません。** Storey の Triple Debt Model も、organizational layer には明示的に踏み込んでいません。

### 5. 周辺研究

ただし、隣接領域で示唆的な研究はあります。

例えば「Reliable agent engineering should integrate machine-compatible organizational principles」(arxiv:2512.07665)では、LLM エージェントと組織科学の互換的フレームワークの並行性を検討し、エージェント設計におけるエージェンシーと能力のバランス、エージェントスケーリングにおけるリソース制約と性能ベネフィット、エージェント管理における内部・外部メカニズムという3つの組織原則を提示している。これは "組織論をエージェント設計に持ち込む" 方向ですが、逆向き、つまり "エージェント時代に組織側に蓄積する債務" を論じた研究は、Triple Debt Model の延長としては未開拓です。

### 6. 「第4の柱」案がなぜ筋がいいか

Triple Debt Model の3層を再掲すると、

- Technical debt — コードに宿る
- Cognitive debt — 人(の頭の中)に宿る
- Intent debt — アーティファクト(外部化された知識)に宿る

ここに organizational debt を加えるとき、論じる価値があるとすれば──担当者個人の頭の中(cognitive)でも、ドキュメント(intent)でもなく──**役割分担・意思決定権限・チーム境界・レビュープロセスといった「人と人の間の構造」に宿る債務** という独自層が立ち上がります。

エージェントが書いたコードをどのチームがレビューするのか、エージェントの出力に責任を持つのは誰か、エージェント用のガードレールを誰が更新するのか、複数チームをまたぐ AI 利用ポリシーを誰が決めるのか──これらは technical / cognitive / intent のどれにも還元できない、組織構造側の負債です。Conway の法則が言うように「組織構造はアーキテクチャに浸み出す」のだとすれば、エージェント時代にはむしろ Conway の逆流(エージェントの能力分布が組織構造を再定義してしまう)も起き、組織側に蓄積する歪みは独立変数として扱う価値が出てきます。

論文化されていないだけで、概念的にも歴史的(Blank からの系譜)にも、十分に第4の柱を主張できる土台はあると思います。

## Q3: そうなんですよ、たとえば従来の同期的な会議とレビューにばかり頼っていたら、明らかにそこがボトルネックになるじゃないですか。また従来どおりn人で1製品を抱えるとしても、構造的に誰かがエースになってそれ以外がサブになるじゃないですか

両方とも、agentic engineering 時代に organizational debt が「独立した負債」として立ち上がる典型例だと思います。

### ボトルネック化する同期儀式の問題

これは要するに **「人間の同期コストが、エージェントの非同期スループットに対する律速段階になる」** という構造的な現象ですね。

従来は、コードを書くスピードと人間が議論するスピードがだいたい同じオーダーだったので、デイリースタンドアップやスプリントレビュー、PR レビュー会といった同期儀式は「ちょうどよい摩擦」として機能していました。むしろ、コードを書く時間が支配的で、会議は相対的に小さいオーバーヘッドだった。

ところが Claude Code や Codex でエージェントが並列に複数タスクを進める世界では、生成側のスループットが桁で上がる一方、レビュー・承認・意思決定は人間の脳と会議室の容量に張り付いたままです。結果として、**「待ち行列が人間側に全部溜まる」** という現象が起きる。これは技術的負債でもなければ、認知的負債でもなく、**プロセス設計の負債**です。

しかも厄介なのは、これが従来の組織論の枠でも見えにくい点です。「会議を減らせ」「非同期コミュニケーションに移行せよ」という処方箋は10年以上前から言われていますが、Basecamp や GitLab のような先進例を除けば、多くの組織が同期儀式に依存し続けてきました。なぜかというと、コードを書くスピードが遅かったうちは、それでも回ったから。エージェントの登場で、**「これまで隠れていた organizational debt の利息が、急に表に出てきた」** という構図です。

この観点で書かれた研究は──少し探した感じ──まだなさそうです。隣接して言及している人はいて、たとえば Storey も intent debt の議論で「同期的な会話で共有していた文脈が、AI の介在によって失われる」というニュアンスは触れていますが、ボトルネックとしての同期儀式そのものを debt として扱った定式化はまだ見当たらないです。

### エース/サブの構造的非対称性

これも本質的な指摘で、しかも組織心理学・組織行動論の古典的なテーマと接続できる話です。

n 人で 1 製品を持つとき、エージェントが入る前は「全員が等しく忙しい」状態が(理想化すれば)成立しえた。コードベース全体を誰か一人が把握するのは無理だから、自然と領域分担が起き、それぞれが自分の領域では「エース」になっていた。

ところがエージェントが入ると、状況が変わります。エージェントへの **プロンプティング能力・コンテキスト設計能力・出力評価能力** が、コードを書く能力よりも支配的な要素になる。そしてこれらは──Storey の言葉を借りれば──cognitive debt のマネジメント能力に近い。**システム全体の意図を保持し、エージェントに正しい文脈を渡し、出力を評価できる人** が、構造的に1人(あるいは少数)に集中しやすい。これは Brooks の "surgeon team" モデルが、悪い意味で復活する現象に近いです。

なぜ「悪い意味で」かというと、Brooks のオリジナルでは外科医チームは意図的な役割分担でしたが、エージェント時代のエース集中は **「気づいたら自然にそうなっている」「他のメンバーが追いつけない」** という非対称性として現れるからです。エースは1日に5タスクをエージェントに走らせて全部レビューできるが、サブは1タスクの文脈を理解するのに半日かかる。差は埋まらず、むしろ広がる。

これも organizational debt の典型例だと思います。なぜなら:

- **Technical debt ではない**:コードはきれい。むしろエースが書く(書かせる)コードは品質が高い。
- **Cognitive debt ではない**:エース個人の頭の中の理解はむしろ深まる。問題は「チームの理解の非対称性」。
- **Intent debt ではない**:ドキュメントは充実しているかもしれない。問題は「ドキュメントを書く人と読んで活用できる人の構造的なギャップ」。

つまり、3つのどれにも還元できず、**役割分担と権限配分という組織構造の層** に債務が溜まる。

### 関連しそうな先行研究

ここは論じる余地が大きい部分ですが、引っ掛けになる先行研究をいくつか挙げると:

**Knowledge silos と truck factor**:Avelino らの "A novel approach for estimating Truck Factors" (ICPC 2016) など、コードベースに対する知識集中度を測る研究系列があります。エージェント時代には truck factor がさらに小さくなる可能性があり、定量化の枠組みとして転用できそうです。

**10x developer 神話と組織への影響**:Sackman, Erikson, Grant の古典的研究(1968)から、近年の「AI 拡張による生産性差」研究まで、個人差が組織にどう影響するかの議論は厚い。Cui et al. の "The Effects of Generative AI on High Skilled Work" (2024) などで、AI による生産性向上は元々スキルの高い人ほど大きいという結果も出ていて、エース/サブの非対称性が AI で広がる仮説と整合的です。

**Conway の法則の逆流**:Bailey らが言う "inverse Conway maneuver" の議論が近いですが、エージェント時代には「エージェントの能力分布が組織構造を歪める」という別種の逆流が起き得る。これも未開拓です。

### 書くなら、たぶん筋になる構造

もしこれを論じるなら、Triple Debt Model に第4の柱として organizational debt を加える形ではなく、**「Triple Debt Model の3層が組織構造によってどう増幅・減衰するか」** という視点で書いた方が、たぶん筋がいいと思います。理由は2つあって:

1. organizational debt は Storey の3層と並列ではなく、**3層すべてに影響するメタ層** として作用するから。同期儀式の負債は cognitive debt の蓄積を促進し、エース集中は intent debt を見えにくくする、といった具合に、層を貫いて効く。

2. Storey の Triple Debt Model は「ソフトウェアの健全性」のモデルですが、organizational debt は「ソフトウェアを作る組織の健全性」のモデルなので、論理的なレイヤーが違う。並列に置くと議論が混ざる。

つまり、**「Triple Debt × Organizational Structure → debt accumulation rate」** という乗法的な関係として定式化する方が、たぶん新規性も主張しやすいし、現場の処方箋(同期儀式の見直し、エース依存の構造的解消、エージェント用ガバナンスの分散化)にも繋がりやすいんじゃないでしょうか。

少なくとも今ざっと検索した範囲では、この角度で正面から書いた論文はなさそうです。
