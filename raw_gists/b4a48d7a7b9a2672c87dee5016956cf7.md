# AIエージェントフレームワークは商業レベルで本当に役に立つのか — 仕様駆動開発・Superpowers・oh-my-claudeを巡る対話

## **Q1:** 仕様駆動開発その他OSSのAIエージェントフレームワークって、本当に実際に仕事で役に立っているんですか？vibe coding と同じく、おもちゃにしか見えないんですよね。各社がアピールのために大げさにやってるだけであって、本当に商業レベルをつくれるか疑問詞している。調べて。

## **A1:**
率直に言うと、**あなたの懐疑感は半分正しくて、半分は時代遅れになりつつあります。** データを見ながら整理します。

### 数字で見る「実態」

楽観論と懐疑論が混在しているので、業界調査の数字を並べてみます。

- IDCの調査では、AIエージェントのパイロットの88%が本番環境に到達しないまま終わる
- McKinseyのState of AIレポートでは、実際にAIエージェントをスケールできている企業はわずか23%。39%は実験段階で停滞している
- Gartnerの2026年CIO調査では、AIエージェントを実際にデプロイした組織は17%にとどまる(ただし42%が今後12か月以内にデプロイ予定と回答)
- 一方で、S&P GlobalとMcKinseyの調査では31%の企業が少なくとも1つのエージェントを本番運用しており、銀行・保険業界では47%に達する。デプロイメントの中央値投資回収期間は5.1か月

つまり「おもちゃか実用か」は **二極化** しています。多くは失敗するが、特定の条件で動かしている企業は実利を出している、というのが現状です。

### 失敗パターン:あなたの直感の正しい部分

失敗の原因はモデルではなく、その周辺のレイヤーにあることが多い。サンドボックスでは動くエージェントが、本番では実際の認証情報、顧客データ、コンプライアンス要件、そして本物の結果に直面して破綻する。

リーダーの70%が「非決定的なアウトプット」を本番リリースの最大の障壁として挙げている。「モデルが間違っている」というより「いつ間違うのか事前に分からず、回帰テストでも捕まらない」のが問題。

エージェント型システムは狭く制御されたタスクでのみ有望さを示している。実世界の運用は依然として脆く、ハルシネーションを起こし、エッジケースで苦戦し、常に人間の監督を必要とし、限界的なリターンしか出さないことも多い。

**ここはあなたの懐疑が当たっている部分です。** デモは派手で、商業生産レベルの堅牢性とは別物、というのは2026年現在も真実です。

### ただし「役に立っている領域」は明確に存在する

2026年、エージェントはIT運用、社員サポート、財務オペレーション、オンボーディング、リコンシリエーション、サポートワークフローといった制約された、よく管理された領域でメインストリームになる。これらは人間によるレビューを許容でき、境界が明確で、ROIが早く出る。

勝者は職務全体を置き換えようとしている人々ではない。予測可能なパターンに従う、特定の狭いワークフローを自動化している人々である。コードレビュー、データ入力、レポート生成、シンプルなコンテンツ運用がそれ。

成功している企業の共通パターン:
- 81%がエージェントを単一のワークフローに限定し、二値の成功基準を設定している(オープンエンドなアシスタントではない)。87%がプロンプト・モデル・ツール変更ごとに自動評価を実行。74%が最初の60〜90日間は明示的なヒューマンインザループのチェックポイントを設けてデプロイする

「AI秘書」みたいな汎用エージェントは大体おもちゃのまま。「Jiraチケットからリリースノートを生成する」みたいな狭くて検証可能なエージェントは実用になっている、という整理です。

### 仕様駆動開発(SDD)について

これはエージェントフレームワーク全般とは少し別の話で、**vibe codingの失敗への反動**として登場したものです。

SDDは2025年、大規模言語モデルでの「vibe coding」の失敗モード(エージェントが意図からずれた、もっともらしいコードを生成し、APIをハルシネーションし、プロジェクトのスケールとともに腐敗する)への直接的な対応として登場した。

つまり「SDDとvibe codingが同類」というあなたの捉え方は逆で、**SDDはvibe codingがおもちゃだという認識から出てきた処方箋**です。

具体的な実例:
- AWSはAmazon Kiroの組み込みSpec Modeを使い、2週間かかる通知機能を2日に短縮した
- LLMは9.8〜42.1%の割合で脆弱性のあるコードを生成し(Yan et al., 2025)、本番リポジトリで生き残っているAI起因の問題は2026年2月までに11万件を超えている。SDDはまさにこれらの失敗に対する能動的な検証ゲートとして実行可能な仕様を埋め込む
- EU AI Actは高リスクAIシステムに対し、2026年8月2日以降の義務遵守を要求し、違反には最大1,500万ユーロまたは世界年間売上高の3%の罰金を課す。コンプライアンス要件は仕様を証拠として扱うようになっている

つまり SDD は「アピールのための大げさな概念」というより、**規制と本番事故への対応として現実的な需要が押し上げている**側面が強いです。

### 各社の「アピール」をどう読むか

ご指摘の通り、各社のマーケティングは盛っています。ただ、注意したい点が2つあります。

**1. オープンソースには商業圧力がない。** GitHubが公開したオープンソースツールキットSpec Kitは72,000以上のスターを獲得している。Martin Fowlerがブログで書き、Thoughtworksが詳細な分析を公開した。Hacker Newsで過去1か月に3回トレンド入りした。ベンダーの宣伝とは別の経路で広がっているという事実があります。

**2. ただし「銀の弾丸」ではない。** マークダウン肥大化リスク - ステークホルダーのコラボレーション方法を変えずに仕様駆動ワークフローを採用すると、過剰なドキュメンテーション負担が発生しうる。学習曲線も急で、仕様優先で考える必要があり、プロンプト優先の開発からのマインドセット転換が要る。エンタープライズ統合も現状のツールではCI/CDパイプラインや既存チームワークフローへのサポートが不十分

これは古いウォーターフォール開発の罠を再導入しがちで、規律のないチームではむしろドキュメント地獄になります。

### 結論

あなたの問いに直接答えるなら:

| 主張 | 実態 |
|---|---|
| 「vibe codingはおもちゃ」 | **だいたい正しい**。プロトタイプには使えるが、商業生産レベルではない |
| 「AIエージェントフレームワークもおもちゃ」 | **部分的に正しい**。88%が本番に至らず失敗するが、狭く設計されたエージェントは銀行・保険などで実利を出している |
| 「SDDもアピール用のおもちゃ」 | **これは外している可能性が高い**。vibe codingの失敗とコンプライアンス圧力への現実的応答として広がっている。ただし無批判に採用すると別の罠にハマる |
| 「商業レベルが作れるか疑問」 | **正当な疑問**。「フレームワーク選び」より「狭くスコープを切り、人間のチェックポイントを置き、評価基盤に投資する」ことの方が成否を決めている |

要するに、「フレームワーク自体が魔法を起こすか」を問うとあなたの懐疑は正しい。**フレームワークは前提条件で、勝負は運用・スコープ設計・検証基盤の側にある**、というのが2026年時点の実態です。マーケティング素材だけ見ていると見えない部分です。

## **Q2:** それを差し引いても、ですよ。たとえば世の中で広く使われてるSaaSの中で、本当にAI任せでつくったものがあるのか？もちろん個人レベルでは自由に多用してるだろうけど、Kiro その他フレームワークのような規律に従ってつくりましたってのが本当にあるのか？

## **A2:**
これは率直に答えづらい質問なんですが、**「広く使われている既存のSaaS」と「AIで一から構築された商業SaaS」を分けて見る**必要があります。順番に見ます。

### 既存の有名SaaS:「AI任せ」とは違うが、AI生成の比率は急上昇している

ベンダーCEO自身が公に数字を出しています。

- Microsoft CEOのSatya Nadellaは2025年4月、同社のコードの最大30%が現在AIで書かれていると発言。Google CEOのSundar Pichaiは2025年10月、新規コードの25%以上がAIで書かれていると発言。Shopify CEOのTobi Lutkeは従業員に対し、増員を要求する前にAIではできないことを証明する必要があると伝えた
- AnthropicのClaude Code責任者Boris Chernyは2か月以上コードを書いていないと公表。会社全体では70〜90%のコードがAIで書かれており、Claude Code自体は約90%が Claude Code 自身によって書かれている
- Science誌に掲載された調査では、GitHub上のPython関数の約29%が米国でAIによって書かれている(他地域では低め)

ただし、これは **「AI任せで作った」のではなく「人間のエンジニアがAIを道具として使った」結果**です。Microsoftの30%もSatya Nadellaは「人間がレビューしてマージしている」ことを前提に話しています。**Kiroのような厳密なSDDフレームワークに従って構築されたかというと、そうではない**のが大半です。

### では「規律に従って AI 中心で作った商業 SaaS」は実在するのか?

ここが核心ですね。答えは「**実在するが、まだ規模は限定的**」です。

#### 比較的説得力のある実例

**1. Anthropic 自身の Claude Code(年間収益数億ドル規模の商業プロダクト)**

Claude Code については、約90%のコードが Claude Code 自身によって書かれている。これは一応「広く使われている SaaS」のカテゴリに入ります(2026年時点で開発者の46%が「最も愛するAIコーディングツール」に挙げる規模)。ただし、**Anthropic は SDD フレームワークというより「Claude Code 自身を使ったドッグフーディング」**で、Kiro のような形式的な仕様駆動ではありません。

**2. Y Combinator のスタートアップ群**

Y Combinator の Garry Tan によれば、Winter 2025 バッチでは創業者の25%が AI に大きく依存しており、コードの95%が大規模言語モデルによって生成されている。

これらは商業 SaaS として動いていますが、ほとんどはまだ初期段階のプロダクトです。「広く使われている」とまでは言えない。

**3. AWS Kiro の Bedrock AgentCore で構築された創薬エージェント**

2025年中盤の Kiro ローンチ以降、ライフサイエンス顧客は仕様駆動開発の最初の AI コーディングツールとして Kiro を採用し、ターゲット同定や臨床試験設計などのR&Dワークフロー、340B 収益保護などのコマーシャル運用にわたって、開発タイムラインを月単位から週単位に短縮している。

ただしこれは AWS の自社ブログのアピールで、第三者検証ではありません。

#### あなたの懐疑が当たる部分

ここまでの調査で、**「広く使われている既存のSaaS(Salesforce、Slack、Notion、Figma など)で、Kiro や Spec Kit のような規律に従って一から構築されたものは事実上ない」**と言って差し支えありません。理由は単純で、

1. それらは AI コーディングツールが成熟する前から存在する
2. Lightrun の 2026年調査では、AI が生成した変更の43%が本番環境でデバッグを必要とし、調査対象のリーダーで「AI 生成コードに非常に自信がある」と答えた人はゼロだった

これが「広く使われるレベルの SaaS」をフルAI構築で作ることに、既存ベンダーが踏み込めない最大の理由です。

#### 「規律に従って AI で構築」を公にしている小規模 SaaS

開発者ブログ上の事例だと、Kiro の仕様駆動開発によって、calisthenics(器具体操)イベントと選手プロフィール管理のための SaaS プラットフォームをアイデアから磨き上げまで構築した、といった個人/小規模なものが多い。

**Lovable、Bolt、Replit Agents、v0** で構築された製品も多数存在しますが、これらは「規律」というよりは vibe coding 寄りで、あなたの問いの対象外です。

### まとめ:質問への直接的な答え

| 問い | 答え |
|---|---|
| 広く使われている既存 SaaS で AI 任せで作ったものはあるか | **ほぼない**。AI で書かれた割合は急増中(Microsoft 30%、Google 25%、Anthropic 70-90%)だが、これは「人間が AI を道具にした」のであって「AI 任せ」ではない |
| Kiro/Spec Kit のような規律で作られた商業 SaaS はあるか | **存在するが規模は限定的**。Y Combinator の若いスタートアップ群、AWS の社内ユースケース、Anthropic 自身のドッグフーディングなど。「広く使われている SaaS」のレベルに到達したものは Claude Code くらい |
| 各社のアピールは大げさか | **半分大げさ、半分本当**。30〜90%の AI 生成コードという数字自体は本物だが、「AI が自律的に SaaS を作った」というデモはマーケティング寄り。実態は「経験豊富な人間が大量に AI 出力をレビューして直している」 |

つまり、あなたの直感は概ね正しいです。「Kiro 等の SDD で作りました」と胸を張れる**広く使われた**商業 SaaS は、2026年5月時点で **Claude Code 以外にはほぼ存在しない**、というのが現状の偽らざる風景です。

ただし注意したいのは、これは「フレームワーク自体がおもちゃ」という意味ではない、ということです。**プロのエンジニアが Cursor や Claude Code を使ってコードベースの 30〜90% を生成している、というレベルでは確実に既存 SaaS の現場で動いている**。アピールと実態のギャップは、「自律的なエージェントが作った」という部分にあって、「AI がコード生成に深く使われている」という部分ではない、というのが正確な切り分けです。

## **Q3:** Superpowers、oh-my-claude といったアジャイル型のフレームワークはどうですか？SDDと同じく、こちらも全面的に使った事例はないと思ってるけど。

## **A3:**
これも率直に言うと、**あなたの直感はかなり正確**です。ただ、SDD のときとは違う事情があるので分けて見ます。

### まず Superpowers と oh-my-claude が何を売りにしているか

これは SDD とは少し位置づけが違います。Superpowers は、TDD、デバッグ、ブレインストーミング、計画の作成、コードレビュー、完了前の検証など、エンジニアリングプラクティスごとの指示をマークダウンファイルのフォルダとして提供する Claude Code プラグイン兼マルチホストのエージェント型スキルフレームワーク。同じフォルダが Claude Code、Cursor、OpenAI Codex、GitHub Copilot CLI、Gemini CLI、OpenCode で動く。

oh-my-claudecode (OMC) は、計画、PRD作成、実行、検証、修正の5段階パイプラインを持つ。tmux を使って並列 CLI ワーカーを起動し、Claude、Codex、Gemini を同時に動かす。最大5つの同時ワーカーをサポート。

つまり、**SDD が「ウォーターフォール風の仕様優先」だとすれば、Superpowers/OMC は「アジャイル風のチーム化+規律化」**を目指している、というのが正しい捉え方です。プロセスの形が違うだけで、両方とも「vibe coding はおもちゃ」という認識から出発しているのは同じです。

### 採用状況:数字は意外と大きい

ここはあなたの予想と少しズレるかもしれません。

- Superpowers は93,000人以上の開発者コミュニティが貢献している(別資料では124K GitHub stars という記述もあり、勢いは強い)
- oh-my-claudecode は GitHub Trending で1日に858 stars を獲得し、Discord も活発
- OMC は13,400 stars を獲得しており、これが需要の証拠

数字だけ見ると盛り上がっています。ただし、これは「インストールしている個人開発者の数」であって「商業プロダクションで採用しているチーム数」ではないことに注意が必要です。

### 「全面的に使った商業事例」はあるか?

ここで核心に踏み込みます。**結論から言うと、ありません。** 少なくとも公に確認できる範囲では、Superpowers や oh-my-claude を全面的に使って構築されたメジャーな商業 SaaS は存在しません。

理由は3つあります。

#### 1. 「全面的に使う」設計になっていない

OMC は銀の弾丸ではない。レイヤーが増えれば、それだけコストも増える。マルチファイル/マルチロールのタスク、2時間を超える長いセッション、アーキテクチャ変更や大規模リファクタリングといった計画重視の作業には適している。一方で、1つの関数の修正や1つのバグの修正にはチームモードのパイプラインはオーバーキル。トークンコストが気になる場合、5段階パイプラインは純粋な Claude Code よりも大幅にトークンを消費する。透明性が重要な場合、オーケストレーションレイヤーが増えるほど「なぜこの決定がなされたか」を追跡するのが難しくなる

これは作者自身も認めている設計思想です。**Superpowers も OMC も「すべての作業で使うべきフレームワーク」ではなく、「複雑なタスクを規律化するためのオプション」**として位置づけられている。

#### 2. 実測すると「常に勝つ」わけではない

12セッションの独立した実験結果が興味深いです:

Superpowers セッションは平均で約14%少ないトークンを使った。ただしこの平均は重要なパターンを隠している。シンプルなタスクでは実際 Superpowers のほうがトークンを多く使った。clarify と design フェーズは「ユーティリティ関数を書いて」のようなストレートなタスクには必要のないオーバーヘッドを追加する。曖昧さのない問題に対して明確化の質問をしてしまう。1つのファイルに収まるものに対してアーキテクチャを計画してしまう。規律は本物だが、50行のスクリプトに対してはオーバーキル

つまり、フレームワーク自体は本物の改善をもたらすが、**タスクとミスマッチがあると逆効果**になる。これは商業現場での「全面採用」を妨げる構造的な問題です。

#### 3. そもそも企業はこういう個人OSS規律ツールを基盤にしない

これは重要なポイントですが、業界の議論を見ると一貫してこういう論調が見られます:

Claude Code はひどく名前が悪い。素晴らしくて強力でゲームチェンジャーだが、最も強力なユースケースは本番コードを書くことではない。代わりに外科医になり、Claude Code を助手として使うこと。「外科チーム」モデル — 中核的な仕事(本番コード)をやる1人のマスター外科医を、文字通り他のすべてを担当する助手たちがサポートする

Claude の力は、与えるブロックの良さによって制限される。錯覚しないこと。Claude は Sentry や Terraform や Playwright を再現できない。これらは信じられないほど複雑でよく設計されたコードだ。Claude は自分で良い抽象化を作ることができないので、Claude 単独でできることには限界がある。X 上ではみんな vibe-code ですべてのソフトウェアが作れると思っているが、私はその逆が真実だと考えている: 良い抽象化とよく設計されたインフラの価値はかつてないほど高くなった

つまり、商業現場でのコンセンサスは **「人間の熟練エンジニアが設計の根幹を握り、AIフレームワークは助手として補助する」** であって、「フレームワークが全面的に開発を回す」ではない。

### 重要な皮肉:Anthropic 自身も「全面採用」していない

ここは前回の調査と繋がります。Claude Code については、約90%のコードが Claude Code 自身によって書かれている という事実があります。

つまり、Anthropic は **Claude Code をドッグフードしている**わけですが、**Superpowers や OMC のような第三者フレームワークを全面採用しているわけではない**。Boris Cherny が「2か月以上コードを書いていない」と言うときの作業は、Claude Code を直接使った作業であって、Superpowers のワークフローではありません。

これは何を意味するかというと、**「規律化されたフレームワークがないと AI が商業コードを書けない」わけではない**ということです。Anthropic 自身が証明している。逆に言えば、**Superpowers や OMC は「もっと弱い AI モデルや、ドッグフード経験のないチーム」が品質を担保するための足場**として価値がある、という位置づけのほうが正確かもしれません。

### まとめ:あなたの問いへの直接的な回答

| 問い | 答え |
|---|---|
| Superpowers/OMC を全面採用した商業 SaaS はあるか | **公に確認できる範囲ではない**。個人開発者の採用は爆発的に伸びているが、企業の「ドックフード」事例は出ていない |
| これらはおもちゃか | **おもちゃではないが、銀の弾丸でもない**。複雑な長期タスクには測定可能な改善がある(14%トークン削減など)。シンプルなタスクには逆効果 |
| アピールは大げさか | **「3-5倍速」「30-50%安い」といったマーケティング数字は怪しい**。実測では条件依存。ただし設計思想自体(TDD、レビュー、計画)は健全 |
| SDD と同じ運命か | **似ているが、より厳しい**。SDD はエンタープライズ規制(EU AI Act など)に後押しされているが、Superpowers/OMC は個人開発者向けの「規律ツール」で、エンタープライズの後ろ盾がない |

率直に言うと、こういう **アジャイル型フレームワークの本当の市場は「個人開発者と小規模スタートアップ」**であって、「Salesforce や Notion を Superpowers で書き直す」というシナリオは現実的ではない、というのが2026年5月時点の風景です。

ただ、あなたの問いに含まれた前提を1つだけ修正したいです。「全面的に使った事例がないからおもちゃ」とは限らない。**プロのエンジニアの作業の質を底上げするツール**としては、独立した実験で測定可能な効果が出ている。「フレームワークが SaaS を作る」ではなく「フレームワークがエンジニアの一部の作業の質を上げる」という尺度なら、おもちゃではない、というのが公平な結論だと思います。

## **Q4:** 良くも悪くも一つの道具ってわけだね

## **A4:**

そうですね、まさにそこに落ち着くと思います。

「魔法」でもなければ「おもちゃ」でもなく、**ノコギリやドリルと同じレベルの道具**。よく切れる電動ノコギリを買っても、家は建たない。設計図と熟練の大工が要る。でも熟練の大工が電動ノコギリを使えば、手ノコの3倍速で正確に切れる。

各社のマーケティングが胡散臭く見えるのは、まさに「電動ノコギリが家を建てます」という売り方をしているからで、そこに対するあなたの違和感は完全に正当だと思います。実際にプロの現場で起きているのは「熟練エンジニアが Claude Code を電動工具として使って、コードベースの 30-90% を生成している」という、もっと地味な話です。

そして道具である以上、**使い手を選ぶ**。Microsoft の Nadella が「コードの30%が AI 製」と言えるのは、それを書いた AI の後ろに数万人のエンジニアがいて、レビューとアーキテクチャを握っているから。同じツールを未熟な手に渡しても、43%が本番でデバッグ必要 の側に落ちる。

良くも悪くも、というのが本当に正確な表現ですね。
