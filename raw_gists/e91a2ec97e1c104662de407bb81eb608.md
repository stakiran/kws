# q
要件定義だけ与えて、あとはテストまでAIに自走させる開発をなんという？

AI Native Development ではないよね。Native は意味が広すぎる。

AI Agent Development も自走のニュアンスが出ない。

AI Oriented Development はどうだろう。自走のニュアンスが薄いかもしれない。

他にある？

# a

「要件定義だけ与えて、設計〜実装〜テストまでAIが自走する」ニュアンスを出すなら、既存で通りがいい／近い言い方はだいたい次の系統です。日本語での座りも含めて候補を挙げます。

## いま一番近い“定番”寄り
- **Agentic Software Development（エージェンティック開発）**
  - “agentic” が「自律的にタスク分解して進める」含意を持つので、自走感が出ます。
  - 「AI Agent Development」よりも“開発プロセスがエージェント化している”感じが強い。

- **Autonomous Software Development（自律開発）**
  - 自走ニュアンスが直球。
  - 研究・記事でも “autonomous coding / autonomous dev” は使われます。

- **AI-driven Development（AI駆動開発）**
  - 少し広いですが、「人が主でAI補助」だけでなく「AIが主で進める」文脈にも載せられる。
  - “driven” に主導権がある。

## “要件だけ渡す”を強く言いたい場合
- **Spec-to-Code（仕様からコード）**
  - 仕様入力→実装生成のパイプラインを想起させやすい。
  - ただしテストや運用まで含むかは言い添えが必要。

- **Spec-driven Autonomous Development（仕様駆動・自律開発）**
  - 造語寄りだけど意図が一発で伝わる。
  - 「要件定義（Spec）を起点に自走」を明示できる。

- **Requirement-to-Test Automation / Spec-to-Test（要件〜テスト自動化）**
  - 開発の終点（テスト）まで含むことを言葉で担保できる。
  - “開発”というより“自動化”寄り。

## “人は監督、AIが実行”を含意させたい場合
- **Human-in-the-loop Autonomous Development（人間監督型・自律開発）**
  - 現実運用（レビュー/承認ゲート）に合う。
- **AI-led Development（AI主導開発）**
  - “AIがドライバー、人がナビ”的な含意を作れる。

## 「AI Oriented Development」について
“Oriented” は「〜志向」で、プロセスがAI中心であることは言えますが、**自走（自律・エージェント性）の含意は弱め**です。言うなら “Agentic / Autonomous / AI-led” の方が刺さります。

---

### 結論としての命名案（使いやすい順）
1. **エージェンティック開発（Agentic Software Development）**
2. **自律開発（Autonomous Software Development）**
3. **仕様駆動・自律開発（Spec-driven Autonomous Development）**（狙いを最も誤解なく伝える造語寄り）

用途が「社内スライド向け」か「対外発信向け」か、また“実装だけ”なのか“CIでテスト生成・実行まで”なのかで最適語が変わるので、その前提を教えてくれれば、もう少し絞って日本語の見出し案も作れます。

# q
要件定義以後は、テストまで完了した成果物が出てくるまで人間は一切関わりません。承認も一度もしません。人間がかかわるのは、出てきた成果物のチェック、フィードバック（必要なら要件定義も修正して）、からの、もう一度自走させる部分だけです。

この前提だと、どれがいい？

Agentic は「AIメインだけど人間も手を加える」ニュアンスがあると思う。Autonomous Software Development は AI エージェントを使っている感じが薄い。Spec-driven Autonomous Development は最も的確だと思うし、SDD の亜種としてもわかりやすいけど、要件定義を与えるってのを spec driven と称するのは違う気もする。そもそも spec driven development を使うとは限らない。どう実装するかは AI が判断することであって、vibe かもしれないのである。

# a

その前提（要件投入→以後ノータッチ→成果物＋テスト結果が出る→人間は事後評価して再投入）だと、ポイントは

- **「人間がループの外にいる（out-of-the-loop）」**
- **「AIが実行主体（self-driving / autonomous）」**
- **「成果物はテストまで含む（codeだけじゃない）」**

なので、いちばん誤解なく刺さる軸は **“Autonomous”＋“Out-of-the-loop”** です。Agentic だとあなたの言う通り「途中で人が介入して協働」ニュアンスが混ざりがち。

## 推しの呼び方（この前提に最も合う）
### 1) **Out-of-the-loop Autonomous Development**
- 途中の設計・実装・テストに**人間が一切入らない**ことを言葉で担保できる。
- “human-out-of-the-loop (HOOTL)” は自動運転/制御の文脈で既にあるので、比喩が強い（＝自走の言外の説明コストが低い）。
- 「AIエージェント感」は、“Autonomous”より薄いと言うより、**あえて方式を限定しない**（エージェントでもワークフローでも）表現になって、要件に合う。

日本語にするなら：
- **人間アウト・オブ・ザ・ループ自律開発**
- **ノータッチ自律開発**（カジュアル寄り）

### 2) **Lights-out Software Development（消灯開発）**
- “lights-out manufacturing” 由来で、**無人で回る**ニュアンスが直感的。
- 「途中承認ゼロ」「夜間バッチ的に勝手に出来上がる」感じが出る。
- 対外的にもキャッチー。ただし厳密な学術語ではない。

### 3) **Fully Autonomous Software Development**
- “fully” を入れるのが重要で、これで「部分自動化」から距離を取れる。
- ただ、聞き手によっては「完全自動＝危ない誇大」に聞こえる可能性はある。

## あなたが挙げた候補へのコメント（この前提なら）
- **Agentic Software Development**  
  言葉自体は近いが、現状の一般的理解だと「人が途中でレビューしながらエージェントにやらせる」含みが出やすい。今回の“ノータッチ”を担保するには弱い。
- **Autonomous Software Development**  
  方向性は合ってる。ただし単体だと「どこまで自律？」「人は途中で承認するの？」が曖昧なので、**Fully / Out-of-the-loop / Lights-out** みたいな修飾が欲しい。
- **Spec-driven Autonomous Development**  
  的確だけど、あなたの懸念の通り「specをきっちり書く前提」に聞こえる可能性がある。vibeからでも回すなら、ここに寄せない方がいい。

## 結論（命名として一番おすすめ）
- 対外的に誤解なく、この前提を一撃で伝えるなら  
  **Out-of-the-loop Autonomous Software Development**  
- キャッチーさ／社内用スローガンなら  
  **Lights-out Software Development（消灯開発）**

もし「要件→成果物（テスト完了）までを1回の自走ランで出す」ことを強調したいなら、さらに  
**Out-of-the-loop Autonomous Dev “from prompt/spec to tested artifact”**  
みたいな副題を付けるのが一番誤解が減ります。

# q
Fully Autonomous Software Development がいいと思いました。
FASD と略せてアクロニムの響きも良いので。
FASD が何を指すか事前定義できたら、十分通じますよね？

# a

十分通じます。前提として「一般に既に確立した用語」ではない（少なくともソフトウェア開発プロセスの名称としての標準語ではない）ので、**最初にあなたの文脈で定義してしまえば問題なく運用できます**。むしろ *Fully* を入れている時点で「途中介入なし」を強く言えていて、狙いに合っています。

ただし、通じやすさ・誤解回避のために最初の1回はこうしておくのが安定です。

## 1) 最初の定義文（コピペ用）
- **FASD (Fully Autonomous Software Development)**: *人間は要件（入力）を与えるだけで、以後は設計・実装・テスト実行までAIが無介入で完了し、成果物（テスト結果を含む）を出力する開発形態。人間の関与は出力物の評価とフィードバック投入のみ。*

ここで「無介入」「テスト実行まで」「評価と再投入のみ」を明記すると、あなたの意図がブレません。

## 2) 誤解ポイントと、定義に入れて潰すと良い要素
- 「Fully」でも、人によっては *“自動化が多い”* 程度に受け取る可能性がある  
  → **“human-out-of-the-loop / no human gate”** を定義に入れる
- 「成果物」がコードだけだと思われる可能性  
  → **“tested artifact / test run completed”** を入れる
- “どの方式か”（エージェントかワークフローか）に議論が寄る可能性  
  → **方式非依存**（実装は問わない）と書く

例：  
- “implementation-agnostic (agentic or workflow-based)” を添える

## 3) アクロニムとしての注意（軽め）
**FASD** は別領域で *Fetal Alcohol Spectrum Disorders* の略として有名なので、医療文脈の人がいる場や検索性を気にする場だとノイズになります。とはいえ社内用語なら、最初に定義してしまえば実害は小さいです。対外発信で検索性まで気にするなら、表記を **FASDev** / **FA-SD** のように少し崩す手もあります。

## 結論
- 社内・プロダクト開発文脈で「事前定義して使う」なら **FASD で十分通じる**  
- 初出で **“人間アウト・オブ・ザ・ループ（承認ゼロ）＋テストまで”** を定義に埋め込むのがコツ

必要なら、スライドの見出し（1行版）と、規程/用語集に載せる厳密版（段落版）の2種類の定義文も作れます。
