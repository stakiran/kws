### コンテキストは動的
まず「コンテキスト」とはどういうものかを決めておきたいです。

ずばり、**コンテキストは動的（Context is Dynamic）** としたいです。

たとえば以下はコンテキストとは言えないと思います:

- ナレッジやリファレンスといった静的な情報
- 個人や組織の暗黙知
- 暗黙知というほどではないが、頭の中または日頃の運用として現れる信念や文化

そもそもコンテキストウィンドウに入りきらないか、入ったとしても情報が雑多すぎてまともな回答が得られません。重要なのは **これら静的なプール（情報源）をいかに解釈するか** でしょう。

一部については、すでに Prompt Engineering でカバーできています。たとえば全部入らないので 5 つだけ選ばせるとして、「最も重要なものを 5 つ選んで」「選び方は reference/kanten.md の観点を使い、reference/score-algorithm1.md に則って求めなさい」のように書けばいいわけです。あるいは、知りたいことを具体的に書けば LLM 側が絞ってくれます。

また、 RAG とも違います。RAG は静的なプールを最適化しているだけであり、静的であることに変わりはありません。取り出しやすくなっているだけであり、どう取り出せばいいかという解釈の部分にはタッチしていません。RAG に限らず、メダリオンアーキテクチャ（[原典](https://www.databricks.com/blog/what-is-medallion-architecture), [2.0](https://zenn.dev/google_cloud_jp/articles/ff74bf18e44f97)）のようなものも同様ですね。

このように静的なプールという切り口を持ってくると、以下が不在であることがわかります:

- 1: 静的なプールに追加する「新しい情報」のつくりかた（創造）
- 2: 静的なプールをどのように解釈するか（解釈）


Human Cotnext 


1 の創造は、たとえばプロジェクトのビジョンやリソースの制約を、どうやって入れるかという話になります。あるいは状況を正確に表現するための言葉や用語を自ら定義します。

問題領域を自身が把握していないなら、顧客その他関係者へのヒアリングが必要かもしれません。いずれにせよ、当事者自ら言語化して書くしかありません。

2 も解釈同様に、LLM に渡す解釈を自分で言語化して書かないといけません。あるいは「どう思う？」など漠然と聞くこともできますが、返ってきた回答を解釈して、自分なりの理解に落とし込まねばなりません。ただし状況次第では LLM の回答そのものか、それを少しいじる程度で事足りることもあります。

私はコンテキストは 2 の部分であると考えます。つまりコンテキストとは **静的なプールから何らかの解釈をもって取り出された情報** と定義するのが良い。筋が良いのではないかと。

ですので、**解釈を変えればプールが同じであってもコンテキストが変わります**。これを動的と表現しています。2 の解釈はここを言っています。

そして、この動的なコンテキストの表現力はプール次第ですので、プール自体のメンテナンスも要ります。特に良質な素材を入れられるかどうかが肝でしょう。1 の創造はこの点を言っています。言語化したビジョンを与えれば一貫性を与えられますし、用語集を与えれば細かなニュアンスの違いも潰せます。良い素材をつくるという意味で、創造と書いています。

### 言語化という営み
Human Context は言語化の営みです。

今までは技術的な営みでした。Window Context と Agent Context は技術的な話であり、仕様を押さえた上で手を動かすことで「最適な設定」を探すものでした。数学ほど厳密ではありませんが、実験によってより良い正解に近づき、その再現性を上げることはできます。十分検証されたものは「パターン」となり、誰でも使えるようになります（一般的には理論よりもソフトウェアの方が使いやすい）。

たとえば LLM には「コンテキストウィンドウの上限」があって、これに収まるようにせねばなりませんし、従量課金の仕組みもあるので「コスト管理」も要ります。それらの現状を可視化する機能やツールがすでにあるので、実際に使って確認しながら作業をしたり、エージェントにそれらを使わせてモニタリングさせたりできます。いずれもパターンとして確立されており、私達の仕事は、それらパターンの理解と実践です。これらを実践することで、比較的かんたんに LLM を思い通りに使えるようになったり、より安く使える（あるいは高すぎて悶絶する展開を避けられる）ようになるわけです。

一方、Human Context は、そうした技術的な営みに **渡す情報をつくるもの** です。プロジェクトのビジョンや、プロジェクトにあった用語体系は誰もつくってくれませんし、技術的な営みではつくれません。またどこかに正解が転がっているわけでもない。あなた自身やあなたの属する組織用のそれらは **自らがつくるしかない** のです。

無論、AI にお任せしてある程度つくってもらうことはできますが、そのまま採用しても理解が追いつかないか、納得できないでしょう。生成 AI の出力をそのまま投げたところで相手にされないし、しないのと同じです。少なくともレビューや手直しは要りますし、おそらくそれでも足りずに、結局自らちゃんとつくる（書く）ことになります。ここでは私達の仕事は解釈です。もっと言えば言語化です。

### 即興的な言語化と、その先
既存の AI エージェントフレームワークを見ていると、Human Context にあたる部分は即興でつくっていることがわかります。

- [Superpowers](https://github.com/obra/superpowers) の brainstorming
- [oh-my-claudecode](https://github.com/yeachan-heo/oh-my-claudecode) の deep-interview

他の層については、すでに即興から一段進化しつつあります。Window Context については、RAG や memory を始めとする永続化の仕組みや、compaction などウィンドウ資源をやりくりする手段もあります。Agent Context についても、MCP や Skill といった形で手続きを外部化・永続化したり、エージェントに必要な各種設定をフレームワークとその設定パラメーターの形で同様に管理できています。

一方で、Human Context はまだ即興の段階にあると考えます。あるいは静的なプールを整備するところで終わってしまっています。何度も言う通り、Human Context とは静的なプールではなく、それを解釈したものです。プロジェクトのキックオフ資料を markdown 化したものを置いておくのではなく、最も重要なビジョンや、使えるリソースが事実上決まっている点を定量化したリストなどを別途定義して置いておきたいのです。

### ではどうすればいいか？
ここを体系的に整備するのが Human Context Engineering の仕事だと思います。

まずは概念的な話をします。層を一つ追加すればいいと考えます。

これまで:

```
[Content]     ドキュメントその他データベース（静的なプール）
      ↓
[Code]        コードや設定ファイル
```

これから:

```
[Content]     ドキュメントその他データベース（静的なプール）
      ↓
[Context]     Contentを解釈したもの（動的な情報、コンテキスト）
      ↓
[Code]        コードや設定ファイル
```

Content と Context の境界は曖昧ですが、言いたいことは一つで、「解釈をもっとちゃんとやる」です。ちゃんとやるためにわざわざ層を設けています。もっと言えば、質の高い解釈を生み出すことと、再現性を持たせて何回も使えるようにすることの二点です。

次に方法にも踏み込んでいきましょう。

> 質の高い解釈を生み出す

質の高い解釈は、濃密な対話と思考によって生み出せます。 **濃密に行う手法** が必要でしょう。AI 前提でやるなら、人間よりも AI との会話に時間を割くはずです。方法は色々あります。

- 役割やペルソナを定義した AI エージェントを動かして議論させるのか
- AI の出力を読みつつ人間が深く思考するのか
- Context 層を「LLM 出力の解釈を貯めたプール」として育てていくのか（[LLM Knowledge Base](https://x.com/karpathy/status/2039805659525644595)など）
- 無限のおもちゃとして会話しまくることで真の解釈に近づいていくと考えるのか etc

私個人はブレストや KJ 法のような営みが大事と考えていて、[創造的思考（クリエイティブ・シンキング）と題して整理した](https://stakiran.github.io/soft-skills-engineering/07_creative_thinking.html)ことがあります。ポイントはひとりで孤独に居ることであり、孤独を邪魔しない環境と孤独に耐える能力が重要だと位置づけています。あるいはチームワークを本番試合とたとえて、練習試合も必要だよねということでソロワーク――つまり個人で過ごすことが練習に相当する、という言い方をすることもあります。

> 再現性を持たせて何回も使えるようにする

次に再現性については、筋のいい **解釈を Content 化する** か、 Content 化はしないが「解釈の仕方」をスキルやエージェントや MCP などで **道具化する** かです。

例として [Agentic Engineering](https://x.com/karpathy/status/2019137879310836075)（AI に開発を任せて人間は監督に徹する）や [FASD](https://stakiran.github.io/12-factor-fasd/)（造語ですが Fully Autonomous Software Development、人間の監督や承認なしで全部 AI におまかせ）で開発するシステムをつくっていると考えてください。

現時点で正解はないし、既存の理論やフレームワークを集めてきただけではうまくいかないでしょう。方向付けや意味付けを自ら開拓していかないといけません。プロジェクトのオーナーは頭の中に完成形があって、これを検証したいとします。日頃から開発メンバーに伝えているつもりですが、なかなか伝わりません。

こういうとき Human Context の目線が役に立ちます。たとえばオーナーが描いているモデルを言語化すればいいのです。次のように言語化してみました:

- Orchestration
    - Harness
        - Agent
            - Tool

概念とその関係性をこのように定義して、「こういう世界を描いてるんだ」「そうすれば各層で差別化の余地を探せる」と言えるわけです。仮に OHAT（オーハット）と名付けます。Tool は決定的な出力を行うプログラムで、エージェントはプロンプトベースで非決定的に出力する + Tool も使う存在、Harness は Agent のラッパーで Window Context や Agent Context の管理も埋め込み、Orchestration は複数エージェントの連携と前提知識（またはその参照）を与える――と、いかがでしょうか。ここまで定義すると、だいぶ議論や検証がしやすいはずです。

これは適当な例ですが、この程度の概念モデルをつくるだけでも結構大変です。少なくともオーケストレーション、ハーネス、エージェントといった多義語を調査・比較して実態を把握したり、「差別化の余地を突くことで生き残りたい」といったオーナーの意思を言語化をしておかないといけません。その上で、エンジニアが設計のときにやるような構造の視座での整理が必要になります。ここでは単純化して階層的にしていますが、もちろん階層だけで上手く表現きるとは限りません。

この OHAT の概念をドキュメント化して Content に入れておけば、以降 AI にも OHAT を意識させることができます（Content化）。あるいは OHAT に基づいて「オーナーの目線でレビューを行うシステム」をつくることもできます（道具化）。おそらくオーナー自身、各層の具体例やイメージがあるはずで、それも含めて組み込めばいいわけです。

ちなみに、この OHAT は上述の創造的思考でつくっています。

- 発散として各種多義語の実態やオーナーの意思（架空なので小説の設定を考える要領で洗い出し）を洗い出す
- 収束として、発散したものを眺めながら関連性や構造を色々反映してみる
- そのうち本質が見えてくる。今回は「Agent のラッパー」や「知識とその参照は上位から与えることにする」という整理を使って、OHAT を出せました

という具合に導きました。

手段としては [vscode-scb](https://github.com/stakiran/vscode-scb) や [scbdown](https://github.com/stakiran/scbdown) を使っています。ローカルで、文章ベースで考えを深めるために、Cosense の記法を持ち込んだオリジナル記法を自製しています。書いて考える上では、少しでも書きやすく残しやすくて読みやすいフォーマットがあると便利なのでわざわざ自製しました。

と、このように、これだけ見ても相当なやり方と考え方に支えられていることがおわかりいただけるはずです。それも既存の事例や理論ではカバーできていません。だからこそ Human Context Engineering として整備していきたいという思いがあります。

仮に理論に持っていくなら、「創造的思考」や「書きながら考える道具の自製」、また「書きながら考える営み」や「多義語の調査と整理の仕方」あたりが章レベルの内容になるでしょう。最後については、もう少し絞って、かつキャッチーにして「バズワード・エンジニアリング」としてもいいかもしれません。

### Human Context の重要性
AI 時代のボトルネックは人間です。

- いくら AI が正しい or 比較的正しい成果を速く出したとしても、人間が納得できねば意味がない
    - いわゆる「正論は通じない」と同じです
- AI と一緒に深い洞察を出したメンバーがいても、周囲が AI を使えないと通じない
    - 一番単純な例は、AI により翻訳能力を手に入れて英語の情報を扱えるようになったとしても、周囲が AI を使えないのだとしたら通じない、です
- AI により議論の量と質は何十倍・何百倍にも加速できるのに、やる気やスキルがないと変わらない
    - 比較的詳しいエンジニア同士であっても、会議で口頭で喋る以外の手段ばかりに頼っている、はよくあります
    - 最も詳しくて時間をかけたメンバーがまとめた詳細レポートは、おそらく読まれません（AI を使えば自分なりに咀嚼していけますがそれもしません）
- ...

この人間ボトルネックの主因は **主体的な内省の不足** だと考えます。もっと言えば、

- 言語化。意思決定や判断を行うための文脈がない（言語化していない）
- 知識管理。言語化はしているが、蓄積・整理して形にしていないため定着しない
- AI 利活用。特に対象からブラックボックス的に知見を引き出す使い方に慣れてない
- 孤独な読み書き。AI を使うための「机上でひとりで読み書きをする」体験に慣れていない

といったスキルやリテラシーがなかったり、そもそもこういうことが必要である（現時点で筋が良い）ことを知らないのです。エンジニアであっても例外ではありません。オープンソースの AI エージェントをバリバリ追いかけて使いこなせるようなベテランやスタッフエンジニアであっても、上記はほとんどできなかったりします。

このあたりはまさに Human Context であり、エンジニアリング化（工学化）して再現性をもたせたい――誰でも扱えるようにしたい、というのが私の思いです。

何が難しいかって、正解がないんですよね。数学のような厳密性はないし、トップダウンでこれさえやればいいと指示されるわけでもない、またソフトウェアのように動けばいいという明確な基準もない。

それもそのはずで、 **Human Context の本質は「どう解釈したら納得がいくだろうか？」** だからです。本質的にベストエフォートであり、終わりはありません。

無論、無限にこもっていても、ただの頭でっかちですので、どこかで行動に移さねばなりませんが、だからといって解釈を怠けていても案外動きづらいわけです。上記のボトルネックが阻みます。AI のおかげで翻訳と製造の能力は上昇したので、なおさらこの人間のボトルネックが効いてきます。

だからこそ、AI を含む IT エンジニアリングの知見と、その他言語化に絡む知見を学際的にフル活用して、このボトルネックを壊していきたいのです。また、壊せると私は考えています。
