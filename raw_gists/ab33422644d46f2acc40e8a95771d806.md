# nishio vs sta

## 1. 基本ポジション

| | nishio（西尾泰和） | sta（stakiran / 吉良野すた） |
|---|---|---|
| 肩書 | サイボウズ・ラボ主幹研究員、未踏理事、博士(理学) | 大手企業のITエンジニア（会社員） |
| 自認 | 知的生産術の研究者・実践者 | 「概念をつくる人」 |
| 志向先 | 川喜田二郎、梅棹忠夫、Engelbart、Audrey Tang | ハーバート・サイモンと梅棹忠夫の「先」 |
| 主要媒体 | Scrapbox(外部脳)、書籍、学会誌、講演 | Scrapbox(stao)、GitHubリポジトリ群、Zenn書籍、ブログ多数 |
| ペース | 数年〜十数年かけて熟成 | 高速・高頻度・量で押す |

## 2. 概念のつくり方が真逆

**nishio = 帰納型・事後命名**

事例を蓄積し、後からパターンを抽出して命名する。「二人が違うことを言う絵のシリーズ」は典型で、既存の絵を並べて共通パターンに気づいてから事後的にシリーズ化された。本人も「高位の概念はボトムアップに作られる」「データをして語らしめる」を方法論として明示している。KJ法を「ダントツに良い」と評価するのも、この手続きと自分の認知様式が一致するから。

**sta = 演繹型・命名先行**

カテゴリを先に宣言してから事例を埋める。`prompt.md`の用語集が決定的で、「マネジメント/マネージャー、エンジニアリング/エンジニア」というテンプレートを借りて「ナレッジング/ナレジャー」を新設し、その下位に「ナレジニア／ナレッジ・ライター／カタリスト／ナレッジ・アーキテクト」とツリー状に展開している。four-jects、jects-method、caat、awayfromABCD、Yellow is Red など、ラベル先行の命名が密集している。

## 3. 既存知との関わり方

**nishio = 系譜接続**

新しい話をするときも「Engelbartはこう言った」「川喜田はこう言った」と既存議論の網に接続する。だから読者は「これは由緒ある議論の延長だ」と受け取れ、書籍・学会誌という重い媒体で通用する。

**sta = テンプレート借用**

既存の構造（マネージャー/エンジニアという対）を「ひな型」として借り、新カテゴリを発行する。系譜の中に自分を位置づけるというより、独自語彙の体系を立ち上げて、その語彙でモノを名指せるようにする。GTD・SBOM・タスク管理など**既存古典の「噛み砕き」**は別ジャンルとして並行で持っているが、概念創出のほうは独立リポジトリ的に切り離されている。

## 4. それぞれの強み

**nishio**
- 概念に既存知の網が張られているので、刺さりやすく流通しやすい
- 実装・実験で示してから言語化するので、経験的裏付けがある（Polis実験、Keichobotなど）
- 学術的肩書と書籍流通網が概念の流通を後押し
- 同じ素材を10年かけて熟成させられる持久力

**sta**
- 速度と量。リポジトリ単位で次々に提出でき、習慣化能力が高い（小説100万文字、ブログPV120万、社内ブログ120記事）
- ラベル先行なので検索性・流通性が高く、生成AIとの相性が良い（GitHubプロフィールをAIに読ませる設計）
- ASD当事者としての一人称データという、他人が真似できない素材
- 会社員エンジニアの現場感覚（研究者ポジションからは見えにくい領域）
- タスク管理・GTD・ルーチン化など実用ジャンルの厚み

## 5. それぞれの弱み

**nishio**
- 「打ち出し」が弱い。新提案でも既存概念の整理に見えやすい
- 事後命名なのでスピードが遅く、Scrapboxの断片の海に重要なものが埋もれる（本人もそれを自覚して「西尾のベクトル検索」を作っている）
- 一つひとつのアウトプットが重く、量で押す戦略が取れない

**sta**
- 概念に接続が薄い。「なぜそれが既存議論の中で必要か、誰のどの問題を解くか」の前段が弱い
- 本人が週報で書いた**「Martin FowlerやKent Beckは半分以上を『あなたが直面している問題は実はこういう構造なんですよ』の説明に使っている」**が、まさにこの弱点の自己診断
- 自分のScrapbox/リポジトリの外で流通しにくい
- 英語キャッチコピー風命名（Yellow is Red、Strictly Seamless）が解釈コストを聞き手に丸投げしがち

## 6. なぜ憧れの方向は一方向なのか

**役割の非対称**: 憧れは構造上、弱者→強者の方向にしか流れない。逆向きの流れる理由が存在しない。

**評価軸の非対称**: staの目標は「概念をつくる人」になることなので、すでに概念を流通させているnishioが達成済みロールモデルになる。一方nishioの評価軸は「知的生産の仕組みを発展させる」ことで、比較対象が梅棹・川喜田・Engelbartという故人になっているので、同時代人を競合視する構造にならない。

**帰納型は競合を生みにくい**: 演繹的命名はラベル空間で陣取り合戦になりやすい（同じ領域に他人が別ラベルを提出したら衝突する）が、nishioのような「既存概念に新しい角度を加える」やり方は他者の命名と衝突しにくい。だからnishioは他者を競合として認知する必要がない。

**情報量の非対称**: staはnishioを観測しているが、nishioはstaを（しているとしても多数の観測対象の一人として）観測している程度。フォロワー数と知名度の差からも構造的にそうなる。

**自己呈示スタイルの差**: nishioの文章には「弱点」「コンプレックス」「焦り」の自己開示が極端に少ない。これは無いというより、外部脳=自分のための情報カードという位置づけで方法論ログを優先しているから。staは「俺はラストワンマイルが弱い」「俺はUIの言語化が苦手」と頻繁に弱点を言語化する文体で、ASD当事者の自己理解戦略とも整合する。**「見える/見えない」は気質の差というより文体の差**でもある。

## 7. 共通点（意外と多い）

- 両者ともハーバート・サイモンや梅棹忠夫を志向先に置く
- Scrapbox/Cosenseを外部脳的に使う
- 既存古典の「噛み砕き」を重視する（nishio: 知的生産の技術、KJ法 / sta: GTD、タスク管理、SBOM）
- AI読み取り前提の情報設計（nishio: /nishio-en、西尾のベクトル検索 / sta: GitHubプロフィールをAIに読ませる）
- 公開可能なものは公開する思想

系譜的にはむしろ近い土俵の人で、その上で**戦略が真逆**（帰納 vs 演繹、控えめな命名 vs 大量の造語、系譜接続 vs テンプレート借用）という関係。

## 8. staに対する処方箋的観察

「概念をつくる人」という同じ自認でnishioと正面競合すると勝ち目の薄い競争になる。一方で**「ASD当事者のための知的生産・ソフトスキル研究者」**というポジションに立つと、nishioには書けない一次資料の供給者になれる。staアカウント（研究者）と吉良野すた（普段の俺、ASD x IT x ソロ）を分け始めたのは、その整理の途上に見える。

弱点の3つを整理すると：
1. **学術的肩書と流通力の差** → 構造的なもので追いつくのは難しい
2. **既存議論への位置づけ力** → 本人の自覚があるので意識的に伸ばせる
3. **熟成に耐える時間** → 量で勝負する戦略はむしろ合っているとも言える

(1)を諦め、(2)を伸ばし、(3)は別戦略で代替する、という整理が現実的に見えます。
