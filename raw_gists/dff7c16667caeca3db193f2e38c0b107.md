# 生成AIによるライトノベル執筆の調査記録

## Q1: 生成AIでライトノベルを生成する試みはありますか？調べて

**A1:** 調べてみますね。

はい、生成AIでライトノベル（および小説全般）を生成する試みは、個人レベルから商業・賞レースまで幅広く行われています。主なものを整理します。

### 専用ツール・サービス

**AIのべりすと**は日本最大級の小説生成AIで、日本語最大級の68.7億パラメータ、約500GBのコーパス（文庫本174万冊分）からフルスクラッチで訓練されているサービスです。小説の書き出しを入力すると続きを自動生成し、推理・ロマンス・ホラー・ファンタジー・異世界転生ものなど様々なジャンルに対応しており、ラノベ系の生成にもよく使われています。

その他、CanvaのマジックライトやChatGPT、Claude、Geminiなども執筆補助に活用されています。

### 個人による長編ライトノベル生成の事例

noteにはChatGPTやClaude 3などを活用してサイバーパンク/バイオパンク的なボーイ・ミーツ・ガールのライトノベル（5万文字以上）を実際に書いた試みが公開されています。長編生成は短編より難しく、まだ目立つ事例が少ない分野です。

### 文学賞での動き

星新一賞は早くからAI作品を受け入れており、応募規定に「人間以外（人工知能等）の応募作品も受付けます」と明記されているという珍しい賞です。

- 2021年、AIを執筆に取り入れた小説「あなたはそこにいますか？」が第9回日経「星新一賞」一般部門優秀賞を受賞（葦沢かもめ氏）
- 第12回（2025年）では、青野圭司氏の「アルゴリズムの檻」が最終選考10作品に残り、執筆の8割をAIに任せた作品として注目されました

### ノウハウの蓄積

実践者からは具体的な手法も共有されており、アイデア出し、コンセプト設定、世界観・キャラ設計、あらすじ作成、本文執筆、推敲といった小説執筆の全工程で生成AIと対話しながら進められるとされています。一方でライトノベルは一見シンプルに見えて複雑で洗練された創作技術の集大成であり、キャラクターの一貫性、伏線、会話の作り方など「見えない職人技」をAIに伝える工夫が必要とも指摘されています。

### 現状のまとめ

短編やショートショートではAI主導でも一定の質に到達しており賞も取れるレベルです。長編ラノベは個人の挑戦例が出始めた段階で、「AIに完全自動で書かせる」より「人間とAIの共作」が主流です。ジャンル的に異世界転生・なろう系のテンプレ性とAIの得意分野が合致しやすく、相性は良いとされています。

## Q2: 方向性が違います。AIを用いてライトノベルを書いている人の具体的な事例や知見が知りたかったのです。賞やプラットフォームがAI製もサポートしてるとか、AIのべりすとのような有償製品には興味はない。

**A2:** 失礼しました。実践者の具体的な事例と知見に絞って調べ直します。

実践者の事例と知見を中心にまとめ直します。

### 個人の長編執筆チャレンジ

**平大典氏（note）— 5万字以上のラノベ生成実験**

主にGemini 1.5 Proを使用。トークン量が1,048,576あるので継続して文章を書かせるのに適していると判断。Gemini 1.5 flashは文章が単調すぎて使わなかったとのこと。企画段階では過去作のあらすじをAIに読ませて方向性を提示させ、約15分で企画が確定。SF・サイバーパンク系のラノベを実際に完成させています。

**結氏（note）— 「なろう」「カクヨム」で連載**

AIにプロットを考えてもらい自分で本文を執筆した作品や、AIにプロットから本文までを書いてもらい自分で整えた作品を、「小説家になろう」や「カクヨム」で連載中。「才能がないと思っていた人」が物語を書き始めるためのツールとしてAIを位置づけているのが特徴。

### 葦沢かもめ氏 — 最も体系化された知見

星新一賞優秀賞も取っているこの人の手法が、おそらく一番参考になります。

**プロットを「1文単位」に分解して生成させる**

「場面ごとの大まかな人物の行動」のレベルにプロットを落とし込んでAIに与えると、割とその通りに書いてくれる。全部AIに任せるのではなく、プロットを1文ずつに分解して、それぞれの1文から続きを書かせる。自分のプロットを骨組みにして、AIで肉付けをしていく——これがコアアプローチ。「部品を作って結合する」プログラミング的発想に近い。

**「三幕構成」をプロンプトに組み込む**

2023年1月末頃、「三幕構成」などの執筆技法をプロンプトに取り入れることでクオリティを改善できることに気付いた。素のChatGPTに「書いて」と言うのではなく、物語論の枠組みを与えるアプローチ。

**AI使用のマナー策定（あしざわ法典）**

画一化を防ぐため基本的なプロンプトを提示しつつ、様々なアレンジ例も載せた「あしざわ法典」を発表し、AIを使った創作のマナー普及を目指した。

### プロ作家からの実践的な指摘（落とし穴）

座談会記事で出ていた問題点：

- どんでん返しを作らせようとすると、AIは「私には感情がありません」と蹴る。何でもかんでも夢オチにしてしまう——感情の起伏に依存する仕掛けは苦手
- 「鏡に映った自分と喧嘩しながら、『そうじゃないんだよ、僕が書きたいプロットは』とやり直してもらう」——自分の中の怪物と向き合う作業になる

### 長編特有の課題：コンテキスト維持

ある程度書き慣れた人の共通認識として、Claudeはコンテキストウィンドウが大きく前半の設定や伏線を後半まで保持しやすいが、1万字を超えるあたりから細かい設定を忘れ始めるため対策が必要とされ、対策としてキャラ設定書・章ごとのプロット・設定リマインダーの3つを事前に用意して、2000〜3000字ずつ生成していく方法が定着しつつあります。

### ジャンル特有の難しさ

ラノベは案外AIが苦手で、「どこかで読んだことがあるような展開ばかり」「主人公の性格が薄く、会話も棒読み」「面白そうなシーンがあっさり終わってしまう。『もっと詳しく』と指示しても、前の内容を繰り返すだけで深みが出ない」といった声が頻出。「一発で完成品を求める」のではなく段階的プロセスにする工夫が必要、というのが現場の答え。

### 知見の総まとめ

実践者の知見を要約すると以下：

1. **AIに丸投げしない**。プロット骨格は人間が握り、AIには「シーンの肉付け」を任せる
2. **生成単位を小さく**。章単位ではなく1シーン2000〜3000字くらいが安定領域
3. **物語論のフレームを与える**（三幕構成、起承転結など）
4. **キャラ設定書・プロット・リマインダーを毎回頭に入れ直す**（長編対策）
5. **得意分野・不得意分野を見極める**（情景描写・地の文は得意、感情のどんでん返しや独自の比喩は苦手）
6. **Claudeは長文・文学的描写向き、Geminiは長コンテキスト向き、ChatGPTは指示忠実**といったモデル使い分け

葦沢氏のnoteやインタビューが、日本語圏で最も体系化された実践知の集積点です。深掘りするならまずそこを当たるのが効率的かと思います。

## Q3: まだ想定が違う。ChatGPT ではなくで Claude Code のような AI エージェントを使うという話です。

**A3:** なるほど、AIエージェント（Claude Codeのような自律的にファイル操作・ツール実行ができるエージェント）でライトノベルを書く試み、ですね。調べ直します。

ありました。むしろこの領域は今、急速に盛り上がっている分野で、知見も体系化が進んでいます。「ソフトウェア開発の手法で小説を書く」という発想が共通項です。

### 代表的な実践事例

#### 1. 葦沢かもめ氏「Claude Codeで小説を書く方法」(2025年8月)

前回紹介したAI小説家がそのままClaude Codeに移行した形。5分間でエンターキーを1回押しただけで1万6000字が生成できているとのこと。基本構造はシンプルで、プロジェクトフォルダに以下を配置：

```
config.md              # 企画書
episode_writing_rule.md # 執筆ルール
settings/characters.md  # キャラデータ
settings/plot.md       # 全体プロット
```

CLAUDE.mdをメモリとして使い、3段階で生成。①plot.mdの12エピソード分のあらすじから各エピソード3000字で生成 → ②1エピソードを序盤・中盤・終盤の3つに分割して詳細化したあらすじを生成 → ③詳細化あらすじをもとに1エピソード3話分を生成。3000字×3話×12エピソードで約10万8千字。

実践的なTipsとして：

- Sonnet4で一度に8000字くらいは生成できそうだが、文字数を守ってくれないこともあって、安全圏の3000字に抑える
- Sonnet4のコンテキスト・ウインドウ20万トークン（日本語で10万字くらい）を超えると確実に文脈は破綻するため、毎回メモリを参照してコンテキストに乗せる指示を入れる
- 「5話まで」と指定したのに4話以降は4-1, 5-1だけしか生成されないなど、Claude Codeはサボることがある

#### 2. Qiita「仕様駆動型小説開発（SDND）」(2026年3月)

これが今最も体系化された手法。35話・12万字を完走した実録です。

**3エージェントの権限分離**：Claude Code のサブエージェント機能（.claude/agents/*.md）で、執筆担当の Writer・品質検査担当の QA・正史管理担当の Editor を定義し、それぞれ書き込み可能なディレクトリを制限。Writer は story/ のみ、QA は qa/reports/ のみ、Editor は specs/ meta/ のみ。Writer が仕様を書き換えて辻褄を合わせることは構造的にできない。

**正史記録の3層分離**：前身プロジェクトでは全話分の Canon を1ファイルで管理し、68話で破綻した。その反省から quick_ref.md（最新状態の要約・約4KB）/ active/（直近5話の詳細記録・約82KB）/ archive/（それより古い記録・必要時のみ読み込み）に分離。直近5話だけを常に読み込む「スライディングウィンドウ」方式。

**品質検査駆動の仕様補記**：仕様は最初から完璧に書けるものではなく、品質検査が穴を見つけ、正史管理が仕様を追記するサイクルで育てる。Writer が書く → QA が「この状況は仕様に定義がない」と指摘する → Editor が仕様を追記する → 次回以降の Writer はその仕様に従う。ソフトウェア開発における「テスト駆動開発」ならぬ「品質検査駆動の仕様開発」が小説でも機能した。

筆者の結論は端的でした：LLM で長編小説を書くのに必要なのはプロンプトエンジニアリングではなく、ソフトウェアエンジニアリングだ。仕様を書く。実装する。テストする。テストが仕様の穴を見つけたら仕様を更新する。このサイクルは、対象がコードでも小説でも変わらない。

#### 3. moneymog氏「レビュー地獄」(2026年1月)

カクヨムコンテスト応募のために、初めての小説を、Claude Codeで書いた。10万字、全36話。仕事をしながら約2週間。異世界ファンタジーを執筆した記録。

なぜClaude Codeを選んだか：ファイル管理が楽だから。36話分のMarkdownファイル、設定資料、プロット、チェックリストを横断的に参照しながら執筆できる。Cursorでも同様のことはできるが、Claude Codeのほうがコンテキスト管理が直感的だった。

ただしタイトルが示す通り教訓は重い：1話あたり15分から30分かかる。36話で少なくとも9時間。実際はもっとかかった。修正すると別の箇所が壊れる。壊れた箇所を直すと、また別の箇所が壊れる。3回目の修正で頭を抱えた。無限ループだ。「Claude Codeにチェックリストを渡して自動チェックさせればいいじゃん」と思うだろう。やった。でも限界がある。チェック対象の文章が長いと、AIも見落とす。全話まとめて渡すとコンテキストが長すぎて精度が落ちる。

具体的な品質チェックリストも秀逸で、「AI臭削除」として、説明的ナレーション（「〜について説明しよう」）→ 描写で示す、過剰な内心描写（「私は〜と思った。なぜなら〜」）→ 行動で示す、定型的表現（「まるで〜のようだった」）→ 具体的な描写に置換、均質な文長 → 短文と長文を混ぜるといった項目を全話分チェック。

#### 4. マッハ・キショ松氏「Claude Codeを使ったAI小説生成システム」(2025年7月)

"AI作家"に自由な制作依頼を出せる仕組みを公開。生成物はGitHub Pages（f43a9a.github.io）で公開中。気付きとして「小説1本に対して複数の外部資料を読み込み、複数の内部資料を生成する」やり方なので、精度を上げたり長編を書こうとしたりするとトークン量がえらいことになると指摘。

#### 5. 葦沢氏の「Skills」記事

最近ではClaude CodeなどのAIエージェントでの小説執筆用Skillsの作り方について解説する記事を公開。skill-creatorでポン出しして満足する人も多いが、ちゃんとドメイン知識を身につけてディレクションしないと100%のパワーを引き出せないと発信。Anthropic Skillsを小説執筆に応用するフェーズに入っています。

### 共通する設計パターン

ここまでの事例を横断すると、ベストプラクティスが見えてきます：

1. **エージェントの役割分離**（Writer / QA / Editor）— サブエージェント機能で文脈と権限を独立させる
2. **ファイルベースの「不変仕様」**— 世界観・キャラ・ルールを `.md` で外部化、`CLAUDE.md` でメモリ化
3. **3層プロット**（全体プロット → 詳細あらすじ → 本文）の段階的詳細化
4. **スライディングウィンドウ**で直近のみ読み込み、過去はアーカイブ参照
5. **生成単位は3000字程度**に抑える（モデルがサボる/コンテキスト破綻を回避）
6. **品質検査が仕様を育てる**サイクル（テスト駆動開発の応用）
7. **`git` でバージョン管理**（コミット数139件など、コードと同じ運用）

### 興味深い哲学的な指摘

葦沢氏のスタンスが象徴的でした：人間が関与せずに、ほぼ全自動で小説ができてしまう状態になるので、小説の無責任な濫造につながってしまうというリスクがある。利用方法によっては、操作した人物に著作権が発生しない可能性がある。私自身の創作活動の最終目標として、私自身の作家性を骨格として通した状態で全てAIで代替させることを目指している。

「自分の作家性を骨格として通したまま全てAIに代替させる」——これがエージェント時代の創作のフロンティアになっていると感じます。
