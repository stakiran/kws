# Stacked Pull Requests とは

## 一言でいうと

新しい仕組みではない。**ブランチを積み重ねて使うワークフロー + その後始末を自動化したもの**のこと。

---

## 大原則

- main からブランチを切って、終わったら main に合流（マージ）する
- これが「出入り」の基本原則

---

## なぜ積み上がるのか

A の完成・合流を待たずに、A の上に B を、B の上に C を積むことがある。動機は2通り。

- **待てない**：開発が忙しく、A を合流する前に次の作業へ進む
- **分けたい**：A と B は論理的に別のまとまりでレビューも分けたい。でも B は A に依存しているので A 無しには成り立たない。巨大な1本の PR にするとレビューが大変なので、依存を保ったまま小さく分けて積む

```
main
 └─ A  → PR #1
     └─ B  → PR #2  (A から切った)
         └─ C  → PR #3  (B から切った)
```

結果として「main 合流前にブランチが積み上がった状態」が生まれる。

---

## 用語

| 用語 | 意味 |
|------|------|
| ベース | PR が「どのブランチと比べた差分か」の比較対象。PR #2 のベースは A |
| リベース | ブランチの土台を差し替える操作。「B を A の上から main の上に積み直す」 |
| スカッシュ | 複数のコミットを 1 個に潰すこと。履歴を綺麗にするためマージ時に使う |

---

## 何が問題になるのか

A をマージで取り込んだあと、B のベースが A のまま取り残される。

### 普通のマージなら問題なし
A のコミットがそのまま main に入るので、B のベースのコミットは「もう main にある」と git も分かる。差分は壊れない。
→ この場合、A → B → C と順に PR していくだけで、特殊なことは不要。

### スカッシュマージだと問題になる
スカッシュは A のコミットを潰して **別物の新しいコミット**を main に作る。中身は同じでも git にとっては別 ID。

- B が土台にしている A のコミット（元 ID）→ main には存在しない
- main にあるのは別 ID のコミット → 中身は同じだが git は同一視しない

→ git は「B には A 分の変更がまだ main に入っていない形で残っている」と誤認し、PR #2 の差分に A の中身まで混ざって表示される。
→ 直すには B を main にリベースし直す手作業が必要。

---

## ベース付け替えについての注意

一本道（A→B だけ）なら、A をマージしても B のベースを A のまま放置して差分は壊れない。**必ず付け替えが必要なわけではない。**

付け替える理由は「壊れるから」ではなく「実務上そうしたいから」。

- **A のブランチを片付けたい**：マージ済みブランチは普通削除する。でも B のベースが A だと A を消すと B が宙に浮く。だから先に B を main へ付け替える
- **B を単独で main にマージしたい**：ベースが A のままだと「B を A にマージする PR」扱いになる。main に出したいならベースを main に向ける

逆に、A を消さず B もまだマージしないなら、ベースは A のまま放置で問題ない。

---

## ツールの役割

Graphite、ghstack、git-spice、Sapling など。

やってくれるのは「マージのたびに後続のベースを付け替える／必要ならリベースし直す」という連鎖作業の自動化。

- スカッシュを使うチーム → この作業が重いのでツールの価値が大きい
- スカッシュしないチーム → 手作業でも十分回せる範囲

---

## まとめ

stacked PRs は、main 合流前に積み上がったブランチを破綻なく一本ずつ main に合流させるための段取り。原則（main 起点・main 合流）が一時的に崩れた状態を、内部的なベースの付け替え・リベースで担保しているもの。
