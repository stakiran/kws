# q

# a
ある。DI（Dependency Injection）ほど一般化していないものも多いが、「X injection」はセキュリティ／ソフトウェア設計の両方の文脈で広く使われる。

## 1) セキュリティ分野の「〜インジェクション」
「外部から入力を“注入”して、想定外の命令やデータとして解釈させる」タイプ。

- **SQL Injection**：入力をSQLとして解釈させて不正操作
- **Command Injection / OS Command Injection**：入力をシェルコマンドとして実行させる
- **Code Injection**：入力をコードとして実行させる（eval等）
- **HTML Injection**：入力がHTMLとして解釈される
- **XSS (Cross-Site Scripting)**：HTML/JS文脈でスクリプト注入
- **LDAP Injection**, **NoSQL Injection**, **XML Injection**, **XPath Injection**
- **Template Injection (SSTI)**：テンプレート式として評価させる
- **Header Injection / CRLF Injection**：HTTPヘッダ等に改行を注入
- **Log Injection**：ログの改ざん・偽装につながる注入

この系統は「injection」と言えばだいたい“脆弱性の類型”として通じる。

## 2) 設計・アーキテクチャ分野の「〜インジェクション」
DIと同じく「必要なものを外から与える（注入する）」という意味で使われることがある。

- **Constructor Injection**（コンストラクタ注入）
- **Setter Injection**（セッター注入）
- **Field/Property Injection**（フィールド/プロパティ注入）
- **Method/Parameter Injection**（メソッド/引数注入）
- （文脈によって）**Configuration Injection**（設定値の注入）など

※ただし、これらは「Dependency Injection の一種（注入のやり方の分類）」として語られることが多い。

## 3) もう少し広い一般用法
「データや要素を流れに差し込む」くらいの比喩で使うこともある（例：fault injection、prompt injection など）。ただし分野依存で意味が変わる。

# a
候補はありますが、英語圏で自然に通じやすいのは **Injection** より **Embed / Integrate / Plugin / Augment** 系です。とはいえ「xxx injection」として名付けたいなら、意味がブレにくく誤解（特にセキュリティの“インジェクション脆弱性”連想）を起こしにくい語を選ぶのがコツです。

以下、用途（既存ツールに“差し込んで”概念を使わせる）に合う候補を強い順で。

## 「xxx injection」案（比較的しっくり来る）
### 1) **Workflow Injection**
既存ツールの上に“作業の型”を注入する、が直感的に伝わる。  
あなたのやりたいこと（テンプレ、入力補助、表示補助、運用ルール）を全部「ワークフロー」として束ねられる。

### 2) **Capability Injection**
既存ツールに「能力（できること）」を足す、という意味で汎用。  
レンダラー追加、UI追加、検証、ナビゲーション強化などを包括できる。

### 3) **UI Injection / Interaction Injection**
「9マス入力UI」「ハニカム表示」など“操作・表示の差し込み”が中心ならこれが最短で伝わる。  
ただし範囲がUI寄りに限定される。

### 4) **Renderer Injection / View Injection**
「既存のMarkdownやCosenseを、別の見え方にする（マンダラ表示など）」を核にするなら強い。  
ただしこれも範囲が“表示”に寄りやすい。

### 5) **Constraint Injection**
「1マス1リンク」「構造制約を与える」みたいな“制約で品質を担保する”思想を前面に出せる。  
尖っていてコンセプト名としては良いが、一般には少し硬い。

## 「concept injection / function injection」はどうか
- **Concept Injection**：言いたいことは分かるが、英語としてはやや抽象的で「何を提供するのか」が伝わりにくい（思想啓蒙っぽく聞こえることも）。
- **Function Injection**：技術用語としては「関数を注入する」になりがちで、DI/FP文脈の連想が強く、あなたの意図（既存ツール拡張）とはズレやすい。

## もし “Injection” を外して良いなら（こちらの方が通じやすい）
- **Concept Embedding**（概念の埋め込み）
- **Workflow Plug-in** / **Workflow Layer**（ワークフローのプラグイン/レイヤ）
- **Augmentation**（拡張・増強。AI文脈でも一般語）
- **Tool Overlay**（既存ツールの上に重ねる）

## 結論（命名として無難で強い）
あなたの文脈だと、まずは

- **Workflow Injection**（既存ツールに“型”を注入する）
- 次点で **Capability Injection**（既存ツールに“できること”を注入する）

が、言いたい範囲を広くカバーしつつ誤解も少ないです。

どれを推したいかは、「注入する最小単位」を何に置くかで決めるのが良くて、確認したいです：
- 注入の中心は **表示（レンダラー）**？ **入力UI**？ **制約/検証**？ **テンプレ/運用**？
中心が分かれば、最も刺さる “xxx injection” を1つに絞って提案します。
