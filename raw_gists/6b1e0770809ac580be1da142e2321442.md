# alauncher についての Q&A

## Q1. https://github.com/stakiran/alauncher これは何？clone して見てみて。すごいの？

### これは何？

**alauncher**(stakiran 作)は、**Windows 用のエイリアス（コマンド別名）生成ツール**です。Go 製、コード約 600 行 + テスト 200 行のシンプルなツール。

仕組みは単純で、`.ala.ini` という 1 つの設定ファイルにエイリアスを書いておくと、`alauncher` コマンドを叩いた時にそれを読んで **`.bat` ファイル群を一気に吐き出す**だけ。あとはその出力先ディレクトリを PATH に通しておけば、コマンドラインから `pd` (= `pushd`) や `hide` (= 秀丸エディタ起動) みたいに使える、というもの。

例えば ini にこう書くと：

```ini
[pd]
rawbin=pushd %*

[hidemaru]
bin=%hidemaru%
prm=%*
alias=hide
```

`pd.bat`、`hidemaru.bat`、`hide.bat` が自動生成される、という具合。変数定義 (`[_variables]`)、空白を含めるための `%s%`、改行のための `%n%` といった気の利いた工夫もあります。

### すごいの？

正直に言うと、**技術的には地味**です。やっていることは「ini をパースして bat を書き出す」だけで、革新的なアルゴリズムも独自概念もない。スター数も控えめだと思います（GitHub 上で見ればすぐわかる規模）。

ただ「すごくないからダメ」かというとそうでもなくて、**「自分の不便を自分で解決した自作ツール」としての完成度はちゃんとある**タイプ。

- Windows の `doskey` は永続化が面倒、PowerShell プロファイルは cmd.exe で効かない、という長年の不便への素直な解。
- 設定が ini 1 つに集約されるので環境間で持ち運びやすい。
- bat を生成する方式なので、生成後はツール本体が無くても動く（ポータブル）。
- Go 1 ファイル + テスト付きで、読めば仕組みが全部わかる規模感。

なので「世界がひっくり返るプロダクト」ではなく「**Windows でガリガリ CLI 使う人がニヤッとする実用小物**」という位置づけ。あなたが Windows でコマンドプロンプトを多用するなら便利だし、そうでないなら特に刺さらない、というやつです。

## Q2. windows で似たことを行うのに、あえてこれを使う意味はあるの？doskey やその他の方法は？

### Windows でエイリアスを実現する主な方法

**1. `doskey`**
cmd.exe 組み込みのエイリアス機能。`doskey ll=dir` みたいに書く。

- 利点: 標準機能、即座に使える、軽い。
- 欠点: **そのセッション限り**。永続化するには `AutoRun` レジストリに `doskey /macrofile=...` を仕込むなど一手間いる。**cmd.exe 限定**で PowerShell では効かない。**他のプログラムから呼べない**（`doskey` は cmd の入力解釈段階で展開されるので、`make` から呼ぶ、タスクランナーから呼ぶ等ができない）。

**2. PowerShell の `Set-Alias` / プロファイル関数**
`$PROFILE` に `Set-Alias ll Get-ChildItem` や関数定義を書く。

- 利点: PowerShell ネイティブ、引数処理が強力、永続化が素直。
- 欠点: **PowerShell 限定**で cmd.exe では効かない。エイリアスは引数を取れないので、引数を渡したいなら関数を書く必要がある（書けば済む話ではある）。

**3. `.bat` / `.cmd` を PATH に置く**
PATH 内のフォルダに `pd.bat` を作って `pushd %*` と書く、という昔ながらの手。

- 利点: **どこからでも呼べる**（cmd, PowerShell, 他プログラムから、エクスプローラのアドレスバーから、even Win+R から）。再起動しても残る。仕組みがゼロ依存で堅牢。
- 欠点: エイリアスが増えると bat ファイルが散らかる、一括管理しづらい、ちょっと修正するのに各 bat を開く必要がある。

**4. WSL / Git Bash の alias**
`.bashrc` に `alias` を書く。Unix 流儀そのもの。

- 利点: bash の作法が使える、強力。
- 欠点: その shell の中限定。Windows ネイティブの cmd や Win+R には届かない。

**5. ランチャー系ツール（Keypirinha, Listary, PowerToys Run, AutoHotkey）**
- 利点: GUI ホットキー起動、ファジー検索、強力。
- 欠点: 「コマンドラインで `pd` と打つ」用途には向かない。GUI ランチャー文化。

### alauncher のポジション

alauncher は実質 **方法 3（bat を PATH に置く）の管理を ini 1 枚に集約したもの**です。つまり「bat 大量生成方式の利点はそのままに、欠点（散らかる、管理しづらい）だけを潰す」ことを狙っている。

なので **あえて使う意味があるのはこういう場合**:

- **cmd.exe と PowerShell の両方で同じエイリアスを使いたい**。bat 方式ならどちらからも呼べる。doskey も PowerShell プロファイルもこれは無理。
- **Win+R やエクスプローラのアドレスバーからもエイリアスを叩きたい**。これも bat が PATH にあるから可能。
- **エイリアス定義を 1 ファイルで管理して Git や Dropbox で同期したい**。`.ala.ini` を持ち運ぶだけで環境再現できる。
- **bat を量産する作業が面倒**。手で 50 個 bat 書くより ini に 50 行書く方が圧倒的に楽。
- **`%s%`（空白）や `%n%`（改行）みたいな小技が欲しい**。`prompt $$ ` の末尾空白とか、bat に直接書くと地味に面倒なやつを ini で素直に書ける。

逆に **使う意味が薄い場合**:

- **PowerShell しか使わない** → 素直に `$PROFILE` に関数を書く方が筋がいい。引数処理も柔軟。
- **エイリアスが 5 個程度** → bat を 5 個手で書けば済む。ツール導入コストが上回る。
- **WSL 中心の生活** → `.bashrc` で完結する。
- **GUI ホットキー起動が欲しい** → 別カテゴリのツール（PowerToys Run 等）の領分。

### 結論

alauncher は「**cmd でも PowerShell でも Win+R でも同じエイリアスを使いたい、かつ定義を 1 枚で管理したい、Windows ヘビーユーザー向けの管理ツール**」。doskey や `$PROFILE` ではカバーできない「シェル横断 + 一元管理」というニッチを、bat 生成という枯れた仕組みで埋めている、という設計判断です。あなたの普段の作業が cmd と PowerShell をまたぐなら検討の価値あり、片方しか使わないならそのシェルのネイティブ機能で十分、という線引きになります。

## Q3. windows の上級者向けって感じ？

うーん、「上級者向け」とは少し違うと思います。

正確に言うと **「Windows の CLI 文化に長く居る人」向け**、という感じ。技術的な難易度は全然高くなくて、ini に数行書くだけなので中級以下でも普通に使えます。bat の知識すら最低限でいい。

ただ、**そもそもこのツールが解決している不便を不便だと感じる人が、そう多くない**んですよね。

- 普段から cmd を開いて作業する
- `pushd` を `pd` で打ちたい、秀丸を `hide` で起動したい、みたいな短縮欲求がある
- cmd と PowerShell を両方使うので両対応したい
- エイリアスが数十個ある

…という人にしか刺さらない。最近の Windows ユーザーの多くは「ターミナルは VS Code か WSL の中だけ」「PowerShell もそんなに深く触らない」「ランチャーは PowerToys Run」みたいな使い方なので、そもそも cmd でエイリアスを大量管理したい場面に出くわさない。

なので正しく言うと:

- **難易度的には**: 中級者でも普通に使える。上級者専用ではない。
- **対象的には**: 上級者というより「**昔ながらの Windows CLI ヘビーユーザー**」「**bat や cmd を今でも現役で使っている人**」のニッチ層。

例えるなら「英語が難しいから上級者向け」ではなく「演歌好きの人向け」みたいな話で、レベルというより**嗜好と作業スタイルの問題**です。Linux で言えば、`.bashrc` を 200 行書いて alias と関数で武装している人が Windows でも同じことをやりたくなった時に手が伸びる、そういう道具。

逆に言うと、もしあなたがこれを見て「便利そう」と感じたなら、Windows CLI 文化の側の人だってことです。「ふーん」で終わるなら、たぶんあなたの作業スタイルでは別の解（PowerShell プロファイル、WSL の alias、PowerToys Run など）の方が自然に効きます。

## Q4. windows ではコマンドやプログラムを素早く起動できることは体験に直結する。プログラムランチャーというジャンルでも、キーワード入力型があった。その良き解になると思ったんだけど、こういう感想自体がニッチ？

いや、その感想は**全然ニッチじゃないし、むしろ核心を突いてます**。

ちょっと整理させてください。

### 「キーワード入力型ランチャー」という文脈

Windows のランチャー文化には大きく 2 系統あって：

**a) GUI 呼び出し型**: ホットキー（例: `Alt+Space`）でランチャーウィンドウを呼び出し、そこにキーワードを打つ。Keypirinha, Launchy, fenrir, PowerToys Run, bluewind など。検索・補完・履歴・スコアリングが効く。

**b) シェル直打ち型**: 既に開いているターミナルでコマンド名を打つ。doskey, bat, alauncher はここ。

あなたが言っている「キーワード入力型」は (a) の系譜で、**fenrir や bluewind みたいな日本の老舗ランチャー文化**は今も根強いファンがいます。「`hi[Enter]` で秀丸が起動」みたいな短いキーワードへの執着は、ニッチどころか **Windows ランチャー文化の本流のひとつ**。

### ただし alauncher の立ち位置はちょっとズレる

ここで気をつけたいのは、alauncher は (a) ではなく **(b) の道具** だという点です。つまり：

- (a) GUI ランチャーの「`Alt+Space → hi → Enter`」 (3 アクション、ターミナル不要)
- (b) alauncher の「ターミナルで `hi[Enter]`」 (ターミナルが既に開いている前提)

体験としては似ているけど、**前提条件が違う**。alauncher が刺さる人は「ターミナルを常に開いている人」で、これは GUI ランチャー派とは別の人種。GUI ランチャー派が alauncher を見ても「いや Alt+Space の方が速い」となる。

### で、あなたの感想がニッチかどうか

「**素早い起動が Windows 体験に直結する**」という感覚自体は **全然ニッチじゃない**、むしろ Windows をパワフルに使いたい人の自然な要求です。Mac の Alfred、Linux の rofi、Windows の PowerToys Run が支持されているのはみんなこの感覚があるから。

ただ「**その解として alauncher が良い**」という結論は、**前提が一段挟まる**：

1. ターミナル中心の生活をしているか？
2. cmd と PowerShell の両対応が必要か？
3. Win+R やエクスプローラのアドレスバーからも同じキーワードで叩きたいか？

この 3 つにイエスがある人にとっては alauncher は「良き解」。**そういう人がニッチ**なのであって、「素早い起動が大事」という感想自体はニッチじゃない。

### 別の見方をすると

あなたの感想を素直に汲むなら、本当に欲しいのは「**キーワード一発でプログラムやコマンドを起動できる体験**」であって、その手段は alauncher でも GUI ランチャーでも構わないはず。なら選び方はこう：

- 普段ターミナル開いてる → alauncher か doskey 系
- 普段ターミナル開いてない → PowerToys Run, fenrir, Keypirinha
- 両方欲しい → 両方入れる（実際これが最強で、ヘビーユーザーは両方併用してる）

なので「alauncher いいじゃん」と思ったあなたの感性は正しいし、その感性をもっと広げると GUI ランチャー側にも宝が眠っているはず、という話になります。Windows でキーストロークを削る世界はけっこう奥が深くて、ハマると楽しいジャンルですよ。
