# Dataplex系のデータカタログ（Entry / Aspect / EntryLink モデル） ← これは何？調べて

EntryLinkまわりをもう少し詳しく確認します。Google CloudのDataplex（メタデータカタログ）のメタデータモデルですね。調べた内容をまとめます。

まず前提として、2026年4月10日付けで「Dataplex Universal Catalog」は「Knowledge Catalog」に名称変更されています。なので最新のドキュメントだと「Knowledge Catalog」表記になっていますが、中身（Entry / Aspect / EntryLink モデル）は同じものです。

## これは何か

ひとことで言うと、Google Cloud上のデータ資産（BigQueryテーブル、Cloud Storage、さらにはオンプレや他社サービスの「カスタムリソース」まで）を統一的に記述・検索・関連付けするための、メタデータの抽象モデルです。旧来の「Data Catalog（タグ/タグテンプレート）」を置き換える、より柔軟な仕組みとして導入されました。

## 3つの中核概念

**Entry（エントリ）** が、カタログに登録される個々のメタデータ単位です。1つのテーブルや1つのファイルセットなどに対応します。Google CloudのリソースについてはBigQueryなら`@bigquery`のように、システムごとのエントリグループが各プロジェクト・ロケーションに自動作成され、エントリも自動生成されます。一方で、自分で管理する非Googleリソース（オンプレなど）は「カスタムエントリ」として手動で登録できます。

エントリは **Entry Group** というコンテナに属し、エントリグループはアクセス制御とリージョンの管理単位になります。また各エントリは **Entry Type** を持ち、エントリタイプは「そのタイプのエントリが必須で持つべきメタデータ」を、必須アスペクトタイプの集合として定義します。つまりエントリタイプでメタデータのスキーマ（最低限の項目）を強制できる、という設計です。

**Aspect（アスペクト）** は、関連するメタデータフィールドの集合です。エントリに付加する「属性のかたまり」と考えるとわかりやすいです。スキーマ情報、概要（overview）、連絡先（contacts）といったシステム定義のアスペクトタイプがあるほか、自分で「Aspect Type」を定義してカスタムなメタデータ項目を持たせられます。アスペクトタイプはメタデータテンプレート（record型のフィールド定義）として作成します。

ここで重要なのが役割分担で、Entry Type は「必須で持たせるアスペクトタイプ」を指定してスキーマを定義し、Aspect Type が実際のフィールド構造を定義し、Aspect がそのインスタンス（実データ）になります。

**EntryLink（エントリリンク）** は、2つのエントリ間のリンク（関係）を表すものです。これが比較的新しい概念で、エントリ同士の「意味のあるつながり」を表現します。各エントリリンクは **Entry Link Type** のインスタンスで、エントリリンクタイプは2つのエントリ間の関係の意味を記述する再利用可能なテンプレートであり、向き（directionality）はエントリリンクタイプのレベルで定義されます。

システム定義のエントリリンクタイプとしては、synonym（同義語）、related（関連）、definition（定義）、schema-join（スキーマ結合）があります。具体例だと、glossaryの同義語同士をつなぐsynonym、関連語をつなぐrelated、用語と実データ資産をつなぐdefinitionといった使い方です。テーブルのカラムとビジネス用語集の用語を結びつけて「カラムの意味」を説明するのにdefinitionタイプを使うのが典型的なユースケースです。さらにエントリリンク自体にもアスペクトを付けて関係性に関するメタデータを追加でき、エントリリンクタイプに必須アスペクトを要求させることもできます。

## 全体像（関係の整理）

階層をまとめると、こうなります。

- Entry Group ＞ Entry（Entry Typeに従う）＞ Aspect（Aspect Typeに従う）
- Entry ←→ EntryLink（Entry Link Typeに従う）←→ Entry

「型」と「実体」のペアで一貫して設計されている、というのが理解のカギです（Entry/EntryType、Aspect/AspectType、EntryLink/EntryLinkType）。

## 注意点として

**リネージとの違い**は混同しやすいポイントです。EntryLinkはあくまでカタログ上の「意味的な関係づけ」を手動・宣言的に張るものです。一方、データの変換フロー（upstream/downstream）は別物で、Data Lineage APIが対応システムから自動で受け取って生成するリネージグラフとして可視化されます。

権限まわりも独特で、「Dataplex Entry and EntryLink Owner」(roles/dataplex.entryOwner)というロールがエントリ操作・エントリリンク操作のフルアクセスを与え、schema-join型のエントリリンク作成権限も含みます。ただしDataplexのAdmin/Editor/Viewerロールは、エントリグループやエントリタイプ・アスペクトタイプといったメタデータリソースへのアクセスを付与しないので、別途カタログ系ロールが必要になる点は設計時に引っかかりやすいです。
