ここまでの議論、Q&Aをそのままの流れで整理します。

## Q. design.md って何？最近 agentic engineering の文脈で出たやつ。

AIエージェント向けに書かれたデザインシステムの仕様ファイル。Google Labs の Stitch プロジェクトから生まれ、最近 Google がオープンソース化した。カラー・タイポグラフィ・スペーシング・コンポーネントなどのデザイン選択を、AIエージェント(Claude Code、Cursor、v0 など)が読んで適用できる構造化形式で記述する。1ファイルが2部構成で、上部に機械可読なデザイントークン(hexコード、フォントサイズ等)、下部に人間向けの根拠やルールをマークダウンで書く。SKILL.md とペアで使われることも多い。

## Q. 要は CSS で指定するような細かいパラメーターを CLAUDE.md みたいに書いたって話？

方向性は合っている。ただし肝は2点。第一に「値(machine向け)」と「理由・ルール(human向け)」を1ファイルで分けている型がある点。第二に CLAUDE.md と違って**ツール非依存**を狙っていて、Cursor・v0・Codex など複数エージェントで使い回せる共通フォーマットとして提唱されている点。一言でいえば「CSSのトークンを、なぜそうしたかの理由とルール付きで、ツール横断のCLAUDE.md風に書いたもの」。

## Q. Google が OSS 化しただけで、別に品質が高いわけじゃないよね。むしろ Stitch の好みでしょ。

その見立ては正確。フォーマット自体は新しくなく、約1年前に Stitch の一部としてリリースされたものを、今回ようやく標準化しただけ。実態として今ちゃんと動くのは(当然ながら)Stitch で、デファクト標準になるには Figma や Cursor 等が乗ってくる必要がある段階。しかも仕様は alpha 版で、トークンスキーマもオープンな議論の最中、ダークモードやレスポンシブ等は未解決。OSS化の動機は品質ではなく、「誰も標準化しなければ各ツールが独自形式を作って分断する」のを防ぎ、OpenAPI のようなデファクト標準の座を取りにいくポジション取り。「Googleが出した=品質保証」では全くない。

## Q. 自チームでデザイナーと詰めて markdown 化すればいいと思うんだけど。新規性って machine readable と human readable を分けてる点くらいでしょ。

実践としてそのまま成立する。ただ「新規性のありか」は少しずれている。機械可読トークンと設計意図の分離自体も新しくなく、DTCG など既存のトークン標準が既にある(DESIGN.md の CLI も DTCG/Tailwind に export する)。本当の新しさは分離ではなく**置き場所と読まれ方**——リポジトリのルートに置き、エージェントが毎回参照する永続的コンテキストにした点。README.md や AGENTS.md がコードに対して持つ位置づけの、デザイン版。技術的発明ではなく配置と慣習の発明。だから自チームで詰めてルートに置きエージェントに毎回読ませれば便益のほぼ全部が得られ、残るのは lint/WCAG/diff の CLI や共通フォーマット共有といった付随メリットだけ。alpha のうちは厳密準拠の実利は薄く、「思想だけ借りて中身は自チームで詰める」が筋が良い。

## Q. ああ、デザイン定義という、エンジニアリングでは射程されづらかったものを明確に捉えた点に意義があるわけか。

その捉え方が一番しっくりくる。これまでデザインシステムは Figma の中・デザイナーの頭の中・口頭レビューに散在し、リポジトリには結果としての CSS 変数が落ちてくるだけで、背後の意図やルールはエンジニアリングの射程外だった。DESIGN.md はその曖昧な領域を、エンジニアリングが既に持つ器(ルートのテキストファイル、Git、lint、diff)に流し込んだ。さらにエージェント前提の時代と噛み合っている——人間だけなら「デザイナーに聞けばいい」で済んだ暗黙知が、毎回ゼロから生成するエージェントには明示的に書き出さないと再現できない。捉える対象(デザイン意図)と動機(再現性)が同時に顕在化した。技術的発明はほぼゼロでも、「言語化・配置の対象になっていなかったものに明確な居場所と形式を与えた」点に意義がある。
