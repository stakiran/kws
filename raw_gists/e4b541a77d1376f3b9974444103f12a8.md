# 決定論的シミュレーションテスト（DST）入門

分散システムの「1000万回に1回」のバグと戦うためのテスト技術、Deterministic Simulation Testing（DST）についての整理。FoundationDB が生み出し、Antithesis が汎用化した。

---

## 1. 出発点：なぜ普通のテストでは足りないのか

分散システムは **非決定論的** である。同じコードを同じ入力で動かしても、毎回まったく同じ動きをするとは限らない。実行結果に影響する「ランダムさ」が至る所に潜んでいるため。

- スレッド／プロセスの実行順序（OSスケジューラ次第）
- ネットワークパケットの到着順・遅延・喪失
- ディスクI/Oが完了するタイミング
- マルチコアでの並行処理のタイミング

これらが絡むと「特定の条件が偶然重なったときだけ起きるバグ」が生まれる。たまにしか再現しないため、見つけても直せない。

---

## 2. 「決定論的」とは

疑似乱数（pseudo-random）は本物の乱数ではなく、**初期値（シード）を決めれば以降の数列が計算で完全に決まる**数式にすぎない。

```python
import random
random.seed(42)
print(random.random())  # 0.6394267984578837 ← どの環境でも必ずこの値
```

- 同じシード → 必ず同じ乱数列（再現できる）
- 違うシード → 違う乱数列（別の展開）

DSTは、システムの動きを左右するあらゆる非決定要因を、このシード由来の乱数の制御下に置く。結果、**同じシードを与えれば1ビットの狂いもなく完全に同じ実行を再現できる**。

---

## 3. 「シミュレーション」とは

決定論を実現するには、現実のOS・ネットワーク・ディスクを使ってはいけない（制御できないから）。そこで、それらを **偽物（スタブ）に差し替える**。

- ネットワーク → 偽のネットワーク層（順序・遅延・喪失を自分で制御）
- ディスク → 偽のディスク（I/Oタイミングも制御）
- 時刻・タイマー → 偽の時計（`sleep` しても実際には待たず、仮想時間だけ進める）
- スレッド実行順序 → 実際はシングルスレッドで動かし、次に進める論理スレッドを乱数で決定

システム全体を一つのプロセスの中で、すべての非決定要因を握りながら動かす。

---

## 4. 具体例：`time.sleep` のループはDSTでどう動くか

```python
import time
for _ in range(100):
    time.sleep(1)
    print('a')
```

- **普通の実行**：現実の時計に従い、実時間で約100秒かかる。
- **DST**：`time` モジュールが偽物に差し替わっている。`sleep(1)` は「このスレッドを仮想時刻 t+1 まで停止」とスケジューラに登録して即座に制御を返す。誰も実際には待たない。仮想時計を一気に進めてスレッドを再開するため、**現実時間ではほぼ一瞬で終わる**。

これが「数年分のテストを数時間に圧縮」の正体。`sleep` で待たないからこそ可能。

### シードが効くのは「選択肢があるとき」だけ

上の1スレッドの例には**分岐点が一つもない**ので、乱数を引く場面がなく、シードは無関係。シードが意味を持つのは、シミュレータが「どっちにしようか」と選ぶ場面があるときだけ。

```python
# スレッドA: sleep(1) を3回 → print('A')
# スレッドB: sleep(1.5) を3回 → print('B')
```

この2スレッドが**同時刻に起きる**瞬間、初めて「どちらを先に処理するか」という選択肢が生まれ、そこで乱数を1個引く。

| 状況 | 選択の余地 | 乱数を引く？ | シードで変わる？ |
|---|---|---|---|
| 1スレッドで sleep ループ | ない | 引かない | **変わらない** |
| 複数スレッドが同時刻に起きる | ある | 引く | 変わる |
| 障害を注入するか選ぶ | ある | 引く | 変わる |
| ネット遅延を何msにするか | ある | 引く | 変わる |

> ⭕ 正しい理解：「選択の余地がある場面で、シミュレータはシードから引いた乱数で選ぶ。選択肢がなければシードは無関係」

## 5. シード = 世界線一個

```
                    ┌─ シード42 → 世界線①（A B A A B B、t=3で分断発生）
あなたのコード ──────┼─ シード43 → 世界線②（A A B B A B、障害なし）
（中身は不変）        ├─ シード44 → 世界線③（ディスク書き込み失敗）
                    └─ シード45 → 世界線④（...）
```

**コード自体は1ミリも変わらない。** 変わるのはシミュレータに注ぐシードだけ。ここから2つの強力な使い方が生まれる：

1. **しらみつぶしの探索**：シードを次々変えて何百万回も走らせ、人間が想像もしなかった障害の組み合わせを炙り出す。
2. **完全な再現**：バグを起こしたシード番号さえ記録しておけば、もう一度走らせるだけで寸分違わず同じバグが再現する。「特定のシードに紐付いた、確定的に再現できるもの」にバグが変わる。

これが分散システム全体に対する **「巻き戻せるデバッガ」**。

## 6. アーキテクチャ：中央の「世界観」に全スタブが問い合わせる

スタブ自身はほとんど何も判断しない。「どうすべきか」を中央の世界観に問い合わせ、返ってきた答えに従うだけ。賢いのはスタブではなく奥の世界観。

```
        ┌─────────────────────────────┐
        │   司令塔（シミュレータ本体）      │
        │   ・仮想時計 t                 │
        │   ・イベントキュー              │
        │   ・乱数（シードで初期化）        │
        │   ・障害注入ポリシー            │
        └─────────────────────────────┘
           ↑        ↑        ↑        ↑
       偽のsleep  偽のネット 偽のディスク 偽のスケジューラ
```

`sleep(1)` 一回の流れ：

```
コードが time.sleep(1) を呼ぶ
   ▼
偽のsleep（スタブ）は自分では決めず問い合わせる：
   ├─→ 仮想時計に聞く：「今いつ？」→「t=5.0」
   ├─→ イベントキューに登録：「俺をt=6.0に起こして」
   └─→ 司令塔に制御を返す（＝眠る）
   ▼
司令塔：キューを見て「次に一番早いイベントは誰だ？」
   ・候補が1つ → そのまま実行（乱数不要）
   ・同時刻に複数 → 乱数源に聞く「どれを先に？」
   ・障害を挟むかも乱数で決定
   ▼
選ばれたスレッドを再開、仮想時計をその時刻まで進める
```

> 普通のスタブが「部品の口を塞ぐ」ものだとすれば、DSTは **「世界そのものをスタブにする」**。

## 7. 何を「世界観」に据えるか

「何を据えるか」＝「何を非決定の源として中央が握るか」がDSTの設計の本丸。共通する思想は **「現実世界で勝手に・気まぐれに振る舞う要素を、ぜんぶ中央に引きずり込んで、シードで操れる従順な部品に変える」**。

| 据える対象 | 現実での非決定の正体 | 中央が握ると何ができる |
|---|---|---|
| 時間 | OS/ハードの時計 | 待たずに圧縮、タイムアウトを操作 |
| **実行順序** | OSスケジューラ・CPU | **競合状態を狙って踏む（心臓部）** |
| ネットワーク | 物理ネットワーク | 遅延・喪失・順序入替・分断 |
| ディスク | ストレージ装置 | 部分書き込み・破損・遅延・失敗 |
| ノードの生死 | 電源・ハード故障 | 任意の瞬間にクラッシュ／復活 |
| アプリ内乱数 | 言語/OSの乱数源 | 再現性を保つために回収 |

- **実行順序**が心臓部。競合状態（race condition）は「命令がどう交差するか」で起きるが、現実ではこの交差順を制御できない。DSTは中央のスケジューラが乱数で交差順を決め、意地悪な順序を狙って踏み、再現できる。
- **ネットワーク分断**は分散システムのバグの王様（通信が切れた間に両側が別々にリーダーを選ぶ等）。
- **ディスク**は「完了したと言いつつ書けていない」「部分書き込み」「サイレント破損」など、正直者である前提のコードを不正直なディスクで殴る。
- **アプリ内乱数**を中央に回収し損ねると再現性が壊れる。外部ライブラリ（例：RocksDB）が中央を経由せず自前でI/O・時間・スレッドを触ると非決定性が漏れる ── これが FoundationDB 方式の限界だった。

---

## 8. 障害判断も全部シード由来

ネット・ディスク・ノードの生死などの「いつ・どこで・どんな障害を起こすか」は、最終的にすべて中央の乱数で決まる。シードが唯一の入力なので、障害が出たらそのシードで完全再現できる。

```
シードが決まる
   → 乱数列が一意に確定
   → 全ての障害判断（ネット・ディスク・生死・実行順序）が一意に確定
   → 実行の全展開が一意に確定
   → 同じシードで同じ障害が同じ順序で寸分違わず再現
```

ただし乱数は生のまま使うのではなく、**「どんな障害をどれくらいの頻度・強さで起こすか」という方針（ポリシー）を通してから**使われる。

```
× 乱数 → 障害
○ 乱数 → 〔ポリシーというフィルタ〕 → 障害
```

理由：純粋な乱数だけだと、確率が高すぎればシステムが即崩壊して意味あるバグが出る前に止まり、低すぎれば何百万回走らせても障害がほぼ起きない。だから「パケット喪失は1%」「分断は平均30秒に一度5秒間」「killは1テストにつき最大2回」のような方針を乗せる。

**この方針の選択もシード由来**なので、再現性は1ミリも損なわれない。

> シードは「乱数列の種」であると同時に「そのテスト一回分の世界線まるごとの設計図」。サイコロの目だけでなく、「今日はどんなルールでサイコロを振る日か」まで、全部その一個の番号に畳み込まれている。

## 9. FoundationDB 方式 vs Antithesis 方式

「中央の世界観をどのレイヤーに据えるか」が両者の違い。

- **FoundationDB 方式**：世界観への問い合わせを **アプリのコードの中に** 作り込んだ。非決定的な部分を全部「差し替え可能」に設計。自前コードは完全に制御できるが、外部ライブラリは問い合わせてくれず漏れる。システムを最初からそう作る必要があり、既存の本番システムには後付けできない。
- **Antithesis 方式**：世界観を **OSより下のハイパーバイザ層に** 置いた。その上で動くプログラムが時計を見ようがスレッドを作ろうがI/Oしようが、全部いやおうなくハイパーバイザを経由する。書き換え不要で、外部ライブラリすら自動的に箱の中に入る。「DSTを大衆化する」というミッションの正体。

### 人と技術のつながり

- DSTの中核技術は FoundationDB（2009年〜、ACID準拠の分散DB）の内製テスト技術として生まれた。
- Apple買収（2015）後にチームが各社へ散り、最も洗練された組織にすらDSTが存在しないことに衝撃を受け、2018年に Will Wilson と Dave Scherer が Antithesis を創業。FoundationDB の COO だった Nick も参加。
- FoundationDB が「データベースを正しく作るために生んだ内製テスト技術」を、Antithesis が「あらゆるソフトに適用できる汎用プラットフォーム」として再構築した。
- 採用例：CockroachDB、WarpStream、TigerBeetle、Aiven（Kafka）など。

---

## 10. 一行まとめ

> **DSTとは「時間・順序・乱数・障害を握る中央の世界観を一つ用意し、システムのあらゆる非決定的な操作を、その世界観への問い合わせに置き換えること」。**

ある種、アプリのために **小さなOS（あるいは小さなコンピュータ）を決定論的な偽物として作り直している** とも言える。Antithesis がハイパーバイザ層まで降りたのは、まさに「小さなOSどころか、小さなコンピュータごと作ってしまえ」という発想だった。
