## 結論：ハシモト氏がGitHubを離れる理由

ミッチェル・ハシモト氏が「GitHubはもはや真剣に仕事に取り組める場所ではない」とまで言った理由は、単なる一時的な障害ではなく、**GitHub Actions、Pull Request、検索、ステータスチェックなど、開発の中核機能が連日のように不安定になり、実際の開発・レビュー・リリース作業を止められる状態が続いたから**だと考えられます。

特に同氏の最新プロジェクトである **Ghostty** では、GitHub ActionsやPull Requestレビューが重要な開発フローになっていたため、GitHubの不具合がそのまま開発停止に直結したのだと思われます。

要するに、彼の不満は次の一点に集約されます。

> ソースコードホスティングやCI/CDは、開発者にとって仕事の土台であり、そこが毎日のように壊れるなら、もう信頼できる作業場所ではない。

## 「バイブコーディング」時代の可用性リスクとは何か

記事タイトルにある「バイブコーディング」時代という表現は、AI支援開発、特にGitHub Copilotやエージェント型開発ツールによって、開発者やAIエージェントが大量のコード変更、ブランチ、Pull Request、CI実行を生み出す時代を指していると思われます。

GitHub側は公式ブログで、2026年4月後半の障害について、**エージェント型開発ワークフローの急激な増加**が背景にあると説明したようです。

つまり、従来は人間の開発者が手動で作っていたPull RequestやCI実行が、AIエージェントによって大量・高速・並列に生成されるようになり、GitHubのインフラに想定以上の負荷がかかった、という構図です。

記事中の見出しにある、

> Pull Request数は9000万に急成長

という部分から推測すると、GitHub上で作成されるPull Requestの数が急増し、レビュー結果、チェック結果、検索インデックス、Actionsの実行キューなどに大きな負荷がかかっていた可能性があります。

## GitHub側の釈明として考えられる内容

GitHubはおそらく、次のような説明をしているはずです。

- 2026年4月後半に複数のサービス障害が発生した
- GitHub Actions、Pull Request、検索、リポジトリ表示などに影響が出た
- 原因の一部は、AIエージェント型開発ワークフローの急拡大
- Pull Request作成数やCI実行数が急増し、既存のシステム設計に負荷がかかった
- 一部サービスでは遅延、結果の欠落、ステータスチェックの不整合などが発生した
- GitHubはユーザーに謝罪し、インフラ増強やシステム改善を進めると説明した

ただし、ハシモト氏の立場からすれば、GitHubの説明が正しかろうと、日々の開発が止まる事実は変わりません。

## 開発者コミュニティーで大きな反響を呼んだ理由

反響が大きかったのは、ハシモト氏が単なる一般ユーザーではなく、Vagrant、Terraform、HashiCorpで知られる著名開発者だからです。

しかも彼はGitHubを長年使ってきた初期ユーザーであり、GitHub文化を支えてきた側の人物です。その人が「もうGitHubでは真剣に仕事ができない」と言ったことは、単なる不満投稿ではなく、GitHubへの信頼低下を象徴する発言として受け止められたのだと思われます。

また、多くの開発者も近年、以下のような不満を感じていた可能性があります。

- GitHub Actionsが不安定
- Pull Requestのチェック結果が遅い、反映されない
- 検索が壊れる、遅い
- 通知やレビュー機能の信頼性が落ちている
- AI機能や新機能の追加に比べて、基盤の安定性が軽視されているように見える
- Microsoft傘下になって以降、GitHubの方向性に違和感がある

ハシモト氏の発言は、そうした不満の「代弁」として広がったのだと思います。

## 彼が移行先に求めているもの

記事の続きでは、おそらくGhosttyをGitHubから別の場所へ移行する話が出ているはずです。

移行先が具体的にどこかは記事本文を見ないと断定できませんが、候補としては以下のようなものが考えられます。

- self-hosted Git
- Forgejo
- Gitea
- SourceHut
- Codeberg
- GitLab
- 独自ホスティング

ハシモト氏の思想からすると、単に「GitHubの代わりにGitLabへ行く」というより、より小さく、制御可能で、信頼性を自分たちで担保できる環境を選びたい、という考えかもしれません。

つまり、これは単なる「GitHubで障害が起きた」という話ではなく、**AI時代の開発量の爆発に、GitHubの基盤が追い付いていないのではないか**という問題提起だと思われます。

## 方向性としては「VCS と周辺機能の再分離」が必要そう

新しい VCS プラットフォームのアイデアとしては、GitHub のような一枚岩ではなく、もっとこういう方向がありそうです。

```text
Git repository      : 分散・複製可能
Review metadata     : 分散・同期可能
Issue / PR          : ローカルファースト
CI/CD               : 外部プラグイン化
Artifact            : 別ストレージ
Search              : ローカルまたは分散インデックス
Identity / auth     : 署名ベース
Web UI              : 薄いビューア
```

つまり、GitHub のように「全部 GitHub が持つ」のではなく、**GitHub 相当の機能を疎結合な部品として扱う**。

---

## アイデア1：ローカルファーストな Pull Request

今の PR は GitHub のデータベース上のオブジェクトです。

でも本来、PR はこういう情報の集まりです。

- ベースブランチ
- 比較対象ブランチ
- 説明文
- コメント
- レビュー
- 承認
- CI 結果
- マージ判断

これらを Git のように、署名付きのメタデータとしてリポジトリ内、あるいは隣接する append-only log に保存できるようにする。

例えば：

```text
.repo/
  code...
.meta/
  prs/
    123.json
  reviews/
    123.alice.sig
    123.bob.sig
  ci/
    123-linux-x86.result
```

あるいは Git notes や別ブランチを使って、

```text
refs/meta/pr/123
refs/meta/review/123/alice
refs/meta/ci/123/build-linux
```

みたいに持つ。

そうすると、中央サーバーが落ちても、

- PR の内容を見る
- レビューを書く
- ローカルで CI を回す
- 後で同期する

ことができる。

GitHub の Web UI はそのメタデータを表示しているだけ、という形に近くなります。

---

## アイデア2：CI/CD を VCS プラットフォームから切り離す

GitHub Actions は便利ですが、GitHub の弱点にもなっています。

CI は本来、VCS と同じ会社が運用する必要はありません。

理想としては、

```text
VCS platform:
  - コード
  - PR
  - レビュー
  - 権限
  - Webhook / event stream

CI providers:
  - build
  - test
  - deploy
  - artifacts
```

のように分離する。

GitHub Actions 的なものはプラグインとして存在してよいけれど、VCS の中核ではない。

例えば PR に対する CI 結果は、外部 CI が署名付きで投稿するだけ。

```json
{
  "commit": "abc123",
  "check": "linux-build",
  "status": "success",
  "provider": "example-ci",
  "signature": "..."
}
```

VCS プラットフォームはそれを検証して表示する。

こうすると、GitHub Actions が落ちても、他の CI に切り替えられる。あるいはローカル runner や企業内 runner が結果を出せる。

---

## アイデア3：クライアントサイド重視の Forge

あなたが言っている「クライアントレンダリング」的な発想はかなり面白いです。

Web アプリでいう CSR は、ざっくり言えば「サーバーは API と静的ファイルを返し、UI の構築はブラウザ側でやる」方式です。

VCS プラットフォームでも近いことはできます。

例えば、

- リポジトリの clone はローカルにある
- diff 計算はクライアントで行う
- blame もクライアントで計算する
- 検索インデックスもローカルで作る
- PR の表示もローカルメタデータから生成する
- サーバーは同期・認証・公開だけ担当する

という形。

現在の GitHub は、巨大な中央サービスが、

- diff を計算する
- PR 状態を管理する
- 検索を提供する
- CI 状態を集約する
- 通知を配る

という構造ですが、これをもっとクライアント側に寄せる。

イメージとしては、

```text
今の GitHub:
  Browser -> GitHub巨大サーバー -> DB/Search/CI/Auth

新しい形:
  Local client -> local repo/local index/local PR metadata
       |
       +-> sync server
       +-> external CI
       +-> artifact store
       +-> identity server
```

この方式なら、中央サーバー障害時にもかなりの作業を継続できます。

## アイデア4：GitHub は「唯一の真実」ではなく「同期ノード」にする

今の GitHub は、多くのプロジェクトにとって single source of truth になっています。

でも Git の思想からすれば、本当は各 clone が完全な履歴を持っている。

そこで、新しい VCS プラットフォームでは、サーバーを「正本」ではなく「同期ノード」として扱う。

例えば、

```text
developer A
developer B
company forge
public mirror
CI node
release node
```

が互いに同期する。

GitHub のような中央 forge が落ちても、

- 社内 forge は動く
- public mirror は動く
- ローカル clone は動く
- CI node は動く
- 後で整合性を取る

という形です。

これはメールや Matrix、Mastodon に近い発想です。

---

## アイデア5：Issue や PR も Git と同じく分散管理する

GitHub から移行しづらい最大の理由は、実はコードではなく、

- Issue
- PR
- コメント
- レビュー履歴
- Star
- Discussion
- Release
- Wiki

などの周辺データです。

コードは `git clone --mirror` で持っていける。

でも Issue や PR は GitHub の DB に閉じ込められている。

だから新しい VCS プラットフォームでは、これらもポータブルにするべきです。

例えば標準仕様として、

```text
OpenDev Metadata Format
  - issues
  - pull requests
  - reviews
  - comments
  - labels
  - milestones
  - ci results
  - release notes
```

みたいなものを作る。

GitHub、GitLab、Forgejo、SourceHut、Radicle が全部その形式を読めるようになれば、プラットフォームロックインがかなり減ります。

## アイデア6：CI 結果を「信頼できる署名付き証明」にする

CI/CD の結果も、今は GitHub Actions の UI に依存しています。

でも理想的には、CI 結果は独立した証明書のように扱えるべきです。

例えば、

```text
commit abc123 は、
linux-x86_64 で test passed
macos-arm64 で test passed
security scan passed
reproducible build hash = ...
signed by ci.example.com
```

という情報を署名付きで配布する。

そうすれば、GitHub が落ちていても、クライアントはその CI 結果を検証できます。

これはソフトウェアサプライチェーンセキュリティにもつながります。

SLSA、Sigstore、in-toto、Rekor みたいな思想に近いです。

---

## アイデア7：Web UI は薄く、CLI/TUI/ローカルアプリを強くする

GitHub は Web UI 中心です。

でも真剣な開発では、ブラウザの巨大 Web アプリに依存しすぎるのも弱点です。

新しい VCS プラットフォームでは、

- CLI
- TUI
- ローカルデスクトップアプリ
- エディタ拡張
- Web UI

を同格にする。

例えば、

```bash
vcs pr list
vcs pr checkout 123
vcs pr review 123 --approve
vcs issue comment 456
vcs ci status abc123
```

が全部ローカルで動き、オンラインになったら同期される。

Web UI は便利なビューアであって、唯一の操作口ではない。

## あり得るプロダクト像

仮に新しい VCS プラットフォームを作るなら、コンセプトはこうです。

> GitHub 互換の便利さを持ちながら、Git の分散性を Pull Request、Issue、CI メタデータにまで拡張した、ローカルファーストな開発コラボレーション基盤。

名前をつけるなら、

```text
Local-first Forge
Distributed GitHub
Offline-first Dev Platform
Composable VCS Platform
```

みたいな感じです。

構成はこう。

```text
Core:
  - Git repository
  - signed identity
  - distributed metadata log

Client:
  - CLI
  - local web UI
  - editor integration

Server:
  - sync
  - auth gateway
  - public web view
  - permission policy

Plugins:
  - CI
  - package registry
  - search
  - security scan
  - deployment
```

GitHub と違って、サーバーが落ちても完全停止しない。

## ただし難しい点もある

もちろん難所はあります。

特に、

- 権限管理
- スパム対策
- 大規模検索
- 通知
- コンフリクト解決
- PR コメントの順序整合性
- セキュリティ
- 企業向け監査ログ
- 大規模 monorepo 対応
- UX の単純さ

あたりは中央集権のほうが作りやすいです。

GitHub が強いのは、単に機能が多いからではなく、**中央に集めたからこそ UX が簡単になっている**ところです。

分散型にすると、自由度は増えますが、ユーザー体験が複雑になりがちです。

だから新しい VCS プラットフォームは、完全分散というより、

> 普段は GitHub みたいに簡単。  
> でも障害時や移行時にはローカル・分散・別サービスで継続できる。

くらいが現実的だと思います。

## まとめると

あなたの問題意識はかなり妥当だと思います。

GitHub の弱点は、Git そのものではなく、**Git の上に巨大な中央集権的開発プラットフォームを作りすぎたこと**にある。

次世代の VCS プラットフォームは、おそらくこういう方向が必要です。

- VCS と CI/CD を分離する
- PR、Issue、Review をポータブルにする
- ローカルファーストにする
- サーバーは同期ノードにする
- CI 結果やレビューを署名付きメタデータにする
- Web UI だけに依存しない
- プラグイン可能な疎結合アーキテクチャにする

一言で言うなら、

> GitHub の便利さを保ったまま、Git の分散性を開発ワークフロー全体に拡張する

というのが、新しい VCS プラットフォームの有力な方向だと思います。
