# Tangled 調査まとめ

GitHub の分散型代替として注目される Tangled について、概要・基盤技術・批判の論点を整理したもの。

---

## 1. Tangled とは

- フィンランド（ヘルシンキ）拠点の **Tangled Labs Oy** が開発する、**AT プロトコル上に構築された分散型ソーシャルコーディングプラットフォーム**。
- 創業者はバンガロール出身の兄弟 **Akshay（ロンドン）と Anirudh Oppiliappan（ヘルシンキ）**。約1年前にスタート。
- 「GitHub の後継」というより、**分散型・開発者主権・欧州発**を掲げた代替（オルタナティブ）という位置づけ。GitLab の代替とも捉えられる。
- 機能面では Git ホスティング、ソーシャル機能（スター・Issue）、静的サイトホスティング（GitHub Pages 相当）、パイプライン実行（GitHub Actions 相当）が一通り揃う。
- 「**Knot（ノット）**」という軽量なセルフホストサーバーで自分のインフラ上にリポジトリを持ち、Tangled に federate（連合）できる。コードの完全な所有権が売り。

### 実績・資金調達

- 2025年3月に招待制で公開 → 2時間以内に Hacker News トップ入り。
- 8,000人超のユーザー、約6,000リポジトリ（オーガニック獲得）。
- €3.8M（約450万ドル）のシードラウンドを調達。
  - リード: byFounders
  - 参加: Bain Capital Crypto、Antler
  - エンジェル: **Thomas Dohmke（GitHub 元 CEO）**、Avery Pennarun（Tailscale CEO）、Mårten Mickos（MySQL / HackerOne 元 CEO）、Sami Honkonen
- 現状はまだ **アルファ版**。今後マイクロ仮想マシン、mission control ダッシュボード、GitHub からの移行ツール、CLI などを予定。

## 2. AT Protocol と ActivityPub の違い

両者は別物。どちらも分散型 SNS のためのプロトコルだが、出自も設計思想も異なる。

| 観点 | ActivityPub | AT Protocol（ATProto） |
|---|---|---|
| 出自 | 2018年、W3C 標準 | BlueSky 製（後発）。AT = Authenticated Transfer |
| 代表的な利用先 | Mastodon, Lemmy, PieFed（Fediverse） | Bluesky、そして Tangled |
| 重視する点 | 連合とコミュニティ統制 | グローバルな一貫性とアプリ間の相互運用性 |
| データ／ID | サーバー（インスタンス）単位。閉鎖されると失うリスク | PDS / PDR に自分のコンテンツと ID を保持、ホスト間で移動可能 |
| データ形式 | 柔軟だがプラットフォーム間で互換性問題が起きやすい | 強い型付けスキーマ＋検証で予測可能・相互運用しやすい |
| 課題 | 複数プラットフォームに分散するが互換性は限定的 | ほぼ全ユーザーが Bluesky に集中 → 一企業支配の懸念 |

- ActivityPub は「RSS とメールの中間」のような仕組み。
- ATProto の最大の売りは **アイデンティティとデータの可搬性**（携帯番号のキャリア乗り換えのように、ホストを変えてもコンテンツ・つながり・ID を保てる）。
- Forgejo の ActivityPub ベース連合との実際的な違いは、**ID がインスタンス単位ではなくプロトコル単位で存在する**こと。

## 3. Tangled 陣営が GitHub を問題視する点

> 「時間が経っても本質的な改善がほとんどない、老朽化したプラットフォーム」 —— Anirudh Oppiliappan

ポイント: 批判の力点は **可用性（アップタイム）や AI リクエストの処理性能ではない**。
**構造・所有権・イデオロギー**、そして **ワークフロー設計の古さ** に向けられている。

### 3-1. Microsoft 買収後の停滞

- 2018年の **75億ドルでの Microsoft 買収** 以降、製品イノベーションとイデオロギーの両面で「慢心（complacency）」が広がったとの認識。
- 特に AI・Copilot まわりで多くの "Microsoft-ism" が染み込んだと批判。
- コード共同作業の領域は **10年以上、本質的に進化していない** という問題意識が出発点。

### 3-2. プルリクエストモデルが2009年から進化していない（最も具体的な「ない」）

- GitHub は2009年に PR を導入、当時は革命的。だがそのモデルが **今日のコード量、特に AI 支援・エージェント型プログラミングの時代に合っていない**。
- 欠けている具体機能 = **Stacked PR（積み重ね型 PR）**。
  - Google / Meta では巨大 PR ではなく、小さなアトミックな変更を積み重ね、各々を独立レビューする方式が当たり前。
  - Tangled は元 Meta のバージョン管理ツール担当エンジニアと共同で実装。最大の魅力の一つに。

### 3-3. API 制限がエージェントの足枷（AI 文脈での「ない」）

- 「捌けない（処理性能）」ではなく、**アーキテクチャ的にエージェントを締め出している** という指摘。
- GitHub は API 制限とアクセスルールに縛られる。
- Tangled は AT Protocol 上にあるため、人間でもエージェントでもプロプライエタリ API に依存せず、リポジトリ作成・PR 提出・レビューが可能。実際に自律エージェントがリポジトリ作成・push する例も観測済み。
- 投資家側も「ソフトウェア開発のボトルネックは、書くことから **スケールしてレビュー・管理すること** へ移った」と認識。

### 3-4. 知的財産の所有権 / コードの AI 学習利用

- 多くの開発者が気づいていないが、**コードが GitHub にあると規約上 Microsoft が AI 学習に使う法的権利を持つ**。
- これを受け入れられない開発者・組織のために、自分のインフラでホストし自分の条件で共有できる仕組みを提供。
- データ主権・法管轄（世界のコードの大部分が米国に保管・米国法域で統治）の問題意識とも接続。

### 3-5. 買収後の「ハイパーエンタープライズ化」

- 買収後、GitHub はエンタープライズ寄りに振り切り、**インディー開発者・オープンソースコミュニティが手薄（underserved）** になった。
- Tangled は「インディー開発者とオープンソースコミュニティをまず第一に」を core thesis に。

---

## 4. 注意点（バランス）

- 上記はあくまで **Tangled 創業者の主張** であり、競合としてのポジショントークを含む。
- GitHub 側も近年 Copilot やエージェント関連機能を投入しているため、「本質的な改善がない」という評価が客観的に妥当かは、実際の機能比較を別途見るのが公平。
