## Q1. tokenmaxxing や turnmaxxing に、engineering effectiveness の考え方を取り入れたい。まず engineering effectiveness ってわかる？

### A1.

当初は一般論として、Engineering Effectiveness を次のように捉えた。

> エンジニアや開発組織が、より少ない摩擦で、より良い成果を、継続的に出せる状態をつくる考え方。

単なる「個人の生産性」ではなく、開発速度、品質、認知負荷、開発者体験、フロー効率、学習効率、意思決定、持続可能性などを含む。

ただし、この回答は一般論に寄りすぎていた。

---

## Q2. いや、違う。OkayHQ の ee-handbook を読める？

### A2.

読める。

OkayHQ の handbook における Engineering Effectiveness は、より明確に、

> エンジニア個人のアウトプットを測ることではなく、エンジニアのワークフローにあるボトルネックや非効率を取り除くこと

として捉えられる。

ポイントは、LOC、チケット数、個人PR数のような output metric ではなく、inputs に注目すること。

ここでいう input は、レビュー待ち時間、ツール品質、ドキュメントアクセス、割り込み、オンコール、集中時間など、エンジニアの仕事の流れに影響するもの。

また、個人評価や監視に使うべきではない。  
Engineering metrics を評価プロセスに入れると、surveillance や gaming を生み、生産性を傷つける。

したがって tokenmaxxing / turnmaxxing に接続するなら、悪い接続はこう。

> 誰が一番トークンを使ったか。  
> 誰が一番AIとやりとりしたか。  
> 誰が一番AIを回したか。

これは個人ランキングになり、OkayHQ 的には危ない。

良い接続はこう。

> AI利用によって、どの feedback loop の latency / friction / context switch が減ったか。

つまり、Tokenmaxxing ではなく、AI feedback loop effectiveness として扱うのが近い。

## Q3. engineering effectiveness は、maker time という望ましいものを最大化するために、maker time を阻むもの（会議や割り込み）を計測するってことだと思う。feedback loop は、後者の方の計測指標になるんだよね。

### A3.

その理解の方がかなり正確。

OkayHQ の文脈では、中心にあるのは次の構造。

> 望ましい状態：エンジニアが Maker Time を十分に持てること。  
> 計測するもの：Maker Time を削る要因や、仕事を前に進めるループの摩擦。

Engineering Effectiveness は、Maker Time を直接「成果」として崇めるというより、

> Maker Time を増やすために、会議・割り込み・レビュー待ち・CI待ち・調査迷子などの friction を測って潰す

という考え方。

その意味で、feedback loop は「理想状態そのもの」ではなく、

> Maker Time 中の前進を遅くする摩擦、または Maker Time の外でエンジニアを待たせる摩擦

を見つけるための観測単位。

整理するとこう。

| レイヤー | 例 | 位置づけ |
|---|---|---|
| 望ましい状態 | Maker Time / Flow state | 最大化したいもの |
| 阻害要因 | 会議、割り込み、Slack、オンコール、レビュー待ち、CI遅延 | 減らしたいもの |
| 計測単位 | feedback loop の latency / frequency / usability | どこが詰まっているかを見るもの |
| 改善対象 | CI高速化、レビューSLA、会議削減、通知設計、AI補助 | Maker Time を守るための介入 |

tokenmaxxing / turnmaxxing に接続するなら、こう言い換えられる。

> Turn 数を増やすこと自体が目的ではない。  
> AI feedback loop を導入することで、Maker Time を増やす、または Maker Time の質を上げることが目的。

AIとのやりとりが多くても、それが割り込み化して Maker Time を細切れにしているなら悪い。  
逆に、AIとのやりとりが少なくても、レビュー前の自己検証、CI失敗調査、設計の抜け漏れ確認が速くなり、まとまった作業時間が増えるなら良い。

したがって、

> tokenmaxxing / turnmaxxing を Engineering Effectiveness 化するとは、AI利用量を測ることではなく、AIが Maker Time を阻害する friction をどれだけ減らしたかを見ること。

## Q4. では、どのような指標や手法を導入したら、feedback loop という friction を測れるだろうか？

### A4.

まず前提として、feedback loop は「friction そのもの」ではなく、friction を観測する単位として扱うのがよい。

導入すべき指標は、「AIを何回使ったか」ではなく、

> ある作業ループが、どれくらい頻繁に発生し、どれくらい待たされ、どれくらい使いにくく、何の前進を阻んでいるか

を見るもの。

---

### 測るべき feedback loop の候補

| Loop | 典型的な friction | 測るべきこと |
|---|---|---|
| 仕様理解 loop | 仕様が曖昧、関連資料が見つからない | 調査時間、質問回数、手戻り |
| コード理解 loop | 影響範囲が分からない | 関連ファイル探索時間、識者依存 |
| 実装 loop | 環境構築、ビルド、型エラー | ローカル待ち時間、失敗回数 |
| テスト loop | テストが遅い、不安定 | 実行時間、flake率、再実行回数 |
| CI loop | CI待ち、失敗原因不明 | CI latency、再実行、原因特定時間 |
| PR review loop | レビュー待ち、指摘往復 | first response time、review round数 |
| リリース loop | 承認待ち、手順不明 | lead time、rollback、手順確認 |
| 障害対応 loop | ログ探索、原因仮説出し | MTTD/MTTR、仮説検証回数 |
| AI assistance loop | AI出力待ち、質が低い、文脈渡しが重い | AI latency、採用率、再プロンプト回数 |

---

### 指標セット1：Latency

どれだけ待たされているかを見る。

| 指標 | 意味 |
|---|---|
| PR first response time | PRを出してから最初の反応まで |
| PR time to approval | PR作成からapproveまで |
| PR time to merge | PR作成からmergeまで |
| CI queue time | CIが走り始めるまで |
| CI duration | CI完了まで |
| CI failure triage time | CI失敗から原因把握まで |
| review idle time | PRが誰にも触られず止まっている時間 |
| question response time | Slack/Issueで質問してから回答まで |
| doc discovery time | 必要な情報を見つけるまで |
| AI response latency | AIに依頼してから使える出力が来るまで |

平均値ではなく、p50 / p75 / p90 を見る。  
friction は「普段は平気だが、たまにめちゃくちゃ詰まる」が痛いので、p90 が重要。

---

### 指標セット2：Frequency

どれだけ頻繁に発生しているかを見る。

| 指標 | 意味 |
|---|---|
| PR数 / engineer / week | review loop の発生頻度 |
| CI実行回数 / PR | CI loop の多さ |
| test rerun回数 | テスト不安定性の兆候 |
| review round数 | 指摘往復の多さ |
| Slack質問数 | 情報探索 friction の多さ |
| blocked状態の発生回数 | 自走不能ポイント |
| AI turns / task | AI feedback loop の頻度 |
| AI-assisted checkpoints / PR | 実装前、レビュー前、テスト前などでAIを挟んだ回数 |

turnmaxxing を入れるなら、単なる総ターン数ではなく、

> 1つの開発タスクあたり、どの工程でAIターンが発生したか

を見る方がよい。

---

### 指標セット3：Usability

その loop が使いやすいかを見る。

| 指標 | 質問例 |
|---|---|
| perceived review friction | レビュー待ちは作業の妨げになったか？ |
| CI trust score | CI失敗は信頼できるシグナルだったか？ |
| doc findability | 必要な情報をすぐ見つけられたか？ |
| AI usefulness score | AI出力は次の行動に使えたか？ |
| context handoff quality | 中断後に再開しやすかったか？ |
| interruption pain | 集中を壊す割り込みが多かったか？ |

5段階評価で十分。  
精密にやりすぎると、計測自体が friction になる。

---

### 指標セット4：Outcome proxy

feedback loop の改善が、最終的に Maker Time の増加や flow の改善につながっているかを見る。

| 指標 | 意味 |
|---|---|
| Maker Time % | 週のうち2時間以上の空きブロックが占める割合 |
| fragmentation score | 1日の会議・予定の散らばり具合 |
| interruption count | Slack/会議/相談による中断回数 |
| focus block survival rate | 確保した集中枠が実際に守られた割合 |
| after-interruption recovery time | 中断後に作業復帰するまで |
| WIP per engineer | 並行タスク数。多いほど認知負荷が高い |
| blocked time / week | ブロックされて進めない時間 |

AI活用と接続するなら、見るべきはこう。

> AI turns が増えた結果、Maker Time が増えたか？  
> それとも AI との会話が新しい割り込み源になったか？

---

### 最初に導入するならこの8指標

| カテゴリ | 指標 | なぜ見るか |
|---|---|---|
| Maker Time | Maker Time % | 最終的に守りたいもの |
| Fragmentation | 1日あたり会議分断数 | 集中時間を壊していないか |
| PR Review | PR first response time p75/p90 | レビュー待ち friction |
| PR Review | review round数 | 指摘往復 friction |
| CI/Test | CI duration p75/p90 | 実装 feedback の遅さ |
| CI/Test | flaky test rerun率 | 信頼できない feedback |
| Knowledge | blocked by unclear info 件数 | 仕様・ドキュメント friction |
| AI Loop | AI-assisted checkpoint率 | AIがどの工程に入っているか |

### AI-assisted checkpoint率

tokenmaxxing / turnmaxxing と Engineering Effectiveness をつなぐ指標として有効。

```text
AIを使った工程:
[ ] 仕様理解
[ ] 影響範囲調査
[ ] 実装方針
[ ] コード生成
[ ] 自己レビュー
[ ] テスト観点
[ ] CI失敗調査
[ ] PR説明
[ ] 使っていない

AI利用で短縮できたもの:
[ ] 調査
[ ] 実装
[ ] レビュー
[ ] テスト
[ ] ドキュメント
[ ] 特になし
````

目的は個人評価ではなく、チームの friction map を作ること。

---

### 手法1：イベントログで latency を取る

GitHub / GitLab / CI / Jira / Slack から、自然に残る timestamp を使う。

```text
PR opened
→ first review/comment
→ changes requested
→ author response
→ approved
→ merged
```

ここから以下を取る。

* first response time
* time in review
* time waiting for author
* time waiting for reviewer
* number of review rounds
* idle time

---

### 手法2：カレンダーから Maker Time と fragmentation を取る

2時間以上の空きブロックを Maker Time として集計する。

```text
Maker Time % = 2時間以上の空きブロック合計 / 勤務時間
Fragmentation = 会議ブロック数 + 会議間の短すぎる隙間数
```

会議時間の総量だけでは足りない。

```text
良い例:
9:00-13:00 会議、午後まるごと空き

悪い例:
9:00, 10:30, 13:00, 15:00 に30分ずつ会議
```

同じ会議量でも、後者の方が Maker Time を壊す。

---

### 手法3：週1の friction pulse survey

ログでは「なぜ詰まったか」は分からない。
週1で短いサーベイを入れる。

```text
1. 今週、最も作業を止めた friction は何ですか？
- 会議
- 割り込み
- レビュー待ち
- CI/テスト
- 仕様不明
- コード理解
- 環境問題
- リリース手順
- AI出力の質
- その他

2. それによって失った時間はどれくらいですか？
- 0-30分
- 30分-2時間
- 半日
- 1日以上

3. それはAIで軽減できそうですか？
- できそう
- 一部できそう
- できなさそう
- むしろAIが原因
```

---

### 手法4：Friction Review を隔週でやる

通常のレトロとは別に、計測された friction だけを見る。

```text
今週/隔週で最も大きかった待ち時間は何か？
それは頻度が高いのか、p90だけ悪いのか？
Maker Time を削っているのか？
AI・自動化・ルール変更・会議設計のどれで潰すか？
```

問いは「誰が悪いか」ではなく、「どこに構造的な詰まりがあるか」にする。

---

### AI / turnmaxxing 用の指標

| 指標                      | 良い使い方                      | 危ない使い方        |
| ----------------------- | -------------------------- | ------------- |
| AI turns / task         | 工程ごとのAI介入頻度を見る             | 個人ランキングにする    |
| AI-assisted checkpoint率 | AIが自己レビューやテスト観点に入っているか     | チェック数を評価に使う   |
| AI rework reduction     | AIでレビュー指摘や手戻りが減ったか         | AIのおかげと断定する   |
| AI latency              | AI待ちが新たな friction になっていないか | 高速モデル利用を強制する  |
| AI output adoption      | 出力が実作業に使えたか                | 採用率で個人評価する    |
| AI interruption score   | AIが集中を壊していないか              | とにかく常時AIを使わせる |

特に良さそうなのは AI-assisted checkpoint率。

```text
AI-assisted checkpoint率 =
AIを使った工程数 / 対象工程数
```

対象工程の例。

```text
仕様理解
設計
実装
自己レビュー
テスト
PR説明
CI失敗調査
```

目的は AI 利用率を上げることではなく、

> AIを入れた工程で、latency / rework / uncertainty / interruption が下がったか

を見ること。

### 最初の実験

1チームで4週間だけやる。

#### Week 1：baseline

* Maker Time %
* PR first response p75
* CI duration p75
* review round数
* 週次 friction survey

#### Week 2-3：AI intervention

以下を導入する。

* PR前AI自己レビュー
* CI失敗時のAI原因要約
* 仕様理解用のAI質問テンプレ
* PR説明のAI生成
* 作業終了時のAI handoff note

#### Week 4：比較

```text
Maker Time は増えたか？
PR first response 待ちは減ったか？
review round は減ったか？
CI失敗の原因特定は速くなったか？
「AIが役に立った」ではなく「friction が減った」と言えるか？
```

結論。

> turnmaxxing を測るのではなく、turn が入った feedback loop の摩擦が減ったかを測る。

## Q5. この指標が良い値を示していたとしても、たくさんのマルチタスク、たとえば6つの作業を並行している状態があったら、作業間を高頻度で切り替える形になっているだろう。これは maker time とは呼べない気がする。これの検出も要る気がする。

### A5.

かなり重要。
これは Maker Time 指標の抜け穴。

カレンダー上は2時間空いている。
PR first response も速い。
CIも速い。
AI checkpoint も入っている。

でも本人はその2時間で、

```text
作業A → 作業B → Slack → 作業C → AI返答確認 → 作業D → PRコメント対応 → 作業Aに戻る
```

のように泳がされているかもしれない。

これは見かけ上は「空き時間」でも、実態は fragmented maker time で、Maker Time とは呼びにくい。

---

### 追加すべき概念：Task Switching / WIP Fragmentation

指標名としては以下が考えられる。

* WIP Fragmentation
* Task Switching Rate
* Context Switching Load
* Maker Time Purity
* Focus Continuity

OkayHQ の文脈に乗せるなら、

> Maker Time Purity

がしっくりくる。

定義はこう。

> カレンダー上の Maker Time のうち、実際に単一の目的・単一の作業文脈に使えた割合。

---

### なぜ必要か

Maker Time は「会議がない時間」ではなく、

> 深い文脈を頭にロードして、前進できる時間

である。

同じ2時間でも中身が違う。

```text
良い2時間:
Task A → Task A → Task A → Task A
```

これは Maker Time。

```text
悪い2時間:
Task A → Task B → Task C → Slack → Task D → Task A
```

これは空き時間ではあるが、認知的には細切れ。

したがって、従来の Maker Time % に加えて、

> その Maker Time がどれだけ pure だったか

を測る必要がある。

---

### 追加指標1：Active WIP Count

```text
Active WIP Count = 直近N日で実際に触った作業文脈の数
```

1日単位なら、

```text
今日触った issue / PR / branch / design doc / incident / support ticket のユニーク数
```

を見る。

目安。

| Active WIP / day | 解釈              |
| ---------------: | --------------- |
|              1-2 | 深い集中がしやすい       |
|              3-4 | やや分散            |
|              5-6 | かなり分散           |
|               7+ | ほぼ割り込み/マルチタスク状態 |

ただし、EM、SRE、platform support、on-call は高くなりがちなので、ロール別に見る。

---

### 追加指標2：Task Switching Rate

```text
Task Switching Rate = 1時間あたりの作業文脈変更回数
```

例。

```text
10:00-10:25 Issue A
10:25-10:40 PR Review B
10:40-11:00 Slack incident C
11:00-11:20 Issue A
11:20-11:35 AI response check for D
11:35-12:00 Issue A
```

この2時間はカレンダー上は Maker Time でも、切り替えは多い。

目安。

| switches / hour | 解釈                   |
| --------------: | -------------------- |
|             0-1 | deep work            |
|             2-3 | fragmented           |
|              4+ | context-switch heavy |

---

### 追加指標3：Focus Run Length

```text
Focus Run Length = 同じ作業文脈を連続して触っていた時間
```

Maker Time が2時間あっても、Focus Run Length が15分なら、その2時間は深い作業ではない。

見るべきもの。

```text
p50 Focus Run Length
p75 Focus Run Length
max Focus Run Length / day
```

特に、

> 1日の中で最長の Focus Run が90分未満

なら、深い作業ができていない可能性が高い。

---

### 追加指標4：Maker Time Purity

総合指標。

```text
Maker Time Purity =
Maker Time ブロック内で、主作業に使われた時間 / Maker Time ブロック全体
```

例。

```text
2時間の Maker Time ブロック
- Issue A: 80分
- PR Review B: 15分
- Slack C: 10分
- AI D: 10分
- その他: 5分

Maker Time Purity = 80 / 120 = 66.7%
```

目安。

| Purity | 解釈                 |
| -----: | ------------------ |
|   80%+ | 良い Maker Time      |
| 60-80% | まあ使えている            |
| 40-60% | fragmented         |
|   <40% | Maker Time とは呼びにくい |

---

### 追加指標5：Return Cost / Resume Latency

シングルタスク脳にとって痛いのは、切り替え回数だけではなく、

> 元の作業に戻るまでの再ロード時間

である。

```text
Resume Latency =
中断後、元の作業で意味のある変更/判断を再開するまでの時間
```

例。

```text
10:00-10:40 Task A
10:40-10:55 Slack interruption
10:55 Task A に戻る
11:10 最初の meaningful commit / edit / comment

Resume Latency = 15分
```

実測が難しければ、日次セルフレポートでもよい。

```text
今日、中断後に元の作業へ戻るのが大変だった回数:
0 / 1-2 / 3-5 / 6+
```

---

### 追加指標6：AI-induced Switching

turnmaxxing 文脈では必須。

AIを使うと、AIの返答が非同期に返ってくる。
すると人間は、

> 今やっている作業を止めて、別スレッドのAI返答を確認する

という動きになりやすい。

```text
AI-induced Switching =
AI返答確認をきっかけに発生した作業文脈変更数
```

良い状態。

```text
Task A の中で AI を使う
→ Task A の判断が進む
```

悪い状態。

```text
Task A 中
→ AI返答Bが来る
→ Bを見る
→ CのAIも返ってきたので見る
→ Slackを見る
→ Task Aに戻れない
```

これは turnmaxxing ではなく、AIによる注意の分散。

---

### 追加指標7：Parallel AI WIP

```text
Parallel AI WIP =
同時に進行中のAI支援タスク数
```

より具体的には、

```text
現在AIに投げていて、まだ人間判断が未完了の作業文脈数

### どう測るか

完全自動化しようとすると監視っぽくなるので、慎重にやるべき。
おすすめは、軽量な自己申告 + 既存ログの集計。

#### 既存ログから取る

| Source              | 推定できるもの                        |
| ------------------- | ------------------------------ |
| Git branch / commit | 作業文脈                           |
| PR comment          | review context                 |
| Issue status change | task touch                     |
| Slack thread reply  | interruption / support context |
| Calendar            | available Maker Time           |
| AI session metadata | AI task context                |
| IDE active file     | 作業文脈。ただし監視感が強いので注意             |

やるなら個人向けローカル可視化に留めるのがよい。

### WIP が多いことを悪と決めつけない

WIP が多い人には、以下も含まれる。

* tech lead
* reviewer
* on-call
* platform owner
* incident commander
* support rotation

したがって、個人評価に使うと危険。

見るべき問いは、

> なぜこの人/チームは6個も並行せざるを得ないのか？

原因の例。

* レビュー依頼が集中している
* 知識が属人化している
* オンコールが重い
* PRが大きすぎる
* 仕様確認が同期的に飛んでくる
* AIエージェントを走らせすぎて人間レビュー待ちが積まれている
* チームのWIP limitがない

### Kanban 的に WIP limit を入れる

検出だけでなく、制御も必要。

```text
個人のActive WIPは原則2まで
Parallel AI WIPは原則2まで
レビューWIPは午前/午後のバッチで処理
AI返答確認は通知ではなくチェックポイントで処理
```

特にシングルタスク型には、

> AIを非同期通知にしない。
> AI返答は決まったタイミングで回収する。

が効く。

```text
悪い:
AI返答が来るたびに見る

良い:
25-50分のFocus Run後にAI返答をまとめて見る
```

AIをSlack化しない、ということ。

---

### tokenmaxxing / turnmaxxing に入れるなら

定義を修正するとこう。

> 良い turnmaxxing とは、AIとの feedback loop を増やしつつ、人間の Focus Continuity を壊さないこと。

指標はセットで見る。

| 増やしたい                   | 減らしたい                |
| ----------------------- | -------------------- |
| Maker Time %            | Task Switching Rate  |
| Maker Time Purity       | Parallel AI WIP      |
| Focus Run Length        | AI-induced Switching |
| AI-assisted checkpoint率 | Resume Latency       |
| Feedback loop latency改善 | Active WIP Count過多   |

結論。

> AI活用で feedback loop が速くなっても、WIP fragmentation が増えていたら失敗。

この補正を入れないと、turnmaxxing は「すごく回してるけど、本人の頭は常に散っている」状態を見逃す。
それは Maker Time の敵である。

# A4一枚要約：AI版 Engineering Effectiveness

AI版 Engineering Effectiveness とは、AI利用量を競うことではない。
目的は、エンジニアの Maker Time を増やし、その質を守ることである。
Maker Time とは、深い文脈を保持して前進できるまとまった時間である。
したがって、測るべきはトークン量やターン数そのものではない。
測るべきは、AIが開発フロー上の摩擦を減らしたかである。

中心になる観測単位は feedback loop である。
仕様理解、コード理解、実装、テスト、CI、PRレビュー、リリース、障害対応など、
各 loop がどれくらい頻繁に起き、どれくらい待たせ、どれくらい使いにくいかを見る。
主な指標は latency、frequency、usability、outcome proxy の4種類である。
例として、PR first response time、CI duration、review round数、doc discovery time、
AI response latency、blocked time、AI-assisted checkpoint率などを測る。

ただし、AIとの turn が増えれば良いわけではない。
AIがレビュー前の自己検証、CI失敗調査、テスト観点整理を速くするなら良い。
一方で、AI返答確認が割り込み化し、作業を細切れにするなら悪い。
そのため、Maker Time % だけでは不十分である。
カレンダー上は空いていても、6つの作業を並行していれば深い作業とは言えない。

そこで Focus Integrity を測る必要がある。
Active WIP Count、Task Switching Rate、Focus Run Length、Maker Time Purity、
Resume Latency、Parallel AI WIP、AI-induced Switching を見る。
特に Maker Time Purity は、空き時間のうち単一文脈に集中できた割合である。
Parallel AI WIP は、AIに投げたまま人間判断待ちになっている並行作業数である。

良い turnmaxxing とは、AIとの feedback loop を増やしつつ、
人間の Focus Continuity を壊さないことである。
悪い turnmaxxing は、AIをSlack化し、返答のたびに作業を切り替えることである。
AI返答は通知で即時処理せず、Focus Run の後にまとめて回収するのが望ましい。
個人評価やランキングには使わず、チームの friction map を作るために使う。
見るべき問いは「誰が多く使ったか」ではなく、
「どこで Maker Time が削られ、AIはその摩擦を減らしたか」である。

```
