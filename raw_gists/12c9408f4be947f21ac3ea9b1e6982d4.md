# HOAST それぞれの具体例

## HOASTの見方

HOASTは抽象度の階層として見るとわかりやすいです。

- **Tool** が最も具体
- **Skill** は Tool を使って仕事を定義する
- **Agent** は Skill を使って役割を果たす
- **Orchestration** は複数 Agent を采配する
- **Harness** は全体の組織設計・運用基盤を担う

---

## Tool の具体例

### Tool
Skill や Agent から呼ばれる具体的なプログラムや外部機能。

### 例

- **Web検索API**
  - Google Custom Search
  - SerpAPI
  - Tavily

- **HTTPクライアント**
  - REST API を叩く自作スクリプト

- **RAG検索**
  - ベクトルDB検索ツール
  - Pinecone 検索
  - Weaviate 検索
  - FAISS 検索

- **ファイル操作**
  - PDFを読むツール
  - CSVを読むツール
  - Excelを読むツール
  - ファイル一覧取得ツール

- **実行系**
  - Python 実行ツール
  - SQL 実行ツール
  - shell 実行ツール

- **開発支援**
  - Git 操作ツール
  - GitHub API ツール
  - テスト実行ツール
  - Linter 実行ツール

- **業務システム連携**
  - Slack 投稿ツール
  - Notion API ツール
  - Jira API ツール
  - Google Calendar API ツール
  - 社内ワークフローAPI

- **MCP**
  - 社内文書検索MCP
  - GitHub MCP
  - Playwright MCP

### Toolの具体像

- 請求書PDFから金額を抜き出すPythonスクリプト
- GitHubのissueを作るAPIラッパー
- 社内規程DBを検索するMCPサーバ
- ブラウザを自動操作するPlaywright

## Skill の具体例

### Skill
特定の仕事をこなすための手順書・指示集。  
必要に応じて Tool 利用方針も含む。

### 例

- **要約 Skill**
  - 長文を3点で要約する
  - 必要なら専門用語を注釈する

- **議事録作成 Skill**
  - 会話ログから決定事項、ToDo、保留事項を抽出する

- **バグ切り分け Skill**
  - ログを読む
  - 再現条件を推定する
  - 必要ならテスト実行 Tool を使う

- **コードレビュー Skill**
  - 差分を見る
  - 設計逸脱、命名、テスト不足を点検する
  - 必要ならリポジトリ検索 Tool を使う

- **競合調査 Skill**
  - Web検索する
  - 3社比較表をつくる
  - 情報源URLを必ず添える

- **FAQ回答 Skill**
  - まずナレッジベース検索 Tool を使う
  - 見つからなければ「不明」と返す

- **経費精算チェック Skill**
  - 領収書をOCRで読む
  - 規程DB Tool で照会する
  - 差戻し理由を定型文で出す

- **テストケース生成 Skill**
  - 仕様書から正常系、異常系、境界値を列挙する

### Skillの具体像

- Claude Code Skills のようなもの
- Issueを読んで実装修正プランを作る手順書
- 問い合わせ文を分類して回答案を作るプロンプト
- 契約書レビュー時に見るべき観点一覧 + 条文検索ツール利用ルール

### Skill と Tool の組み合わせ例

- **Skill**
  - 障害一次対応 Skill

- **内包する Tool 利用**
  - ログ検索 Tool
  - メトリクス参照 Tool
  - インシデント管理票作成 Tool

- **流れ**
  - まず障害レベルを判定
  - 関連ログを検索
  - 既知障害と照合
  - 必要なら起票

---

## Agent の具体例

### Agent
役割を持って動くAIの実行単位。  
Skill を道具として利用する。

### 例

- **調査 Agent**
  - Web検索 Skill
  - 競合調査 Skill
  - 要約 Skill
  - を使って調査報告書を作る

- **実装 Agent**
  - コード読解 Skill
  - 修正案作成 Skill
  - テスト実行 Skill
  - を使って実装する

- **レビュー Agent**
  - コードレビュー Skill
  - セキュリティ点検 Skill
  - を使ってPRを査読する

- **CS一次回答 Agent**
  - 問い合わせ分類 Skill
  - FAQ検索 Skill
  - 返信文生成 Skill
  - を使って回答する

- **採用候補者スクリーニング Agent**
  - 履歴書要約 Skill
  - 評価観点照合 Skill
  - 面接質問生成 Skill
  - を使う

- **営業提案 Agent**
  - 顧客情報整理 Skill
  - 提案骨子作成 Skill
  - 競合比較 Skill
  - を使う

- **SRE監視 Agent**
  - アラート解釈 Skill
  - ログ確認 Skill
  - エスカレーション Skill
  - を使う

### Agentの具体像

- 「あなたは経理担当です。申請内容を精査してください」というAI
- 「あなたはシニアソフトウェアエンジニアです。issue を解決してください」というAI
- 「あなたは市場調査担当です。情報源つきで調査してください」というAI

### 並列実行の例

- 3体の調査 Agent を同時起動
  - 1体目は日本市場担当
  - 2体目は米国市場担当
  - 3体目は中国市場担当
- あとで Orchestration が結果を統合する

## Orchestration の具体例

### Orchestration
複数 Agent や処理フローをどう動かすかを決める仕組み。

### 例

- **研究レポート作成オーケストレーション**
  - テーマを受け取る
  - 調査 Agent を3体並列起動
  - レビュー Agent に矛盾チェックさせる
  - 統合 Agent に最終レポートを書かせる

- **ソフトウェア開発オーケストレーション**
  - Issue を入力
  - 要件整理 Agent
  - 実装 Agent
  - テスト Agent
  - レビュー Agent
  - の順に流す
  - 失敗したら前段に差し戻す

- **CS自動応答オーケストレーション**
  - 問い合わせ分類 Agent
  - FAQ回答 Agent
  - 返金判定 Agent
  - 人間エスカレーション判定 Agent
  - を条件分岐で動かす

- **採用選考オーケストレーション**
  - 応募書類評価 Agent
  - 課題評価 Agent
  - 面接サマリ Agent
  - 総合判定 Agent
  - を束ねる

- **障害対応オーケストレーション**
  - 検知
  - 一次切り分け Agent
  - 影響範囲分析 Agent
  - 広報文面作成 Agent
  - 復旧後のポストモーテム Agent
  - を連携させる

### 実装の具体例

- LangGraph のグラフ定義
- CrewAI の crew 設計
- AutoGen の multi-agent 会話制御
- 自作ワークフローエンジン
- Airflow や Temporal で Agent 実行を管理する構成

### Orchestrationの具体像

- どの Agent を、どの順番で、何体、どんな条件で動かすか
- 失敗したら再試行するか、人間に渡すか
- 結果を誰が統合するか

## Harness の具体例

### Harness
個々の Orchestration や Agent を越えて、全体を組織として成立させる枠組み。

### 例

- **社内AI基盤**
  - 共通モデル接続
  - 認証認可
  - 監査ログ
  - コスト管理
  - プロンプト配布
  - ツール接続基盤
  - 評価基盤

- **マルチエージェント運用環境**
  - Agent レジストリ
  - Skill カタログ
  - Tool カタログ
  - 実行履歴管理
  - 権限制御
  - 事故時停止機構

- **AI開発プラットフォーム**
  - dev/stg/prod 環境
  - ABテスト
  - オンライン評価
  - プロンプトバージョン管理
  - モデル切替
  - 観測性

- **企業全体のAIオペレーティングモデル**
  - どの部門がどの Agent を所有するか
  - レビュー責任者は誰か
  - 障害時の連絡系統はどうするか
  - 法務・情報セキュリティ審査をどう通すか

- **AIワーカー群を抱える「会社」のような仕組み**
  - 営業Agentチーム
  - 開発Agentチーム
  - サポートAgentチーム
  - それらを横断する共通基盤とガバナンス

### 実装の具体例

- OpenAI / Anthropic / Gemini など複数LLMへの共通接続基盤
- MCPサーバ群の社内標準運用
- Prompt / Skill の社内レジストリ
- LLMOps 基盤
  - LangSmith
  - Helicone
  - Weights & Biases
  - Arize Phoenix
  - OpenTelemetry ベースの監視
- 権限・監査
  - どの Agent がどの Tool を使ったかを記録
  - 個人情報アクセスを制限

### Harnessの具体像

- 「Agent 個体」ではなく「Agent 組織全体」を運営する仕組み
- 「現場の個別ワークフロー」ではなく「全社で安全・継続的に回すための土台」

## ひとつの具体的シナリオで全部対応づける

### シナリオ
ECサイトの問い合わせ対応自動化

### Tool
- 注文検索API
- 返品規程DB検索
- Slack通知API
- CRM更新API

### Skill
- 問い合わせ分類 Skill
- 返品可否判定 Skill
- 返信文作成 Skill
- 人手エスカレーション Skill

### Agent
- 受付 Agent
- 返品判定 Agent
- 返信 Agent
- 監督 Agent

- 受付 Agent が分類
- 返品系なら返品判定 Agent へ

- 高リスク案件は監督 Agent を通す

### Harness
- 全問い合わせログの監査
- 顧客情報アクセス制御
- Agent ごとのKPI管理
- モデル利用料の予算管理
- 誤回答率の継続評価
- プロンプト更新の承認フロー

## 補足意見

HOASTの肝は、  
**「なんでも Agent に背負わせない」**  
ことだと思います。

- まず Tool を分離する
- 次に Skill を分離する
- それでも役割が必要なときに Agent を立てる
- そして複数 Agent が出てきたら Orchestration を考える
- 最後に本番運用・組織運営まで視野に入れると Harness が要る

という順で考えると整理しやすいです。
