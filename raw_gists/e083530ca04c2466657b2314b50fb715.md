# GTD LLM Wiki

A pattern for running a complete GTD system with an LLM agent as the maintainer.

This is an idea file. Copy-paste it to your own LLM agent (Claude Code, OpenAI Codex, OpenCode, etc.) and instantiate a version that fits you. The document communicates the pattern; your agent figures out the rest in collaboration with you.

## The core idea

GTD (Getting Things Done) is a productivity methodology by David Allen. Its premise is simple: get everything out of your head, file it into the right "boxes," review those boxes on a regular cadence, and act from the lists rather than from memory. Done right, you get *clarity* (頭がすっきり), *direction* (次やることがはっきり), and *calm* (忘れる不安からの解放).

The problem is that GTD is high-maintenance. Inboxes need to be processed. Projects need to be broken down. Lists need to be tagged, pruned, and reviewed weekly, monthly, yearly. Most people abandon GTD not because it doesn't work, but because the bookkeeping grows faster than they can keep up. Humans get bored. They forget to update cross-references. They skip the weekly review three times in a row and the system rots.

LLMs don't get bored. They don't forget to update a cross-reference. They can touch fifteen files in one pass.

The idea here: **run the entire GTD system as a directory of markdown files, with the LLM as the maintainer.** You do the things only you can do — capture what's in your head, decide what matters, execute the actions. The LLM does everything else — sorting the inbox, breaking down projects, maintaining tags, running reviews, keeping the wiki consistent.

The wiki is not a static knowledge base. It is your *operational* GTD system. The Inbox is a markdown file. The Project list is a markdown file. The Next Actions list is a markdown file. The LLM reads, writes, and reorganizes them according to the rules in this document.

You never (or rarely) write the wiki yourself. You talk to the LLM. The LLM updates the files. You read the files (or browse them in Obsidian, or however you prefer) to see the current state of your life.

This is the GTD analogue of the "personal knowledge wiki maintained by an LLM" pattern: same architecture, same division of labor, but pointed at a different problem.

## Architecture

Three layers, same as any LLM wiki:

**Raw inputs** — things that arrive from the outside. Voice memos, screenshots, emails you forwarded, scraps you typed at 2am, meeting notes, web clippings. These are immutable; the LLM reads from them but the source of truth for an item is whatever the LLM has filed.

**The wiki (the GTD system itself)** — a directory of markdown files representing each GTD box, plus a few support files. The LLM owns this layer entirely. It moves items between boxes, breaks projects down, retags, prunes, logs. You read it.

**The schema** — a `CLAUDE.md` / `AGENTS.md` style file that tells the LLM how the wiki is laid out, what the conventions are, and what workflow to follow for each operation (capture, clarify, organize, reflect, engage, plus the four review cadences). This is the configuration file. You and the LLM co-evolve it as you learn what works.

## The boxes (as files)

A minimal but complete instantiation:

```
gtd/
├── CLAUDE.md              # the schema / operating manual for the LLM
├── inbox.md               # capture lands here, unprocessed
├── projects.md            # active projects with one-line status
├── next-actions.md        # today / this-week / backlog, with tags
├── routines.md            # recurring actions (daily, by weekday, monthly)
├── calendar.md            # date/time-bound commitments
├── waiting-for.md         # things blocked on someone else
├── someday.md             # things you want to do eventually, not now
├── reference/             # filing cabinet — anything not actionable
│   ├── index.md
│   └── <topic>.md
├── projects/              # one file per project that needs more than a line
│   └── <project-slug>.md
├── core.md                # values, vision, goals, areas of focus (the "軸")
└── log/                   # append-only records
    ├── daily.md
    ├── weekly.md
    ├── monthly.md
    └── yearly.md
```

A few notes on this layout:

- `inbox.md` and `next-actions.md` are the most-touched files. Everything else is updated less frequently.
- `projects.md` is the one-line summary view. `projects/<slug>.md` is where a project's sub-tasks, brainstorm, and notes live. A project only gets its own file once it's big enough to need one — small ones stay as a single line in `projects.md`.
- `core.md` holds the four "axis" boxes (values, vision/ideal, goals, constraints) in one file. They're small, related, and reviewed together.
- `reference/` is the dumping ground for anything that's just information, not a thing to do. Notes you took, articles you clipped, manuals, decisions you've made. The LLM maintains `reference/index.md` so you can find things.
- `log/` is append-only. The LLM never edits past entries. Daily entries feed weekly reviews, weekly feeds monthly, monthly feeds yearly. Each entry starts with a consistent prefix so `grep "^## \[" log/weekly.md | tail -8` shows the last two months.

## Operations

GTD has five canonical phases: capture, clarify, organize, reflect, engage. The LLM's role in each:

**Capture.** You dump something into the chat — a thought, a worry, a task, a half-formed idea, a photo of a receipt, a forwarded email. The LLM appends it to `inbox.md` verbatim (or near-verbatim), without judgment, without processing. Capture is fast and dumb on purpose. You should be able to capture in five seconds. If the LLM starts asking "should I file this as a project?" during capture, it's doing it wrong.

There are three capture modes worth supporting explicitly: **mind-sweep** (you sit down and dump everything for an hour or two — usually at the start, then occasionally), **regular** (a few minutes every few days), and **ambient** (whatever just popped up, in five seconds). The LLM handles all three the same way: append to `inbox.md`. The schema can note timestamps if you want them.

**Clarify.** Every few days you tell the LLM "process the inbox." For each item, the LLM asks (or decides, if obvious): is this trash, reference, someday/maybe, or actionable? If actionable, is it a single action or a project? If it's a less-than-two-minutes thing, the LLM flags it for you to just do now. Otherwise, the LLM files it: into `projects.md`, `next-actions.md`, `someday.md`, `reference/`, or just deletes it. The inbox should be empty when clarify is done.

Clarify is conversational. The LLM should not silently file ambiguous items. When in doubt, it asks. A good clarify session is the LLM going down the inbox, proposing a destination for each item, and you confirming or correcting in one or two words. The LLM remembers your corrections within the session and gets better at proposing.

**Organize.** This is the "break-down" step (ブレイクダウン). For each new project, the LLM proposes a desired outcome ("what does done look like?"), a few candidate next actions, and any waiting-for or calendar items that fall out of it. It writes these into `next-actions.md`, `waiting-for.md`, `calendar.md`, and creates `projects/<slug>.md` if the project is non-trivial. It tags next actions with contexts (`#home`, `#errand`, `#deep`, `#admin`, `#phone`, or whatever vocabulary you and the LLM settle on in the schema). Tags are the LLM's lever for showing you a short, relevant list when you're in a particular context.

The LLM also maintains the **shape** of `next-actions.md`. By default that's three sections:

```
## Today
- [ ] specific action with context tag  #context
- [ ] ...

## This week
- [ ] ...

## Backlog
- [ ] ...
```

Things move between sections during reviews and on request ("move this to today," "push this to next week," "I'm not going to do this — kill it").

**Reflect (review).** The hardest part of GTD. The LLM runs four review cadences, and *only the LLM running them* makes this realistic:

- **Daily** (5–10 min, every evening). The LLM walks you through what got done today, what's left, what's on tomorrow's calendar, and prompts: any captures from today not yet processed? Anything to move to today vs. push? It writes a short entry to `log/daily.md`.

- **Weekly** (1–2 hours, once a week, ideally Sunday morning). The big one. The LLM reads the past week's daily logs, walks through every project in `projects.md`, asks for an updated one-line status on each, identifies projects with no next action (the classic GTD failure mode) and proposes one, surfaces stalled waiting-fors that need a nudge, prunes `someday.md`, and writes a structured entry to `log/weekly.md` summarizing the week and surfacing patterns.

- **Monthly** (1–2 hours, once a month). The LLM rolls up the four weekly entries, reviews `core.md` (specifically goals and constraints), asks whether your current projects actually serve your goals, flags any drift, and proposes adjustments. Writes to `log/monthly.md`.

- **Yearly** (a few hours, around year-end). The LLM rolls up the monthly entries, reviews values and vision in `core.md`, runs a retrospective, and helps you redraft `core.md` for the coming year. Writes to `log/yearly.md`.

Each review has its own trigger list (a checklist of questions the LLM walks you through). These live in the schema and you tune them over time. The LLM should not improvise reviews — it should follow the trigger list, because the value of a review is the *consistency*.

**Engage.** This is just "do the work." The LLM's job here is small but important: when you ask "what should I do now?" or "what's on for today?", it reads `next-actions.md`, filters by any context you mention ("I'm on the train" → `#mobile` and `#read`), and shows you a short, ranked list. It doesn't generate a sermon. It shows you the list and gets out of the way.

When you finish something, you tell the LLM, it marks it done, and (this matters) it logs what got done to `log/daily.md` so the weekly review has material to work with. Without that log, reviews degenerate into "uhh what did I do this week."

## The axes (core.md)

GTD's "horizons of focus" (高度モデル) live in one file because they're small and reviewed together:

```markdown
# Core

## Values (高度5)
What you won't compromise on, regardless of circumstance.
- ...

## Vision (高度4)
What you want your life to look like in the long term. Maintained states and aspirational states both count.
- ...

## Goals (高度3)
1–2 year milestones that move you toward the vision. Should be concrete enough to know if you hit them.
- [ ] ... (by YYYY-MM)
- [ ] ...

## Areas of focus / constraints (高度2)
Roles, responsibilities, ongoing commitments, things you can't ignore. Health conditions, monthly cost of living, who you're responsible to.
- ...
```

The LLM references `core.md` during monthly and yearly reviews to check whether the system is still pointed where you want it. It does *not* reference it constantly during clarify or organize — that would make every small decision feel existential. Axes are for periodic recalibration, not daily anxiety.

## The schema (CLAUDE.md)

This is where conventions live so the LLM stays consistent across sessions. A starter:

```markdown
# GTD Wiki — Operating Manual

## File map
- See architecture above.

## Conventions
- Inbox lines are timestamped: `- [YYYY-MM-DD HH:MM] raw text`.
- Next actions are checkboxes with a context tag: `- [ ] action  #context`.
- Project lines in projects.md: `- <slug>: <one-line status>  [→ projects/<slug>.md]` (link only if a project file exists).
- Log entries start with `## [YYYY-MM-DD] <type> | <short title>`.
- The "two-minute rule" applies during clarify: if it'll take less than two minutes, prompt the user to just do it rather than filing it.

## Tags / contexts
- #home, #office, #errand, #phone, #computer, #deep, #admin, #mobile, #read, #ask:<person>
- (Add/remove as you settle in. Fewer is better.)

## Workflow rules
1. Capture: append to inbox.md, no processing.
2. Clarify: go item-by-item, propose destination, confirm with user, file.
3. Organize: every new project gets an outcome, a next action, and tags. Big projects get their own file.
4. Reflect: follow the trigger list for the cadence. Don't improvise. Write to log/.
5. Engage: when asked "what now?", read next-actions.md, filter by context, return a short list.

## Trigger lists
### Daily review
- What got done today?
- What's still open from today's list?
- What's on tomorrow's calendar?
- Any unprocessed captures?
- Anything to push from today to this-week or vice versa?

### Weekly review
- Read the seven daily logs from this week.
- Walk through every project: status? next action? stalled?
- Walk through waiting-for: nudge anyone?
- Sweep someday: anything to activate? anything to drop?
- Sweep next-actions: anything stale? anything to retag?
- Summarize the week: Y (did) / W (learned) / T (next).

### Monthly review
- Roll up the four weekly entries.
- Re-read goals and constraints in core.md.
- Are current projects serving the goals? What's drift?
- Adjust projects, goals, or constraints as needed.

### Yearly review
- Roll up the twelve monthly entries.
- Re-read values and vision in core.md.
- What changed about who you are this year?
- Redraft core.md for the coming year.
```

You'll grow this file over time. That's the point.

## Indexing and logging

Two special files keep the system navigable as it grows:

**reference/index.md** — a catalog of reference pages. Each entry is one line: link, one-sentence summary, optional tags. The LLM updates it whenever it files something new to `reference/`. When answering "what do I know about X?", the LLM reads the index first, then drills into the pages that look relevant.

**log/\*.md** — chronological, append-only, never edited. Consistent prefix on each entry (`## [YYYY-MM-DD] daily | ...`) makes the log greppable. Daily feeds weekly feeds monthly feeds yearly — each review reads the level below it, not the raw inbox of memory.

## Why this works (and where it doesn't)

The thing GTD is bad at, by itself, is the *maintenance*. Updating every project's status. Re-tagging actions. Pruning someday lists. Running the weekly review when you really don't want to. Catching the project with no next action — the silent failure that rots the whole system. These are the tasks LLMs are unusually good at: pattern-matching across many files, doing tedious bookkeeping without complaint, asking the same review questions in the same order every single week.

The thing the LLM is bad at is *knowing what matters to you*. It can't decide whether the half-finished novel goes into Projects or Someday. It can't tell whether the gnawing feeling about your job is a project ("update résumé") or a value-level question for the next monthly review. You have to do that. But the LLM can ask the right question at the right time, and write down what you answer, and remember.

This pattern works best when you're willing to talk to the LLM every day (even briefly), and willing to do a real weekly review with it. If you ghost the system for a month, no LLM will save it — the inbox will be enormous and the review will be miserable. GTD is still GTD. The LLM just removes the friction that kills it for most people.

## Optional: tools and conveniences

- **Obsidian** as a viewer. The wiki is just markdown files in a folder; point Obsidian at it and you get backlinks, graph view, and a nice reading experience while the LLM does the writing.
- **Voice capture.** A keybind or shortcut that pipes a voice memo (transcribed) straight to the LLM with "capture this" gets ambient capture down to a few seconds, which is what GTD actually needs.
- **A search tool over the wiki.** At small scale (a few hundred files), the LLM reading `reference/index.md` is enough. At larger scale, something like [qmd](https://github.com/tobi/qmd) — hybrid BM25/vector search over markdown — gives the LLM a proper retrieval layer.
- **Git.** The wiki is just markdown. Put it in a git repo and you get version history, branching, and remote backup for free. Weekly reviews become diffable.

## A note on adoption

Don't try to instantiate the whole thing on day one. The fastest way to abandon a GTD system is to set up an elaborate one before you've internalized the loop. A reasonable progression:

1. **Day 1.** Just `inbox.md` and `next-actions.md`. The LLM appends captures; you clarify daily; you act from the list. That's it.
2. **Week 2.** Add `projects.md` and start breaking captures into projects + next actions during clarify.
3. **Month 1.** Add the daily and weekly review cadences with trigger lists. Add `waiting-for.md`, `calendar.md`, `someday.md`, `reference/` as you start needing them.
4. **Month 2+.** Add `core.md` and the monthly review. Tune the schema. Stop fiddling with structure and just use it.
5. **Year 1.** Add the yearly review. By now the system is yours, not this document's.

## Note

This is intentionally a pattern, not an implementation. The exact filenames, tag vocabulary, trigger lists, review cadences, and conventions should be argued out between you and your LLM until they fit how you actually live. Some people don't need `someday.md` — they delete instead. Some people need three project files for different life domains. Some people want timestamps everywhere; others find them noisy. Pick what works, drop what doesn't. The LLM's job is to maintain whatever system you settle on; this document's job is to describe a shape that's known to hold up.

Share this file with your LLM agent, talk through what to keep and what to cut, and start with the smallest version that captures and clarifies. The rest accretes.
