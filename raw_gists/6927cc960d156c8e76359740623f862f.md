# オープン・ナレッジ・フォーマット（OKF）

**バージョン 0.1 — ドラフト**

OKF は、*知識* ——すなわちデータやシステムを取り巻くメタデータ、コンテキスト、
そして編集・整理された知見—— を表現するための、オープンで、人間にもエージェント
にも扱いやすいフォーマットである。人間が著述し、エージェントが生成し、組織間で
交換し、その双方によって消費されることを想定して設計されている。

このフォーマットは意図的にミニマルである。YAML フロントマターを備えた markdown
ファイルのディレクトリにすぎない。スキーマレジストリも、中央集権的な権威も、
必須のツールも存在しない。ファイルを `cat` できれば OKF は読めるし、リポジトリを
`git clone` できれば OKF を配布できる。

---

## 1. 動機

AI エージェントのための知識表現の領域は急速に進化しており、互換性のない多くの
慣習が次々と生まれている。OKF は、知識は次のような、広くアクセス可能で確立された
フォーマットで表現するのが最善である、という立場を取る。

- **人間が** ツールなしで **読める**。
- **エージェントが** 専用 SDK なしで **パースできる**。
- バージョン管理上で **差分を取れる**。
- ツール・組織・時間を越えて **可搬性がある**。

このフォーマットは最小限の主張しか持たない。知識のコーパスを *自己記述的* にする
ために必要な、ごく少数の構造的な慣習だけを標準化する。それを超える部分はすべて
生成者に委ねられる。

### ゴール

1. **エンリッチメント・エージェント** が書き込める普遍的なフォーマットを定義する。
2. **コンシューマ・エージェント** がそれをどう読み、たどるべきかを示す。
3. システムや組織を越えた知識の **交換** を促進する。
4. コンテンツが意味のある形で消費されるために存在していなければならない、
   ごく少数の **必須** フィールドを標準化する。

### 非ゴール

- 概念タイプの固定的な分類体系を定義すること。
- ストレージ・配信・クエリ基盤を規定すること。
- ドメイン固有のスキーマ（Avro、Protobuf、OpenAPI など）を置き換えること
  —— OKF はそれらを *参照* するのであって、包含するものではない。

## 2. 用語

- **ナレッジ・バンドル（Knowledge Bundle）** —— 自己完結的で階層的な知識ドキュメント
  の集合。配布の単位。
- **概念（Concept）** —— バンドル内の知識の単一の単位。1 つの markdown ドキュメント
  として表現される。具体的な資産（テーブル、API）、抽象的なアイデア（メトリクス、
  業務プロセス）、あるいはその中間のあらゆるものを記述しうる。
- **概念 ID（Concept ID）** —— バンドル内における概念ファイルのパスから `.md` 拡張子
  を除いたもの。たとえば `tables/users.md` の概念 ID は `tables/users` となる。
- **フロントマター（Frontmatter）** —— markdown ファイルの先頭に `---` で区切られた
  YAML メタデータブロック。
- **本文（Body）** —— フロントマター以降のすべて。
- **リンク（Link）** —— ある概念から別の概念への標準的な markdown リンク。暗黙的な
  親子階層を越えた関係を表現するために用いる。
- **引用（Citation）** —— 本文中の主張を裏づける外部ソースへの、概念からのリンク。

---

## 3. バンドル構造

バンドルは markdown ファイルのディレクトリツリーである。ディレクトリ構造は
ドメインから独立している —— 生成者は、捕捉したい知識にとって意味のある形で自由に
概念を組織化する。

```
path/to/bundle/
├── index.md                      # 任意。プログレッシブ・ディスクロージャ用のディレクトリ一覧。
├── log.md                        # 任意。更新の時系列履歴。
├── <concept>.md                  # バンドル直下の概念。
└── <subdirectory>/               # サブディレクトリは概念をグループに整理する。
    ├── index.md
    ├── <concept>.md
    └── <subdirectory>/
        └── …
```

バンドルは以下のいずれかとして配布してよい（MAY）。

- git リポジトリ（推奨 —— 履歴・帰属・差分を提供する）。
- ディレクトリの tarball または zip アーカイブ。
- より大きなリポジトリ内のサブディレクトリ。

### 3.1 予約ファイル名

以下のファイル名は階層のどのレベルでも定められた意味を持つため、概念ドキュメント
に使用してはならない（MUST NOT）。

| ファイル名   | 用途                                                   |
|--------------|--------------------------------------------------------|
| `index.md`   | ディレクトリ一覧。§6 を参照。                          |
| `log.md`     | 更新履歴。§7 を参照。                                  |

それ以外のすべての `.md` ファイルは概念ドキュメントである。

タグそのものは引き続き第一級の概念である —— §4.1 の `tags` フロントマターフィールド
を参照。OKF は、タグごとにドキュメントを集約するための別個のファイル形式を規定しない。
タグ閲覧ビューを求める生成者は、消費時にフロントマターを走査して動的に生成できる。

## 4. 概念ドキュメント

すべての概念は UTF-8 の markdown ファイルである。次の 2 つの部分から成る。

1. ファイル先頭の独立行 `---` と、終端の独立行 `---` によって区切られた
   **YAML フロントマターブロック**。
2. 自由形式のコンテンツを含む **markdown 本文**。

### 4.1 フロントマター

```yaml
---
type: <Type name>                  # 必須（REQUIRED）
title: <任意の表示名>
description: <任意の一行要約>
resource: <基となる資産の正規 URI（任意）>
tags: [<tag>, <tag>, …]            # 任意
timestamp: <ISO 8601 日時>          # 任意。最終更新時刻。
# … その他、生成者が定義したキー／値のペア
---
```

**必須（Required）:**

- `type` —— 概念の種類を示す短い文字列。コンシューマはこれをルーティング、
  フィルタリング、表示に用いる。値の例：
  `BigQuery Table`、`BigQuery Dataset`、`API Endpoint`、`Metric`、
  `Playbook`、`Reference`。

  `type` の値は中央で登録されるものではない（**not** registered centrally）。
  生成者は説明的で自己説明的な値を選ぶべきである（SHOULD）。コンシューマは未知の
  タイプを優雅に許容しなければならない（MUST。通常は汎用的な概念として扱う）。

**推奨（Recommended、優先順位順）:**

- `title` —— 人間が読める表示名。省略された場合、コンシューマはファイル名から
  タイトルを導出してもよい（MAY）。
- `description` —— 概念を要約する一文。`index.md` の生成、検索スニペット、
  プレビューに用いられる。
- `resource` —— 概念が記述する基となる資産を一意に識別する URI。物理的な資源では
  なく抽象的なアイデアを記述する概念では存在しない。
- `tags` —— 横断的な分類のための短い文字列の YAML リスト。
- `timestamp` —— 意味のある最終変更の ISO 8601 日時。

**拡張（Extensions）:** 生成者は任意の追加キーを含めてよい（MAY）。コンシューマは
ラウンドトリップ時に未知のキーを保持すべきであり（SHOULD）、認識できないフィールド
を含むドキュメントを拒否すべきではない（SHOULD NOT）。

### 4.2 本文

本文は標準的な markdown である。生成者は、自由形式の散文よりも、構造的な markdown
—— 見出し、リスト、テーブル、フェンス付きコードブロック —— を優先すべきである
（SHOULD）。構造は人間による読みやすさとエージェントによる検索の双方を助けるからである。

必須の本文セクションは存在しない。以下のセクション見出しは **慣習的** な意味を持ち、
該当する場合に使用すべきである（SHOULD）。

| 見出し         | 用途                                                   |
|----------------|--------------------------------------------------------|
| `# Schema`     | 資産のカラム／フィールドの構造的記述。                |
| `# Examples`   | 具体的な使用例。多くはフェンス付きコードブロック。    |
| `# Citations`  | 本文中の主張を裏づける外部ソース。§8 を参照。         |

### 4.3 例：リソースに結び付いた概念

```markdown
---
type: BigQuery Table
title: Customer Orders
description: One row per completed customer order across all channels.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders, revenue]
timestamp: 2026-05-28T14:30:00Z
---

# Schema

| Column        | Type      | Description                              |
|---------------|-----------|------------------------------------------|
| `order_id`    | STRING    | Globally unique order identifier.        |
| `customer_id` | STRING    | Foreign key into [customers](/tables/customers.md). |
| `total_usd`   | NUMERIC   | Order total in US dollars.               |
| `placed_at`   | TIMESTAMP | When the customer submitted the order.   |

# Joins

Joined with [customers](/tables/customers.md) on `customer_id`.

# Citations

[1] [BigQuery table schema](https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders)
```

### 4.4 例：リソースに結び付いていない概念

```markdown
---
type: Playbook
title: Incident response — data freshness alert
description: Steps to triage a freshness alert on the orders pipeline.
tags: [oncall, incident]
timestamp: 2026-04-12T09:00:00Z
---

# Trigger

A freshness alert fires when `orders` lags more than 30 minutes behind
its expected SLA. See the [orders table](/tables/orders.md).

# Steps

1. Check the [ingestion job dashboard](https://example.com/dash).
2. …
```

## 5. クロスリンク

概念は標準的な markdown リンクを用いて他の概念にリンクしてよい（MAY）。次の 2 つの
形式がサポートされる。

### 5.1 絶対（バンドル相対）リンク

`/` で始まり、バンドルルートからの相対として解釈される。

```markdown
See the [customers table](/tables/customers.md) for the join key.
```

これは **推奨される** 形式である。ドキュメントがサブディレクトリ内で移動されても
安定しているためである。

### 5.2 相対リンク

標準的な markdown の相対パス。

```markdown
See the [neighboring concept](./other.md).
```

### 5.3 リンクの意味論

概念 A から概念 B へのリンクは *関係* を主張する。具体的な関係の種類（親子、参照、
結合、依存など）は、リンクそのものではなく周囲の散文によって伝えられる。グラフ
ビューを構築するコンシューマは、通常すべてのリンクを型のない関係の有向辺として扱う。

コンシューマはリンク切れを許容しなければならない（MUST）。対象がバンドル内に存在
しないリンクは不正ではなく、単に未執筆の知識を表しているだけかもしれない。

## 6. インデックスファイル

`index.md` ファイルは、バンドルルートを含む任意のディレクトリに置いてよい（MAY）。
これはディレクトリの内容を列挙し、**プログレッシブ・ディスクロージャ**（個々の
ドキュメントを開く前に、何が利用可能かを人間やエージェントに見せること）を支援する。

インデックスファイルはフロントマターを含まない。本文は 1 つ以上のセクションを用い、
それぞれが見出しの下に概念をグループ化する。

```markdown
# Section / Group Heading

* [Title 1](relative-url-1) - short description of item 1
* [Title 2](relative-url-2) - short description of item 2

# Another Section

* [Subdirectory](subdir/) - short description of the subdirectory
```

エントリにはリンク先の概念のフロントマターにある description を含めるべきである
（SHOULD）。生成者は `index.md` を自動生成してもよい（MAY）。コンシューマは存在しない
場合に動的に生成してもよい（MAY）。

---

## 7. ログファイル（任意）

`log.md` ファイルは、そのスコープに対する変更履歴を記録するため、階層のどのレベルに
置いてもよい（MAY）。形式は日付ごとにグループ化されたエントリのフラットなリストで、
新しいものが先頭に来る。

```markdown
# Directory Update Log

## 2026-05-22
* **Update**: Added new BigQuery table reference for [Customer Metrics](/tables/customer-metrics.md).
* **Creation**: Established the [Dataplex Playbook](/playbooks/dataplex.md).

## 2026-05-15
* **Initialization**: Created foundational directory structure.
* **Update**: Added progressive-disclosure guidelines to the root [index](/index.md).
```

日付の見出しは ISO 8601 の `YYYY-MM-DD` 形式を使わなければならない（MUST）。ログ
エントリは散文である。先頭の太字の語（`**Update**`、`**Creation**`、
`**Deprecation**` など）は慣習であって必須ではない。

---

## 8. 引用

概念の本文が外部資料を出典とする主張を行う場合、それらの出典はドキュメント末尾の
`# Citations` 見出しの下に番号付きで列挙すべきである（SHOULD）。

```markdown
# Citations

[1] [BigQuery public dataset announcement](https://cloud.google.com/blog/products/data-analytics/...)
[2] [Internal data quality runbook](https://wiki.acme.internal/data/quality)
```

引用リンクは、絶対 URL でも、バンドル相対パスでも、外部資料を第一級の OKF 概念として
ミラーリングする `references/` サブディレクトリへのパスでもよい（MAY）。

---

## 9. 適合性（Conformance）

バンドルが次を満たすとき、OKF v0.1 に **適合（conformant）** している。

1. ツリー内のすべての非予約 `.md` ファイルが、パース可能な YAML フロントマター
   ブロックを含む。
2. すべてのフロントマターブロックが、空でない `type` フィールドを含む。
3. すべての予約ファイル名（`index.md`、`log.md`）が、存在する場合にそれぞれ §6 と §7
   で述べられた構造に従う。

コンシューマは、それ以外のすべての制約を緩やかな指針として扱うべきである（SHOULD）。
とりわけコンシューマは、以下を理由としてバンドルを拒否してはならない（MUST NOT）。

- 任意のフロントマターフィールドの欠落。
- 未知の `type` 値。
- 未知の追加フロントマターキー。
- リンク切れ。
- `index.md` ファイルの欠落。

この寛容な消費モデルは意図的なものである。OKF は、バンドルが成長し、リファクタリング
され、一部がエージェントによって生成されても、有用であり続けることを目指している。

## 10. 他のフォーマットとの関係

OKF は、いくつかの確立されたパターンに意図的に近い。

- markdown ＋ フロントマターをエージェント可読の知識ベースとして用いる
  **LLM「wiki」リポジトリ**。
- 階層的な markdown とクロスリンクを用いる、Obsidian や Notion のような
  **パーソナル・ナレッジ・ツール**。
- カタログメタデータを別個のレジストリではなくソースコードと並べて保存する
  **「メタデータ・アズ・コード（metadata as code）」** のアプローチ。

OKF が主に異なるのは、**仕様化されている** 点である —— ツールを規定することなく、
相互運用性のために必要なごく少数のルールを確定させている。

## 11. バージョニング

本ドキュメントは OKF バージョン **0.1** を規定する。将来の改訂は `<major>.<minor>`
の形式でバージョン付けされる。

- **マイナー** バージョンの引き上げは、後方互換な追加（新しい任意フィールド、新しい
  慣習的セクション見出し）を導入する。
- **メジャー** バージョンの引き上げは、破壊的変更（必須フィールドのリネーム、予約
  ファイル名の変更）を行いうる。

バンドルは、バンドルルートの `index.md` フロントマターブロックに
`okf_version: "0.1"` を含めることで、対象とする OKF バージョンを宣言してよい（MAY。
これは `index.md` でフロントマターが許可される唯一の場所である）。宣言されたバージョン
を理解できないコンシューマは、バンドルを拒否するのではなく、ベストエフォートでの消費
を試みるべきである（SHOULD）。

## 付録 A —— 最小限のバンドル例

```
my_bundle/
├── index.md
├── datasets/
│   ├── index.md
│   └── sales.md
└── tables/
    ├── index.md
    ├── orders.md
    └── customers.md
```

`datasets/sales.md`:

```markdown
---
type: BigQuery Dataset
title: Sales
description: All sales-related tables for the retail business.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales
tags: [sales]
timestamp: 2026-05-28T00:00:00Z
---

The sales dataset contains transactional tables, including
[orders](/tables/orders.md) and [customers](/tables/customers.md).
```

`tables/orders.md`:

```markdown
---
type: BigQuery Table
title: Orders
description: One row per completed customer order.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders]
timestamp: 2026-05-28T00:00:00Z
---

# Schema

| Column        | Type      | Description                  |
|---------------|-----------|------------------------------|
| `order_id`    | STRING    | Unique order identifier.     |
| `customer_id` | STRING    | FK to [customers](/tables/customers.md). |
| `total_usd`   | NUMERIC   | Order total in USD.          |

Part of the [sales dataset](/datasets/sales.md).
```
