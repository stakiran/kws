## Q1. exported JSON をどう加工して SQLite に入れているか

`grasp import --cosense <json>` → **`import_export_to_sqlite()`**（`sqlite_store.py:133`）が入口。
処理は2段階。

```
JSON → CosenseStore（メモリ上のパース＋グラフ生成） → SQLite（テーブルへ書き込み）
```

### 段階1: JSON のパース（`cosense.py`）

`CosenseStore.from_cosense_export()`（`cosense.py:371`）が `json.load` して `data["pages"]` を回す。

- **page**: `id` / `title` / `created` / `updated` / `views` を取り、
  `norm_title = normalize_title(title)`（casefold + 連続空白を1個に畳む = identity 正規化）を付与。
- **line**: Cosense の line は文字列か `{text, created, updated, userId}` の dict。
  `line_id = f"{page_id}:{index}"` を振る（行リンクの identity）。
- **edge（ここが肝）**: JSON に edge は **入っていない**。各行の text を
  `parse_cosense_links()` に通して内部リンク（単一ブラケットのページリンク・ハッシュタグ）を抽出し、
  `Edge(source_page_id, line_id, line_index, target_title, target_norm)` を生成する。
  ＝ Scrapbox の自動双方向リンクをここで materialize している。
- 同時に `title_collisions`（norm が衝突するタイトル）も検出。

URL・icon/img・装飾・cross-project リンク `[/proj/page]` は除外。

### 段階2: SQLite への書き込み（`sqlite_store.py`）

1. **project 名決定**: 引数 → `data["name"]` → ファイル stem の順。
2. **`ensure_store_schema()`**（`:395`）: スキーマが無い/古ければ `SCHEMA`（`SCHEMA_VERSION = "5"`）を流す。
   tmp に作って `os.replace` でアトミック差し替え。
3. **1トランザクション内で**（`:151`）:
   - `_delete_project()` で同名 project を全削除（**再 import は丸ごと置換、冪等**）。
   - `INSERT INTO projects`（集計値: pages/lines/edges 数）
   - `executemany` で `pages` → `lines` → `edges`（段階1で生成した derived edge をそのまま挿入）
   - **`rebuild_unresolved_targets()`**: どの page にも解決しないリンク（＝赤リンク）を集計し、
     `unresolved_targets` と `unresolved_target_examples` テーブルへ materialize。
   - `metadata` に `schema_version`, `source_type=cosense`, `title_aliases={}` 等を記録。
4. **`_cache_import_source()`**（`:470`）: 元 JSON を DB の隣
   （`<store>.imports/<project>.cosense.json`）にコピー＋manifest 保存。
   スキーマ版が上がったとき `recover_store_from_import_cache()` で再ビルドするため。

### データモデルの特徴（`SCHEMA`, `:40`）

- 全テーブルが `project` 列でマルチプロジェクト共存。FK は `ON DELETE CASCADE`。
- `pages(project, norm_title)`, `edges(project, target_norm)` に index。
  backlink / related / paths はこの正規化キーで JOIN する。
- **edge と unresolved_targets（赤リンク）は import 時に事前計算して materialize** するのが設計の中心。

### Markdown 取り込みとの差

`import_markdown_folder_to_sqlite()`（`:273`）も同じテーブルに入るが、
manifest hash 差分で **インクリメンタル更新** する点が cosense（毎回フル置換）と違う。

## Q2. 「Cosense のリンク記法のネットワークをそのまま入れてるだけ？」

ネットワークの **素材** はリンク記法（単一ブラケット・ハッシュタグ）由来で、
それ以上の意味解析（NLP）はしていない。が「そのまま」ではなく、いくつか導出・加工がある。

1. **方向の導出**: JSON にあるのは「ある行に `[X]` と書いてある」片方向の事実だけ。
   `Edge(source→target)` を持ちつつ、読み出し時に `target_norm` で逆引きJOINして **双方向** に見せる。
2. **identity 正規化**: `target_title` 直結ではなく `norm_title`（casefold＋空白畳み）一致でノード同定
   （Scrapbox の name=identity 問題への対処）。
3. **赤リンクをノード化**: 解決しないリンク先（`unresolved_targets`）を集計して一級ノードとして持つ。
4. **行を edge の出所として保持**: 「どのページのどの行が」リンクしているかまで持つ（行リンク）。
5. **2-hop は計算側**: `related`（2-hop）は保存せず adjacency から読み出し時に計算。

→ 正確には「**リンク記法を素材に、正規化で identity を直し、双方向・赤リンク・行という
Scrapbox のグラフモデルを materialize したもの**」。

## Q3. SQLite に入れたデータからどう探すか / LLM 任せか

結論: **LLM に自由クエリを書かせるのではなく、grasp が固定 CLI コマンド集合を提供し、
LLM はその中から選んで叩くだけ。** SQL も FTS クエリも LLM には書かせない。

### とっかかりは2種類

**A. ノードを既に知っている場合（タイトル/ID 起点）**

- **`read <title>`**（`:2826`）: 主力。1ノード起点で
  「本文 + backlinks + related(2-hop) + そのページの赤リンク」を一発で返す。
- `backlinks` / `related` / `path <a> <b>`（最短経路, BFS）/ `link-stats` / `peek`。

**B. ノードを知らない場合（キーワード起点）= 本当の「最初のとっかかり」**

- **`search <query>`**（`:2459`）: 本文 line の literal substring 検索。段階フォールバック:
  1. `lines.text LIKE '%q%'`（literal）
  2. 0件なら **sql_loose**（長音「ー」除去で再LIKE, `:2537`）
  3. それでも0件なら **python loose**（記号無視、行数上限 `PYTHON_LOOSE_SEARCH_MAX_LINES`）
  - `mode=boolean`（AND/OR）, `scope=page` も持つ。FTS5 trigram は検証済みだが **未実装**。
- **`suggest <partial>`**: タイトル部分一致。
- **`unresolved`**: 赤リンクを link_count 順にランキング。

### 外したときの "次の一手": `recovery_hints`

`read` / `path` でノード解決に失敗すると、grasp が自動で
`suggest` + `search` + `suggest_unresolved_targets` を束ねて `recovery_hints` として返す（`:2783`）。
LLM がゼロから探索戦略を立て直さなくていい。

### 分担

| 担当 | やること |
|---|---|
| **LLM** | どのコマンドを、どの title/keyword で叩くか **選ぶ** だけ |
| **grasp** | 正規化・ランキング・段階フォールバック・2-hop/最短経路計算・recovery_hints |

## Q4. recovery_hints のアルゴリズムは？ LLM 任せ？

**完全に決定論的。LLM 呼び出しは一切ない。** 3つの固定クエリを束ねるだけのファンアウト。

```python
recovery_hints = {
  "suggest":            self.suggest(query, limit),              # タイトル部分一致
  "search":             self.search(query, limit),              # 本文 substring（段階フォールバック）
  "unresolved_targets": self.suggest_unresolved_targets(query)  # 赤リンク部分一致
}
```

各手も SQL ＋文字列正規化だけ:

- **`suggest`**（`:1666`）: `norm_title LIKE '%q%'`、`CASE WHEN ... LIKE 'q%'` で **前方一致を先頭**、
  あとは views desc → title。
- **`suggest_unresolved_targets`**（`:2802`）: `unresolved_targets` を `target_norm LIKE` で引き、
  長音除去版（`loose_recovery_key`）も OR で足す。link_count 等でランキング。
- **`search` の正規化カスケード**:
  1. `normalize_title`（casefold＋空白畳み）
  2. `sql_loose_search_key`（`:4103`）= NFKC → **長音「ー」除去**
  3. `loose_search_key`（`:4109`）= 上記 ＋ **カタカナ→ひらがな**（コードポイント -0x60）

→ 「賢い探索アルゴリズム」ではなく、**解決失敗時に機械が必ず出す決まり切った候補集合**。
知能は正規化ヒューリスティックに固定で埋め込まれ、意味的判断だけが LLM 側に残る。

## Q5. embedding ベースの曖昧検索は実装されてる？

**実装されていない。** かつ wiki 上でも明示的に「未実装」かつ "今は入れない" 方針。

- `grasp/` のソースに embedding/vector/cosine 等は **一件もヒットしない**。
- `pyproject.toml`: `dependencies = []` → **依存ゼロ・標準ライブラリのみ**。
  numpy も sentence-transformers も sqlite-vec も無い。
- `wiki/grasp-backlog.md:130` に `- vector search。` と **1行だけ backlog 入り**（候補止まり）。
- `grasp-backlog.md:177` の Graph-native reasoning primitives は、出典の Tier 3
  =「**embeddings 無しの純グラフ操作で『Markdown 束には出せない』価値が出る所**」と定義。

### 実装されている「曖昧検索」の正体

- 曖昧吸収 → ルールベース正規化ラダー（長音除去・カナ→かな）
- 関連性 → グラフ構造（2-hop `related`、`path`、backlink ランキング）
- 高速化候補 → FTS5 trigram（これも未実装、`fts5-trigram-search.md`）

ベクトル類似度・意味的近傍・LLM rerank は **意図的に見送られている**
（`fts5-trigram-search.md` の「correctness 優先で決定論を維持」と同じ温度）。

## まとめ

grasp の検索系のテーゼは **「embedding 無しでどこまでやれるか」** に寄っている。

- 取り込み: Cosense JSON をパースして edge / 赤リンクを **import 時に materialize**、project ごとに正規化キー付きで丸ごと置換挿入。
- 検索: 固定 CLI コマンド集合（`read` / `search` / `suggest` / `related` / `path` / `unresolved` …）。
- 知能の所在: 正規化・ランキング・フォールバック・recovery_hints は **grasp 側に決定論で固定**。
  LLM は「どのコマンドを何で叩くか」の選択と意味的判断だけ。
- embedding / vector search / FTS5 hybrid は **未実装**（backlog 候補）。
