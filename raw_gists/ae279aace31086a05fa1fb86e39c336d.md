# Lean Q&A
## Q1: leanってどういう仕組みなの？

Leanの仕組みについて説明しますね。

### 基本的な考え方

Leanは「**証明＝プログラム**」という考え方（カリー＝ハワード対応）に基づいています。数学の命題を「型（type）」、その証明を「その型を持つプログラム（項）」として扱うんです。

たとえば「PならばQ」という命題は、関数の型 `P → Q`（Pを受け取ってQを返す関数）に対応します。この命題を証明するということは、実際にそういう関数を書くことと同じ、というわけです。

### 何をチェックしているのか

Leanの中核は「**型チェッカー（type checker）**」と呼ばれる小さなプログラムです。ユーザーが書いた証明を、Leanは厳密な推論規則に従って一段ずつ検証していきます。

- 各ステップが論理的に正しい推論規則に従っているか
- 最終的に証明したい命題の「型」と、書かれた証明の「型」が一致するか

これらを機械的にチェックします。型が合えば証明は正しい、合わなければエラーになる、という仕組みです。

### 土台になっている理論

Leanは「**依存型理論（dependent type theory）**」、特に Calculus of Inductive Constructions という理論をベースにしています。これは「値に依存する型」を扱える強力な体系で、たとえば「長さnのリスト」みたいに型自体に値を含められます。これによって、複雑な数学の概念もそのまま型で表現できるんですね。

### 実際の証明の書き方

実用上は、いきなり関数（証明項）を書くのは大変なので、「**タクティク（tactic）**」という対話的な仕組みを使います。「両辺を展開する」「帰納法を使う」「この補題を適用する」みたいな指示を順番に与えていくと、Leanが裏で実際の証明項を組み立ててくれます。ゴールが少しずつ変形されていって、最終的に「証明完了」になればOKです。

### 信頼性の担保

Leanの面白いところは、**カーネル（核となる検証部分）を意図的に小さく保っている**ことです。タクティクや証明探索など便利機能はたくさんありますが、最終的な正しさの判定は小さなカーネルだけが行います。だから「カーネルさえ正しければ、その上の便利機能にバグがあっても誤った証明が通ることはない」という設計になっているわけです。

ちなみに記事中の「ブラジル人の科学者」はLeonardo de Mouraさんですね。Microsoft Researchで開発が始まり、現在は数学者コミュニティ（mathlib という巨大な数学ライブラリの整備など）と一緒に発展しています。

## Q2: 具体例を示してほしい。命題とあるので論理学？しか扱えないのだろうか

いい質問です。論理学だけじゃなくて、**数論、代数、解析、幾何など数学全般**を扱えます。実際、フェルマーの最終定理の形式化プロジェクトも今Leanで進行中です。

具体例をいくつか見てみましょう。

### 例1: 単純な論理（PならばQ系）

「PかつQ ならば QかつP」という命題：

```lean
theorem and_comm (P Q : Prop) : P ∧ Q → Q ∧ P := by
  intro h
  exact ⟨h.2, h.1⟩
```

`intro h` で「P ∧ Q が成り立つと仮定して h と名付ける」、`h.1` がP、`h.2` がQ、それを逆順のペアにして返す、という証明です。

### 例2: 数論

「nが偶数ならn²も偶数」：

```lean
theorem even_sq (n : ℕ) (h : Even n) : Even (n * n) := by
  obtain ⟨k, hk⟩ := h     -- n = 2k と分解
  exact ⟨2 * k * k, by rw [hk]; ring⟩  -- n² = 2(2k²)
```

ここでは自然数 `ℕ` を扱っていて、「Even n」は「ある k が存在して n = k + k」という定義です。

### 例3: 代数の計算

「(a+b)² = a² + 2ab + b²」：

```lean
example (a b : ℝ) : (a + b)^2 = a^2 + 2*a*b + b^2 := by
  ring
```

`ring` は「可換環の等式なら自動で証明する」タクティク。実数 `ℝ` 上でも当然動きます。1行で終わります。

### 例4: 解析（極限）

「定数列の極限は定数」みたいな話も書けます：

```lean
example (c : ℝ) : Filter.Tendsto (fun _ => c) Filter.atTop (nhds c) := by
  exact tendsto_const_nhds
```

ε-δ論法もちゃんと展開できますし、mathlib という標準ライブラリには微積分・測度論・関数解析まで入っています。

### 例5: 幾何・トポロジー

「連続関数の合成は連続」とか、「コンパクト集合の閉部分集合はコンパクト」みたいな定理も全部形式化されています。

### なぜこんなに広く扱えるのか

最初に話した「依存型理論」が強力で、**集合・関数・群・環・位相空間・多様体…なんでも型として定義できる**からです。たとえば群なら：

```lean
class Group (G : Type) where
  mul : G → G → G
  one : G
  inv : G → G
  mul_assoc : ∀ a b c, mul (mul a b) c = mul a (mul b c)
  one_mul : ∀ a, mul one a = a
  mul_inv : ∀ a, mul a (inv a) = one
```

こんな感じで「群とは何か」を定義してしまえば、その上で群論の定理を全部証明できる、というわけです。

ちなみに mathlib（Leanの数学ライブラリ）は今や**150万行以上**あって、学部〜大学院レベルの数学のかなりの範囲がカバーされています。最近だと Terence Tao（テレンス・タオ）さんが自分の論文をLeanで形式化したり、ピーター・ショルツェさんの「Liquid Tensor Experiment」が完了したりと、最先端の数学でも実際に使われています。

なので「論理学しか扱えない」どころか、むしろ**現代数学そのもののデジタル化**を目指しているツールなんです。

## Q3: これ、定義次第では何でもつくれるよね。むしろつくった定義を則ったまま積み上げていくことを重視してる？で、数学はその積み上げをまさにやってきたものなので、それをleanの形に変換すればいい？

まさにその通りです。核心を突いた理解だと思います。

### Leanがやっていること＝定義からの積み上げ

Leanは「正しさ」を判定しているように見えて、実は**「与えられた定義と推論規則に忠実に従っているか」だけをチェックしています**。極端な話、変な定義をしても、それに従って正しく推論していればLeanは通します。

たとえばこんなこともできてしまいます：

```lean
def myWeirdAdd (a b : ℕ) : ℕ := a * b   -- 「足し算」を掛け算と定義
theorem silly : myWeirdAdd 2 3 = 6 := by rfl
```

これはLean的には完璧に正しい証明です。`myWeirdAdd` の定義に従えば 2 × 3 = 6 だから。「いや足し算ってそういうものじゃないでしょ」というのは**定義の妥当性の話**であって、Leanの守備範囲外なんですね。

つまりLeanは「定義の良し悪し」は判定しません。**定義から結論への道筋に飛躍やズルがないかだけを見ている**わけです。

### 数学はまさに「積み上げ」だった

おっしゃる通り、数学は19世紀末〜20世紀初頭にかけて「**全部を定義から積み上げ直す**」という大プロジェクトをやってきました。

- カントールが集合を定義
- ペアノが自然数を公理化
- デデキントが実数を有理数から構成
- ヒルベルトが幾何を公理化
- ツェルメロ・フランケルが集合論の公理系（ZFC）を整備
- ブルバキが「数学全体を集合論から積み上げる」という巨大プロジェクトを実行

この流れがあったからこそ、現代数学は「**最終的にはZFC公理系＋論理から全部導ける**」という建前で動いています。実際の数学者は普段そこまで遡りませんが、原理的には可能だ、という共通理解があるんです。

### だからLeanへの「変換」が成立する

Leanがやっているのは、まさにこの**原理的な積み上げを実際にコンピュータ上で実行する**ことです。基礎は依存型理論（ZFCとは少し違うけど同等以上の表現力）で、そこから：

```
基本的な型・命題
  ↓
自然数、整数、有理数、実数
  ↓
群、環、体、位相空間
  ↓
線形代数、微積分、測度論
  ↓
代数幾何、数論、…
```

という風に、人類が数百年かけてやってきた積み上げを、形式的に再構築しているわけです。mathlibの150万行というのは、まさにこの再構築の成果です。

### ただし「変換」は機械的じゃない

ここが面白いところで、**数学の論文をLeanに「翻訳」するのは結構大変**なんです。理由は：

**1. 論文は省略だらけ**
「明らかに」「同様にして」「読者に委ねる」みたいな表現は、Leanには通用しません。全部埋める必要があります。

**2. 暗黙の前提が多い**
人間の数学者は「文脈から当然こういう意味だろう」と読み取りますが、Leanは全部明示しないと動きません。

**3. 定義の選び方で難易度が変わる**
同じ概念でも定義の仕方は複数あって、どれを選ぶかで後の証明のしやすさが大きく変わります。mathlibでは「どの定義を採用するか」自体が議論の対象になります。

**4. 既存の補題を探すのが大変**
mathlibに既にある定理を再発明しないように、適切な補題を見つける作業も必要です。

### 結果として何が起きているか

Terence Taoさんが言っていることなんですが、Leanで形式化する過程で**元の論文の証明のギャップやミスが見つかることがよくある**そうです。人間が「自明」と思っていたところに、実は微妙な仮定が必要だったり、と。

つまりLeanは単なる「数学のデジタル化」ではなく、**数学の積み上げの厳密性を限界まで突き詰めるツール**として機能しているわけです。あなたの理解通り、「定義に則って積み上げる」ことを重視している ── というか、それしかしていない、というのが本質です。
