このドキュメントは、既存リポジトリ `grasp`（`D:\work\github\third\grasp`）を fork して、
**より薄い「Cosense JSON → SQLite グラフストア + 2系統検索」ツール** を作るために
Claude Code に渡す input。そのまま貼って作業を依頼する想定。

## 0. 前提・ゴール

- ベース: `grasp`（Cosense exported JSON を SQLite に取り込み、リンクのネットワークを CLI から辿る AI 向けツール）。
- 本 fork のゴール: **grasp の核（JSON→SQLite + リンク網構築）だけ残して大幅に薄くし、
  検索を「決定的」と「LLM 任せ（text-to-SQL）」の2系統にする**。
- grasp の `sqlite_store.py` は 4400 行あるが、その大半（正規化ラダー / recovery_hints /
  cross-project edge / markdown incremental / unresolved materialize 等）は本 fork では **不要**。

### 残すもの / 捨てるもの

**残す（核）**
- Cosense JSON のパース、特に `[リンク]` 記法の分類（=ネットワーク構築の肝、ここは手を抜かない）
- `norm_title`（タイトル正規化）によるリンク解決
- 双方向リンク・赤リンク・2-hop related・最短経路（グラフ操作）
- マルチプロジェクト対応

**捨てる（薄くする）**
- 行（lines）テーブルの分割 … 行リンクが要らなければ本文1カラムでよい（下記で要判断）
- `unresolved_targets` の事前 materialize … 都度 GROUP BY / VIEW で十分（大規模化したら後で）
- 正規化ラダー（長音除去・カナ→かな等の loose 検索フォールバック）
- `recovery_hints`（解決失敗時の候補ファンアウト）
- cross-project edge の materialize、markdown incremental import

## 1. 一番重要: リンクパースは grasp から移植する

「`[リンク]` を辿ってネットワークを作る」部分が唯一シンプルにならない難所。
Cosense の `[...]` は多重定義されているので、内部ページリンクだけ正しく拾うには分類が要る。

分類対象（grasp の挙動に合わせる）:

- `[ページ名]` → 内部リンク ✅
- `[http://...]` / `[https://... 表示名]` → 外部URL ❌
- `[/project/page]` → cross-project リンク（内部エッジにはしない）
- `[* 太字]` / `[- 取消]` / `[/ 斜体]` → 装飾 ❌
- `[$ 数式]` / `[foo.icon]` / `[foo.img]` → 記法 ❌
- `[[太字]]` → 二重ブラケットは太字でリンクではない ❌
- `` `[コード内]` `` → インラインコード内は無視 ❌
- `#ハッシュタグ` → 内部リンク（境界判定あり）✅

**指示: grasp の `grasp/cosense.py` にある以下を移植・流用すること（ゼロから書き直さない）。**
- `parse_cosense_links()`（`cosense.py:51`）
- `is_internal_cosense_link()`（`cosense.py:26`）
- `parse_cosense_hash_tag()` ほかハッシュタグ境界判定（`cosense.py:182`〜）
- `normalize_title()`（`cosense.py:21`、casefold＋連続空白畳み）
- cross-project は `parse_cosense_cross_project_links()`（`cosense.py:113`）を参考に、
  本 fork では「内部エッジから除外する」だけでよい（横断は将来）。

## 2. スキーマ（最小）

```sql
CREATE TABLE pages (
  project TEXT, id TEXT, title TEXT, norm_title TEXT,
  created INT, updated INT, body TEXT,        -- 行を分けず本文まるごと（行リンク不要なら）
  PRIMARY KEY(project, id)
);
CREATE TABLE links (                          -- 導出した有向エッジ
  project TEXT, source_id TEXT,
  target_title TEXT, target_norm TEXT,
  line_idx INT                                -- 行リンクを残す場合のみ
);
CREATE INDEX ix_links_target ON links(project, target_norm);
CREATE INDEX ix_pages_norm  ON pages(project, norm_title);
```

これで成立する操作:
- **backlink** = `SELECT * FROM links WHERE project=? AND target_norm=?`（双方向は導出、保存しない）
- **赤リンク** = `links.target_norm` のうち `pages.norm_title` に存在しないもの（都度 GROUP BY / VIEW）
- **2-hop related** = `links` を2回 JOIN

**判断ポイント（実装前に決める）**: 行リンク（どの行からのリンクか）を残すか。
- 残さない → `pages.body` 1カラムで最小。
- 残す → grasp 同様 `lines` テーブルを足し、`line_id = "{page_id}:{index}"`、`links.line_idx` を埋める。
- 推奨: **まず残さない（body 一本）で始める**。必要になったら lines を足す。

### `norm_title` は捨てない（重要）

「シンプルに」で消したくなるが、`norm_title`（casefold＋空白畳み、できれば NFKC も）だけは必ず残す。
無いとリンク解決が完全一致になり、大文字小文字・全半角の揺れで双方向リンクが切れる。
列1個のコストで効く（grasp の identity-without-name の肝）。

## 3. マルチプロジェクト: 別 .db ファイル方式（推奨）

| 案 | 中身 | 採否 |
|---|---|---|
| A. project 列（grasp 方式） | 1 DB に全 project、各テーブルに `project` 列 | cross-project 横断をやるなら必須。今回は見送り |
| **B. project ごとに別 .db** | `store/<project>.db` | **採用**。実装最小、独立、削除も楽 |

本 fork は「もっとシンプルに」が主目的なので **B（ファイル分割）** を採用する。
JOIN もロックも考えなくてよい。`[/project/page]` の横断検索が欲しくなった時点で初めて A 相当に寄せる。
（スキーマの `project` 列は B でも冗長に持っておいてよいが必須ではない。）

## 4. 検索: 2系統

### (1) 決定的検索
- `LIKE '%q%'` で始める（正しさ優先）。
- 日本語は2文字クエリが多い。FTS5 trigram は速いが literal substring semantics を壊す罠がある
  （grasp の `wiki/entities/fts5-trigram-search.md` 参照）。最初は入れない。

### (2) LLM にクエリを任せる検索（text-to-SQL）= 本 fork のオリジナリティ
grasp があえて持たなかった部分。素直に text-to-SQL で実装する。

- スキーマ（§2 の DDL）をプロンプトに入れ、LLM に `SELECT` を書かせる → 実行 → 結果整形して返す。
- **安全策必須: 接続を読み取り専用で開く。**
  ```python
  con = sqlite3.connect("file:store.db?mode=ro", uri=True)
  con.execute("PRAGMA query_only = ON")
  ```
  `mode=ro` で破壊は防げるが `query_only` も併用が堅い。LLM が `DROP`/`UPDATE` を書いても弾く。
- グラフ操作（backlink / 2-hop / 赤リンク）は SQL で自然に書けるので、
  text-to-SQL は「決定的コマンドに無い任意の切り口」を開く役割。

**役割分担を UX に出す**: 決定的＝速い・再現可能・予測可能 / LLM＝任意・柔軟・非決定的。

## 5. CLI surface（最小案）

grasp のサブコマンド群を参考に、最小限へ削る。

- `import --cosense <json> [--project <name>]` … JSON 取り込み（project 省略時はファイル stem か JSON の `name`）
- `read <title>` … 1ページ + backlinks + related(2-hop) + 赤リンクを返す（核となる「近傍同梱」）
- `backlinks <title>`
- `related <title>` … 2-hop
- `search <query>` … 決定的（LIKE）
- `ask <query>` … LLM text-to-SQL（read-only）  ← 新規
- `unresolved` … 赤リンク一覧
- （任意）`path <a> <b>` … 最短経路。最初は省いてよい

出力は JSON（`--json`）を基本に、AI が消費しやすい形にする。

## 6. 作業手順（Claude への依頼内容）

1. `grasp` を fork（or 新規リポジトリにコピー）し、新パッケージ名で開始。
2. `cosense.py` のリンクパース関連関数（§1 のリスト）を移植。テストも一緒に持ってくる
   （`tests/test_cosense.py` の該当部分）。
3. §2 のスキーマで SQLite ストアを実装（まず body 一本・行リンク無しで）。
   import は「JSON パース → pages 挿入 → links 導出して挿入」だけ。再 import は project 単位で置換（冪等）。
4. マルチプロジェクトは §3 の別ファイル方式で。
5. 決定的検索（LIKE）と `ask`（read-only text-to-SQL）を実装。
6. CLI を §5 の最小 surface で組む。
7. 単体テスト: パース分類・リンク解決（norm 揺れ）・backlink 双方向・赤リンク・2系統検索。

### 受け入れ条件
- Cosense JSON を入れて `read <title>` で backlinks/related/赤リンクが返る。
- タイトルの大文字小文字・全半角・空白揺れがあっても双方向リンクが繋がる（`norm_title` 経由）。
- 複数 project を別ファイルで持て、混ざらない。
- `search` が決定的に動く / `ask` が read-only で LLM 生成 SQL を安全に実行する。
