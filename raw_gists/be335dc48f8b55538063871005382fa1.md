# OKF と Metadata as Code をめぐる考察

OKF（Open Knowledge Format）の仕様とサンプルを読みながら、「metadata as code とは何か」を起点に、データ／ナレッジ／フォーマットの境界を順に詰めていった議論のまとめ。

---

## 1. Metadata as code とは

カタログのメタデータを専用レジストリ／サービスに預けるのではなく、**ソースコードと同じくプレーンテキストのファイルとして書き、Gitで管理する**考え方。Infrastructure as Code の発想をメタデータに持ち込んだもので、本質は「**何を真実の源（source of truth）にするか**」の選択。

| | 従来のカタログ（registry型） | Metadata as code |
|---|---|---|
| 真実の源 | サービス内のDB（Collibra, Dataplex等） | リポジトリ内のファイル |
| アクセス手段 | 専用API/SDK/クエリ言語 | `cat` / `git clone` |
| 変更履歴 | サービス独自の監査ログ | git log / blame / diff |
| レビュー | 管理画面のワークフロー | Pull Request |
| 可搬性 | ベンダーロックイン | ディレクトリごと持ち出せる |

OKFはこれを「YAML frontmatter付きmarkdownのディレクトリ」という最小形式に固めた仕様。**厳格なスキーマ保証を犠牲にして、可読性・可搬性・人とエージェントの協働可能性を最大化する**という一貫したトレードオフが貫かれている。

---

## 2. マスターはどちら側か（仕様駆動開発との対比）

「ドキュメント側をマスターにするのか？」——答えは **「する。ただし全部ではなく、宣言した部分だけ」**。

メタデータには性質の違う二層があり、マスターの所在が層ごとに違う:

| 層 | 例 | マスターはどちら | 同期の向き |
|---|---|---|---|
| **観測された事実** | 物理スキーマ、データプロファイル、クエリ使用統計 | **データ側（BigQuery/Catalog）** | source → file（読み取り専用） |
| **キュレーションされた知識** | overview、生成description、推奨クエリ、guidelines、glossaryリンク | **ファイル側** | file → service（publish） |

mdcode の `catalog.yaml` がこれを構造で表現している:

- `snapshot:` … サービスから**ローカルに引いてくる**もの（`data-profile`, `query-usage` を含む）
- `publishing:` … ファイルを**マスターとして書き戻す**もの（`overview`, `descriptions`, `queries`, `guidelines` など）

観測専用フィールド（`data-profile` 等）は snapshot にあって publishing に無い＝「ローカル文脈としては読むが、書き戻さない」。

IaC の比喩が近い:

- `snapshot` = `terraform refresh`（現状をstateに取り込む）
- `publishing` = `terraform apply`（ファイルを正として現実を寄せる）
- 観測専用フィールド = data source（読むだけ、applyの対象外）

→ **真実の源が層ごとに分かれ、その境界を `catalog.yaml` で宣言する**のが肝。仕様駆動開発より一歩進んだ「選択的・双方向同期」。

## 3. 「カタログ」とは

3つの意味で使われる:

| 用語 | 何を指すか |
|---|---|
| データカタログ（一般概念） | データ資産のメタデータを集めた目録・索引システム（図書館の蔵書目録と同じ発想） |
| Knowledge Catalog | その具体的サービス（Dataplexベース、Entry / Aspect / EntryLink で管理） |
| `catalog.yaml` / `catalog/` | そのカタログをローカルにファイル化したスナップショットと設定 |

三層構造:

- **データ本体** → BigQuery にある
- **そのデータについての知識** → Knowledge Catalog（サービス）が持つ
- **その知識をファイルにした表現** → OKF / metadata-as-code

---

## 4. 「データ」とは（データ資産）

ここでの「データ」は「ソフトウェアが扱うあらゆるバイト列」ではなく、**データベース／データウェアハウスに溜まった、分析・業務に使う構造化されたデータ資産**（data asset）。

| | データカタログの「データ」か |
|---|---|
| BigQueryの注文テーブル、ユーザーイベント | ✅ ど真ん中 |
| 業務DBのテーブル | ✅ そう |
| アプリ実行中のメモリ上の変数 | ❌ 違う（実行時の一時状態） |
| 設定ファイル、ログの1行 | ❌ ふつう対象外 |
| ソースコードそのもの | ❌ それは「コード」 |

コード／データ／メタデータの区別:

| 用語 | 中身 | 例 |
|---|---|---|
| **コード** | 処理の手順 | `SELECT ... FROM orders`、Pythonスクリプト |
| **データ** | 処理される対象の中身 | orders に入っている注文100万件 |
| **メタデータ** | データについての説明 | 「ordersは注文1件1行。order_idは一意」 |

---

## 5. データとナレッジは段階が違う

リポジトリ名が `knowledge-catalog`、フォーマットが Open **Knowledge** Format なのは意図的。仕様冒頭も "knowledge — the metadata, context, and **curated insight**" と宣言している。

古典的な DIKW の梯子:

| 段階 | 中身 | orders テーブルでの例 |
|---|---|---|
| **データ** | 生の値、文脈なし | `order_id=A1, total_usd=42.00` の100万行 |
| **情報** | 値に意味・構造がついたもの | 「total_usd は注文金額（USD）」 |
| **ナレッジ** | 人/エージェントが解釈・関連づけ・判断を加えたもの | 「freshness alertが出たらまずingestion jobを見ろ」 |

本質的な違い: **データは掘れば出てくるが、ナレッジは誰かが書かないと存在しない**。

- データは*抽出*するもの（機械が観測できる事実）
- ナレッジは*著述*するもの（人の経験・判断・文脈）

「データの延長」と捉えるのは半分正しく半分違う。ナレッジはデータに紐づく（`resource:` で実テーブルを指す）が、生成のされ方・真実の源・更新の仕方が違う。**ナレッジはデータの上に乗る別の層**であって、地続きだが同じものではない。

---

## 6. OKF が定義しているのは「ナレッジ」か「メタデータ」か

OKFが規範として強制しているのは、ほぼ**メタデータと構造（封筒）だけ**:

- 必須は実質「frontmatterがパースでき、`type` が非空」の一点のみ
- 本文について仕様は明言: **"There are no required body sections."**
- 未知のtype・欠けたフィールド・壊れたリンクを拒否してはならない（寛容な消費モデル）

→ **OKFが強制する唯一の中身は「これは何の種類の概念か」というタグ一個**で、これは紛れもなくメタデータ。ナレッジ本体は形式上は空でも適合する。

ただし関係は「入力→出力」ではなく「**外装→中身**」:

- frontmatter（メタデータ）= 封筒の宛名 → 機械がルーティング・検索・グラフ化する面
- 本文（ナレッジ）= 封筒の中身 → 人とLLMが実際に読む面

| | OKFが定義する | OKFが定義しない |
|---|---|---|
| 対象 | メタデータ＋構造（封筒） | ナレッジの中身（本文） |
| 強制力 | `type` 必須、構造規則 | 本文は自由・必須セクション無し |
| 役割 | 運ぶ・見つける・つなぐ | 書く・読む・判断する |

> OKFの正体は "Knowledge **Format**" というより "Knowledge **Packaging** Format" に近い。

---

## 7. OKF はナレッジを作る装置ではない

**OKFはナレッジを作る装置ではなく、作ったものを載せる宛先**。「作る」のは人かエージェント、OKFは成果物の置き場であり流通形式。

ただし時系列を完全分離して「作った後にあとから載せる」と捉えるのはややズレる。実装上、エンリッチメントエージェントは**最初からOKFの形に向かって書いている**:

- ❌ 知識を完成させる → 別途OKFに移し替える
- ✅ OKFという器**に直接書き込みながら**知識を作る

それでも「フォーマットそのものは執筆を助けない／生成手段に中立」という核心は揺るがない:

> **OKF（フォーマット）＝作らない。器と作法を決めるだけ。**
> **エージェント／人＝作る。その成果をOKFの形で書き出す。**

---

## 8. 「LLM Wiki がベース」に面影がない問題

仕様 §10 は OKF を "LLM wiki repositories" に近いと位置づけるが、実物にwikiの面影は薄い。理由は、**主張している"近さ"が見た目や性格ではなく"配管（フォーマットの仕組み）"のレベルだけ**だから。

配管レベルなら一致する:

- markdownファイルのディレクトリ ✅
- YAML frontmatter ✅
- 概念間をmarkdownリンクで相互参照（≒wikilink）✅
- index.md でナビ／progressive disclosure（≒wikiの目次）✅

しかし**性格**はまるで違う:

| LLM wiki のイメージ | OKFの実物（`events_.md`） |
|---|---|
| 人が書いた散文中心の記事 | エージェント生成の機械的な列定義300行 |
| 百科事典的・自由形式 | `type: BigQuery Table` / `resource:` でデータ資産に固定 |
| 有機的に育つ・緩い構造 | データ資産の階層をそのまま写したリジッドな構造 |
| 人の協働編集が主 | 生成→PRレビューが主、出自はカタログのexport |

**性格を決めているDNAは wiki文化ではなく、Dataplex系のデータカタログ（Entry / Aspect / EntryLink モデル）**:

- **wikiから来たもの**：器の素材（md + frontmatter + リンク + index）＝**配管**
- **データカタログから来たもの**：何を概念とし、どう型付けし、何に紐づけるか＝**性格**

→ **仕組みはwikiの末裔だが、顔つきはデータカタログ**。§10末尾が "OKF differs primarily in being **specified**" と締めている通り、wikiは「同じフォーマット族だが我々はそれを仕様化した」という位置づけで挙げられているにすぎない。

---

## 結論

- **Metadata as code** = メタデータの真実の源をサービスからリポジトリへ移し、Git・PR・diff という開発ワークフローをそのまま知識管理に転用する設計思想。
- **OKF** = その思想を「YAML frontmatter付きmarkdownのディレクトリ」に固めた、**ナレッジを載せて流通させるための最小のメタデータ＋構造の規約**。ナレッジそのものを作りも規定もしない。
- マスターは層ごとに分かれ（観測事実＝データ側／キュレーション＝ファイル側）、その境界を `catalog.yaml` で宣言する選択的・双方向同期。
- 素材はwikiの末裔だが、性格はデータカタログ。名前の "Knowledge" に引きずられず、実体は **Knowledge Packaging Format** と読むのが正確。
