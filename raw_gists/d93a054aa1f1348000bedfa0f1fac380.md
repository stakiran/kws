# q
GitHubを介した議論の仕組みを立ち上げようとしている。

- 一つのリポジトリをつくって、1話題1フォルダを掘る
- 1フォルダの中には人物ごとにファイルをつくる。また人物としてAIエージェントを定義することもできる
- リモートとの同期はpushとpullのみを使い、またgitの操作はエージェントに任せる

これにより、各人は衝突することなく話題ごとに書いていけます。こうして様々な話題に関する意見を並べることで議論をしていこうというものです。

# a
設計としては成立します。衝突（コンフリクト）を避けつつ「話題ごと・人物ごと」に並列で書けるので、GitHubのIssue/Discussionよりも“文章を積み上げる討議ログ”に向きます。実装するなら、次の点を決めておくと運用が安定します。

## 1) リポジトリ構造（最低限の規約）
- `topics/<topic-id>_<slug>/`
  - `README.md`（話題の概要・論点・現在の結論・未決事項）
  - `participants/`（任意：参加者定義）
  - `statements/`
    - `<person>.md`（人物ごとの主張を追記していく方式）
    - もしくは `<person>/<yyyy-mm-dd>-<n>.md`（発言をファイル分割する方式）
- 迷ったら「発言をファイル分割」がおすすめです。  
  同一ファイル追記は、同時編集が起きたときに衝突しやすいからです（“人物ごと”でも同時に同じ人物ファイルを書いたら衝突します）。

## 2) 衝突を本当に避けるためのルール
あなたの案の「人物ごとにファイル」は良いですが、さらに安全にするなら：

- **1発言 = 1ファイル**（追記ではなく新規作成）
- ファイル名に **時刻＋ランダム（またはUUID）** を入れる  
  例：`statements/taro/2026-02-14T101500Z_a3f9.md`
- これで、複数人（やAI）が同時に書いても基本衝突しません。

## 3) 議論として成立させるためのメタ情報
単に文章が並ぶだけだと収束しづらいので、各発言にヘッダを持たせると便利です。

例（Markdownの先頭にYAML）：
```md
---
author: taro
role: human
timestamp: 2026-02-14T10:15:00Z
topic: 001_remote-work-policy
stance: pro | con | neutral
replies_to: 2026-02-14T09:50:00Z_xxxx
claims:
  - id: C1
    text: "AはBよりコストが低い"
evidence:
  - url: ...
---
本文…
```

最低限は `author / timestamp / replies_to` があると「会話の接続」が追いやすいです。

## 4) AIエージェント運用の注意点（push/pullだけで回すなら）
- エージェントが勝手に書き換えないように、基本は **追記（新規ファイル追加）のみ**に権限/ルールを寄せる
- `main` 直pushを許すか、PR必須にするかを決める  
  - **衝突回避と速度**優先：直push（ただし保護ルールは弱めになる）
  - **品質と荒らし耐性**優先：PR（ただしレビューが必要で遅くなる）
- “push/pullのみ”でも、GitHub側の保護（branch protection）やCODEOWNERSでガードはできます

## 5) 「議論」になりやすい補助ファイル
話題フォルダに次を置くと収束しやすいです。
- `README.md`：現状の要約（誰かorエージェントが随時更新）
- `questions.md`：未決の問いの箇条書き
- `decisions.md`：合意事項と根拠、反対意見も併記
- `glossary.md`：用語揺れ防止

## 6) この方式のメリット/デメリット
**メリット**
- 非同期で強い、履歴が残る、話題単位に整理できる
- “人ごとファイル”で責任境界が明確
- 生成AIが参加してもログとして監査しやすい

**デメリット**
- Gitに慣れてない人には心理的コスト
- 体系化（要約・争点整理）をしないとログが増えるだけ
- 検索・通知はIssue/Discussionより弱い（ただし工夫で補える）

1) 参加者は何人規模？（10人/100人/社外含む等）

## 1) フォルダ命名（タイムスタンプ入り）案
目的が「衝突防止＋俯瞰性」なら、**時系列ソートできて、人間が読める**形式が良いです。

例（推奨）：
- `topics/2026-02-14_1530__remote-work-policy/`

  - `__`以降に短いslugを入れて俯瞰性を確保

- `topics/2026-02-14_1530__remote-work-policy__a3f9/`

「秒」まで入れるだけでも実用上かなり減ります：
- `topics/2026-02-14_153012__remote-work-policy/`

---

## 2) ディレクトリ構造（最小・運用安定）
「細かいファイル分割なし」「front matterなし」前提で、構造はシンプルにしておくのが正解です。

```
topics/
  2026-02-14_1530__remote-work-policy/
    README.md        # 論点・問い・前提・参考リンク（編集担当は決める）
    participants.md  # 参加者一覧（任意。handleと表示名だけでも）
    statements/
      taro.md
      hanako.md
      agent_alpha.md
```

- 各人は **自分の `statements/<name>.md` だけを編集する**（最重要ルール）
- これで「人同士のコンフリクト」は基本消えます  
  ※ただし「同一人物が複数端末/複数エージェントから同時に編集」すると競合します。ここが唯一の穴。

---

## 3) “1人1ファイル追記”で競合を現実的に潰す方法
あなたの条件だと競合の主原因はこれです：

- 同じ人物ファイルを、別タイミングで pull せずに追記 → push
- mainが進んでいたら non-fast-forward で弾かれる
- pullしてマージ時に同じ行近辺を触っているとコンフリクト

これに対して、**ルール＋エージェントの手順**で吸収します。

### 3.1 追記フォーマットを固定して「機械マージ」しやすくする
front matterは不要でも、各人ファイルの追記単位を揃えると競合が激減します。

例：各発言をこの“ブロック”で追記（手書きでOK）
```md
## 2026-02-14 15:45

（本文…）

---
```

ポイント：
- 末尾に必ず区切り `---` を入れる
- **常にファイル末尾に追記**する運用にする（途中編集は極力しない）
- 後から推敲したい場合は「追記で訂正」を基本にする  
  （どうしても過去部分を直すと競合しやすい）

### 3.2 エージェントのpush手順を「必ずrebase」に寄せる
「push/pullのみ」縛りだと、実際は pull 時に merge が走って汚れたり競合したりします。ここはエージェント側の実装でカバーできます。

エージェントに以下を徹底させると安定します：

1. `git pull --rebase`
2. 自分のファイル末尾に追記
3. `git add statements/<me>.md`
4. `git commit`
5. `git pull --rebase`（push直前にも再同期）
6. `git push`

これで「他人のコミットが先に入っていた」ケースをかなり吸収できます。

### 3.3 同一人物の同時編集を禁止（ソフトロック）
唯一残るのが「同じ人物ファイルを同時に触る」問題です。人間なら意識で避けられますが、AIエージェントを複数走らせると起こります。

軽量な対策（ブランチ無しでも可能）：
- **同一人物IDのエージェントは常に1つだけ稼働**（運用ルール）
- どうしても並列化したいなら、ローカル側でキューイング（順番にコミット）する

リポ内でロックファイルを作る方法もありますが、ロックファイル自体が競合点になるのでおすすめしません。

## 4) READMEの位置づけ（意思決定しないなら“問いの整理”に寄せる）
「意思決定は扱わない」場合、READMEは結論を書くより、次の2つが強いです。

- **前提の固定**（議論が前提違いで拡散するのを防ぐ）
- **未解決の問いの棚卸し**（読む人が何を読めばよいか分かる）

READMEテンプレ（簡素）例：
```md
# テーマ：リモートワーク規定

## 目的
意思決定はしない。意思決定に必要な情報と、各人の思考のログを残す。

## 前提・制約
- 対象組織：
- スコープ：
- 期限：

## 主要な問い
- Q1:
- Q2:

## 参考資料
- ...
```

このREADMEの更新担当を「人」か「要約エージェント」のどちらかに固定すると、俯瞰性が維持されます。

## 5) main一本運用での最低限の安全策
ブランチを切らないなら、破壊的変更や事故を避けるための“ゆるい柵”だけ置くのが実務的です。

- `topics/**/statements/*.md` 以外は原則触らない（運用ルール）
- 可能ならGitHub側で
  - 強制push禁止
  - 直push可にするなら最低限「履歴改変しない」文化を徹底

※「ブランチ保護を強くするとPR必須」になりがちで、あなたの要件と衝突します。ここは“事故が起きない運用”の方が重要です。

## 「alice.md は alice しか触らない」なら、通常はコンフリクトではなく“弾かれる”だけ
Gitで起きる典型はこうです：

- Aliceがローカルで `alice.md` を編集してコミット
- その間に、別の場所から（= 同じAliceが別端末/別エージェントで）`alice.md` を更新して先にpushしていた
- Aliceがpushすると **non-fast-forward** で拒否される
- ここで `pull`（= fetch + merge/rebase）をすると、両方とも `alice.md` を変更しているので、**差分の当たり方によってはコンフリクトが起き得る**

つまり「alice しか触ってない」場合に残るリスクは **“同一人物による並行更新（複数端末・複数エージェント・同時実行）”** で、これは *他人との競合* ではなく *自分自身の競合* です。

逆に言うと、あなたの仕組みが安定する条件はシンプルで：

- 各 `person.md` について、**同時に複数の書き込み主体を走らせない（直列化する）**
  - 「その人のエージェントは常に1つだけ稼働」
  - あるいは「書き込み要求をキューに入れて順番にコミット」

これさえ守れれば、フォーマット自由でもほぼ破綻しません。

## 「pull と自動解決で吸収」の現実的なやり方
“自動解決”を確実にしたいなら、エージェントの内部手順は次のどれかに寄せるのが安全です。

- **(推奨) rebase運用**：`git pull --rebase` を基本にして、push直前にももう一度 `pull --rebase`
- それでもコンフリクトした場合：
  - 可能なら「人間に解決を促す」
  - あるいはエージェントが **両方の内容を残す** 形で解決（例：片方を末尾に退避して“要手動整理”と印を付ける）
    - ※“内容を捨てない”が最重要です。自動マージで綺麗にするより、ログの完全性を優先。

ただしここでも、そもそも「同一人物の同時更新」を直列化できていれば、コンフリクト処理自体がほぼ不要になります。

## 結論
- フォーマット統一を捨てて「各自が好きに書く」は、この用途（深い思考のログ、意思決定しない）に合っています。
- 成否を分けるのはフォーマットではなく、**同一人物ファイルへの書き込みを必ず直列化すること**です。
- 直列化 + `pull --rebase` で、あなたの「push/pullのみ・main一本・1人1ファイル」でも十分運用可能です。

必要なら、「同一人物の直列化」を実現する具体案（ローカルキュー、エージェントの排他、コミット単位の設計）を、あなたのエージェント実装形態（1人1プロセス？サーバ常駐？複数ジョブ？）に合わせて詰められます。
