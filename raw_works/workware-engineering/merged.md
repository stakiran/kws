# ワークウェアとは
ワークウェアの定義、成立条件、関連用語を述べます。

## ワークウェア
定義は以下のとおりです。

**ワークウェア(Workware)** とは、

> 仕事に関するやり方・あり方・考え方・捉え方といった概念のこと。

> 仕事で使う概念的道具のこと。

### 定義の説明
まず **「概念」** であるという点がポイントです。この「概念」は非常に幅広いもので、定義上でもかなり広く捉えています。

次に **「道具」** であることも外せません。つまりワークウェアとは使うものです。

最後に **「仕事」** というキーワードもあります。ここでいう「仕事」も広義なものを指し、[コトバンク](https://kotobank.jp/word/%E4%BB%95%E4%BA%8B-73099) から引用すると `『何かを作り出す、または、成し遂げるための行動』` （デジタル大辞泉）や `『すること』『しなくてはならないこと』` （精選版 日本国語大辞典）を指します。なので、いわゆる労働はもちろん、趣味や日常生活における活動や行動も含むことができます。

### ワークウェアが満たすべき3つの条件
定義を満たすとはどういうことか、を3つの条件として定めています。

| No | 条件名 | 解説 |
| ---- | ---- | ---- |
| 1 | Named      | 名前がついていること |
| 2 | Conceptual | 観念ではなく概念であること |
| 3 | Practical  | 実用的であること |

まず Named、名前がついている必要があります。名前のないものはワークウェアではありません。

次に Conceptual、観念ではなく概念でなくてはなりません。ここで観念とは自分にしか理解できない、あるいは自分でも理解できないものを指し、一方、概念とは他者が理解できるものを指します。つまり他者にも理解できるように、言語化して表現せよ、ということです。

もう一つは Practical、実用的であることも求めます。これは要するに「使える」ことです。ワークウェアは道具ですので、使えるものである必要があります。概念を使うというのはわかりにくいですが、たとえば「言及できる」「生かせる」「試せる」「つくって動かせる」が挙げられます。

## ワークウェアエンジニアリング(WWE)
ワークウェアを扱うことを **ワークウェア・エンジニアリング(Workware Engineering)** と呼びます。省略すると **WWE** です。いちいち「ワークウェア・エンジニアリング」と書くと長ったらしいので、この略語はよく使います。

扱うとはつくること、改造すること、単に使うことから探すことも含みます。もちろんやり方は各自が自由に行えばいいのですが、先人の知恵は活用したいものです。ワークウェアを扱うのにも色々とコツや原則があります。万人が上手く扱えるようにするために、エンジニアリング（工学）として整備するのです。

さて、工学というと難しく聞こえますが、単に先人や筆者の知恵を誰でも使えるように上手くまとめましょう、というだけです。学問というほどかしこまってはいませんし、科学や数学、あるいはプログラミングのような厳密な世界でもありません。もちろん、本書でまとめた WWE が WWE のすべてではありません。

とはいえ、このような試みはありそうでなかったため、本書における WWE は、WWE なる理論の出発点になると思います。本書もそのつもりで書いています。

## メタワーク
ワークウェアをつくることを **メタワーク(Metawork)** と呼びます。 -ing をつけるとメタワーキングです。

既存の言葉と対応させて説明すると、書き物をつくることをライティング、ソフトウェアをつくることをプログラミングと呼びますが、同樣にワークウェアをつくることをメタワーキングと呼びます。

直接すると「仕事のための仕事」といった意味になります。

### (余談) メタワークを定義するまで
メタワークとは、直訳すると「仕事のための仕事」や「超仕事」となります。要するに仕事そのものではなく、仕事をするために必要な何かをするということです。

たとえばタスクをこなしていくことは（直接的に）仕事をしていると言えますが、タスクをこなすためにタスク管理を行うことや健康な心身を維持することは（直接的な意味での）仕事ではないでしょう。しかし、全く関係無いかというと、そうではなく、間接的な仕事とでも言えます。さて、直接的にせよ、間接的にせよ、仕事を行う際のやり方は様々です。もっと言えば考え方や捉え方も違うでしょうし、使う道具も人それぞれです。

以上を整理すると、次のようになります。

- 仕事
    - 直接的な仕事（タスクをする）
    - 間接的な仕事（タスク管理をする、体調維持に努める）
- 仕事で使う道具
    - 物理的な道具（そのうち機械はハードウェアと呼んだりします）
    - OS、プログラム、アプリ、Webサービスなど（ソフトウェアと呼ばれます）
    - ★ やり方・考え方・捉え方など（これをワークウェアと呼びます） 

特に ★ 部分を指す言葉がないのでワークウェアと名付けました。また、このワークウェアをつくるという行為を指す言葉もないので、メタワークと名付けました。

純粋に考えるなら、メタワークとは「仕事で使う道具」をつくること全般、となるでしょう。しかし、このような定義にしても旨味はありませんし、物理的な道具やソフトウェアについてはすでに言葉があります（工作、製造、プログラミング等）。欲しいのは「ワークウェアをつくる」という部分ですから、ここに対してメタワークという言葉を充てることにしました。

# ワークウェアのイメージをつかむ
ワークウェアなるもののイメージを掴んでいただくために、色々な説明を試みます。

## ワークウェアは万人向けの第4潮流
ワークウェアと聞いてライフハックや仕事術、あるいは自己啓発といったものを思い浮かべた方もいらっしゃるでしょう。かなり意識しています。というより、それらを改めてまとめなおしたものがワークウェアです。

 No | 名前 | 時期 | 主な対象 | 情報 |
| ---- | ---- | ---- | ---- | ---- |
| 1 | 知的生産の技術 | 1970 | 研究者 | [★1](ref#1) |
| 2 | 仕事術・自己啓発 | 2001 | ビジネスマン | [★2](ref#2) |
| 3 | ライフハック | 2005 | インターネット利用者 | [★3](ref#3) |
| 4 | ワークウェア | 2024 | 万人 | 本書 | 

ワークウェアは、以前から存在する知的生産の技術、仕事術、自己啓発、ライフハックなどをすべて含みます。

加えて、各自が自分なりのワークウェアをつくることもできます。この、いわば概念的道具の DIY ともいうべき着眼点は大きな特徴です。ライフハック以前の潮流ではあくまでも「誰かがつくったものを使う」だけでしたが、ワークウェアは違います。自身でつくることもできるのです。そして、これを誰でも行えるように整備したものがワークウェア・エンジニアリングです。

## 発明、芸術、哲学とワークウェア
ワークウェア（というよりワークウェアをつくる「メタワーク」という営み）は、発明・芸術・哲学との立ち位置で説明することもできます。

次のとおりです。

|            | 実用的である | 実用的でない |
| ---------- | ------------ | ------------ |
| 実物がある | 発明  | 芸術  | 
| 実物がない | **メタワーク** | 哲学 | 

直感的な感覚を言うと、メタワークとは発明という実用実物の世界と哲学という概念的な世界を足して二で割ったようなものです。

## ハードウェア、ソフトウェア、ワークウェア
ワークウェアという名前を聞いて、ハードウェアやソフトウェアを思い浮かべた方も多いかと思います。そのとおり、意識しております。というより、それらに続く第三のポジションであると考えています。

 No | 名前 | イメージ | 何を扱うか(-的) |
| ---- | ---- | ---- | ---- |
| 1 | ハードウェア/Hardware | 固体(Solid) | 物理的 |
| 2 | ソフトウェア/Software | 液体(Liquid) | 電子的 |
| 3 | ワークウェア/Workware | 気体(Gas) | 概念的 |

ワークウェアは概念的道具であると定義しましたが、概念というものは、まるで気体のように実体がないものです。ソフトウェアという言葉もすでに実体がないものとして扱われていますが、電子的な意味ではたしかに存在しています。一方、概念はそれさえもありません。ソフトウェアよりも、さらに実体がないものだと言えます。

ワークウェア・エンジニアリングではワークウェアという概念を、気体のように実体のないものを扱うことになります。当然ながら従来の体系――たとえばソフトウェア・エンジニアリングとは異なるあり方になります。

## 仮説検証を素早く行える
サマリー:

- ワークウェアは概念という仮説
- 概念を単に扱うことでも検証できる
- 従来の検証ほど厳密ではないが、意味はある

### 検証スピードという課題
現代は先の読めない VUCA の時代であり、仮説検証はますます重要になってきています。

皆さんは仮説検証と聞いて何を思い浮かべるでしょうか。何らかの実験または製造が伴う、と捉えているかと思います。たとえば机上の議論は仮説検証ではありません。机上の空論という言葉もあります。そもそも検証という言葉が「証明」や「証拠集め」の意ですから、当然と言えば当然です。

一方で、仮説検証にはそれなりのコストが伴います。検証を行うのがそれなりに大変だからです。たとえばプロトタイプなどソフトウェアをつくって使ってもらう必要があります。このコストを減らす取り組みは多数存在しますが、それでも「検証」の重さからは逃れられていません。

### 権威
では、この重さから逃れて、もっと検証のスピードを早めるにはどうすればいいでしょうか。

**概念のまま扱ってみる** ことです。言及して議論したり、自分なりに想像やシミュレーションをしてみたり、あるいはメソッドやプロセスであるならば自分で試してみることもできますし、原理原則や格言であるならば心がけることができます。当たり前のことを言っているように聞こえるかもしれませんが、これが難しいのです。Googleが「[心理的安全性](https://rework.withgoogle.com/jp/guides/understanding-team-effectiveness#identify-dynamics-of-effective-teams)という概念が有用です」というのと、見ず知らずの個人、たとえば筆者が「[精神的安全性](https://scrapbox.io/workware/%E7%B2%BE%E7%A5%9E%E7%9A%84%E5%AE%89%E5%85%A8%E6%80%A7)という概念が有用です」というのとでは全然違います。前者は扱ってみようと思えますが、後者はそうは思えないのではないでしょうか。実際、後者は前者ほど検証されてきたものではありません（というより全く検証していません）。「使ってほしいならならまずは検証して正しさを保証しろ」が本音ではないでしょうか。

これをワークウェア・エンジニアリングでは **権威(Authority)** と呼びます。知識として定着したとか、学問的に証明されたとか、市場に受け入れられたとか、有名人が支持しているとか、そういったものです。そういう意味で、私達は権威にとらわれているとも言えます。仮説検証もその例外ではなく、仮説検証とは事実上「権威性を保証するための作業」ということができます。保証するためには「確かさ」や「数の力」が必要ですから、実験のように論理的かつ実証的にアプローチしたり、ソフトウェアのように道具として使ってもらえるものを用意したりするわけです。

### カジュアルな検証
話を戻しましょう。検証のスピードを早めたいのでした。権威の概念を持ち出すと、一つの光明が見えてきます。要は「権威性を保証する」のが重たいのですから、保証しなければいいのです。**権威性を保証しない形の検証**、というものを考えればよいです。仮にそのようなやり方があるとしたら、実験やソフトウェアも要らない――それらを行ったりつくったりしなくてもいいかもしれません。

あります。それがワークウェア(※1)です。ワークウェアという概念的道具を、自分なりに扱うのです。実験で証明されてないから、あるいはソフトウェアの形で使えるようになってないから話にならないではなく、自分なりに扱ってみるのです。自分一人でもできますし、もちろん色んな人にそうしてもらって、そのフィードバックを集めて、議論もしていった方がなお良いのは言うまでもありません。

ワークウェア・エンジニアリングは、このようにワークウェア（概念的道具）を自分なりに扱うという形の検証を重視します。これを **カジュアルな検証** と呼びます。また、そのためには「権威」へのとらわれを緩める必要もあります。小難しく言うと、別のメンタルモデルが必要です。ワークウェア・エンジニアリングでは **権威からの解放(FFA, Freedom From Authority)** と呼んでいます。

- ※
    - 1 ちなみに「ワークウェアをつくるのも難しいんじゃないの？」と思われた方もいらっしゃるかもしれません。程度によりますが、ちゃんと実験したりソフトウェアをつくったりするよりはかんたんです。自分なりに概念を言語化すればいいだけだからです。もちろん、ワークウェア・エンジニアリングではワークウェアをつくるためや伝えるための方法論や原理原則も扱います。

### カジュアルな検証に意味はあるのか？
権威にとらわれていると、カジュアルな検証なるものはバカバカしく聞こえるかもしれません。

もちろん権威は大事ですし、最終的には目指すべきことです。しかし、最初から常にとらわれている必要があるかというと、イエスではないでしょう。数学など厳密な世界ではイエスと言えそうですが、それでも厳密な定理をつくる前段階でカジュアルに検証することはありえると思います。

幸いにも、皆さんにも馴染みの概念があります。「アイデア」です。権威が大事だからといって常にアイデアには一切頼らない、見向きもしないといったことはないでしょう。特に昨今は冒頭でも述べたとおり、現状を見直したり新しいことを考えたりする必要性もよくあります。そういうときは、権威のともなわないアイデアに頼ることもあります。アイデアを出して、それを解釈して、議論して、といったことは比較的誰でも行っている、あるいは行ったことがあるかと思います。

ワークウェア・エンジニアリングは、このような営みを強調します。もっと気軽に使っていいし、何なら重用してもいいと捉えます。ワークウェアというアイデアをつくって、それを自分なりに扱ってみる。そうして得た気付きを他者に伝えたり、またワークウェアをつくったりして……という風な営みです。これだけで権威に至ることはないかもしれませんが、そのための下地を整えたり、選択肢を広げたりすることはできます。ついでに言えば、このような活動は純粋に楽しいですし、コミュニケーションの一環にもなりますし、何より誰もが自発的に自分なりにワークウェアを考えるという習慣、そして風土が形成されていきます。中長期的目線になるとは思いますが、組織も強くなっていきます。

このように、カジュアルな検証を行う意味はかなりあります。

## ソロワークという視座が得られる
サマリー

- ソロワークも今後ますます重要となってくるが、あまりフォーカスされていない現状がある
- ワークウェアとソロワークは切っても切り離せない、ゆえにワークウェア・エンジニアリングによってソロワークにもフォーカスできる

### チームワークとソロワーク
ワークウェア・エンジニアリングはどちらかといえば個人的な営みです。というのも、ワークウェアは自分事化して捉えるものだからです。前項にて権威のお話をしましたが、では権威に頼らずに捉えるにはどうしたらいいでしょうか。別の言い方をすると、権威という正解に頼れない状態で、「これだ」と選ぶにはどうしたらいいでしょうか。二つあります。

- 周囲に頼る
- 自分に頼る

周囲に頼るというのは、たとえば周囲の人の意見を聞いたり、自分が属する組織の見解、あるいは関わっている立場や仕事の文脈に則ったりするということです。場や空気や他者といったものが常に存在しています。このような制約で動くことを **チームワーク** といいます。昨今では不文律を明示する取り組みもよく見られるようになりました（いわゆる「ビジョンを定める」とか「理念への共感」とかいったもの）。

もう一つ、周囲には頼らず自分に頼ることもできます。わかりにくいかもしれませんが、内省やブレインストーミングのイメージです。自分の内から考えをひねり出して、こねて、時には寝かせて、最後に選ぶところまでやります。つまりは自己完結です。既存の何かを参考にすることももちろんありますが、どう解釈してどう使うかは自分の思うがままです。乱暴に言えば「好き勝手にこねくりまわす」とか「ダシにする」とも言えるでしょう。このように自己完結的に動くことを **ソロワーク** といいます。

### ソロワークへのフォーカス
さて、チームワークとソロワーク、現状重視されているのはどちらでしょうか。

チームワークです。ソロワークは縁のない人も多いのではないでしょうか。たとえば読書感想文を書いたり、就職活動で自己分析したり、社会人としてキャリアを考えたり、あるいは結婚や資産運用や賃貸vs持ち家をどうするか決めたり、といったことが苦手な人（あるいは常識や周囲の声にとりあえず従う人）は多いと思います。

そんなソロワークですが、今後ますます重要になってくると考えます。個々人が主体的に生きること、そしてそれらが尊重される世の中になってきているからです。多様性という言葉は耳にタコができるほど聞いているかもしれません。前項でも挙げましたが VUCA の時代でもあり、正解の無いシチュエーションも増えています。このような世の中に対抗するには、多様な手段や発想を臨機応変に組み合わせて探索的に進んでいくスタイルが求められます。もっと言えば、1:個々人が自分なりに個性を発揮する → 2:個性を上手く使って高めていく（いわゆるシナジー）の2ステップが必要です。

では、Step1の発揮とはどうやって行うのでしょうか。ここで使えるのがワークウェアです。

自分の個性をワークウェアという形で表現して、周囲にお出しするのです。周囲はこれを自分なりに解釈して感想を言ったり、フィードバックを出したり、議論してもいいでしょう、もちろん実際に使ってみてからそうしてもいいですし、別のワークウェアや似たワークウェアを出しても構いません。それらの反応を受けて、自分はまたワークウェアを見直し、必要なら改良してまた出します。このように、ワークウェアをベースとした（媒介とした）コミュニケーションを行っていくわけです。これを **ワークウェア・ベースド・コミュニケーション(Workware Based Communication)** と呼びます(※1)。

- ※
    - 1 ものづくりを行う方は「何かをつくって」「それをレビュー（批評）してもらう」という形のコミュニケーションを知っているはずです。ワークウェア・ベースド・コミュニケーションも同じです。それをもっとカジュアルに行うという話です。なにせワークウェアはつくるのが（比較的）かんたんですし、カジュアルな検証をするのだ、してもいいのだという点も前項のとおりです。

### ソロワークとワークウェアの対応
ソロワークの話に戻りますが、ワークウェアを扱う（つくったり伝えたり使ったりする）にはソロワークが必要、という対応になります。逆を言えば、ワークウェアを扱うという営み――つまりワークウェア・エンジニアリングを行うことは、ソロワークを行うことに直結します。

ワークウェア・エンジニアリングによって、ソロワークの視座を手に入れ、また鍛えることができます。

ソロワークを知り、ワークウェアを扱えるようになり、ワークウェア・ベースド・コミュニケーションを行っていけば、チームワークもより洗練されるでしょう。チームワークとソロワークの両輪が大事だと筆者は考えます。

## 限界突破できる
サマリー:

- 私達は様々な限界に阻まれている
- たとえばリモートワークは場所的拘束の限界を突破し、かつてない QoL の実現に寄与した
- ワークウェアには様々な限界を突破できるポテンシャルがある

### リモートと限界突破
前項にて多様性と VUCA を挙げましたが、もう一つ、リモート（リモートワークのリモート）もよく聞くようになりました(※1)。現在は対面とのバランスを考慮したハイブリッドワークが主流ですが、リモートの価値自体はもはや疑いようがないと思います。

このリモートというあり方、なぜ良いのかを考えたことはあるでしょうか。筆者は「拘束」という言葉で捉えています。出社したり会議したりといった営みは人を場所的にも時間的にも拘束し、有限で貴重なリソース（特に時間）を著しく消耗させます。拘束されると消耗されます。

リモートの本質は拘束を軽減するところにあります。少なくとも場所的拘束は壊します。オンライン会議に頼らなければ時間的拘束も壊せます。これによりリソースに余裕が生じて QoL が向上します。加えて、業務中にも余裕が担保されれば、勉強や改善や創造といった投資活動あるいは雑談など単に満足度を上げるための活動にも時間を回せるでしょう。

これを小難しく言うと、時間や身体というリソース上の限界を、リモートという手段によって突破した、と言うことができます。このようなことを **限界突破(リミッター解除/Limiter Removal)** と呼ばせてください。

- ※
    - 1 ちなみに筆者はこれらを総称して VUCARD と呼んでいます。VUCA + Remote + Diversityです。解説すると「先が見えず正解もわからない世の中に対しては、リモートによる拘束軽減と多様性による手段の拡充が重要である」。

### 限界突破とワークウェア
限界突破は他にもあります。

たとえば **ブレインリミット(Brain Limit, 頭の処理能力の限界)** です。暗算などはわかりやすいですが、人間の脳内処理場も無限ではないため、頭だけで行うのには限界があります。これを突破するために、いわゆる「書く」ことを行いますよね。暗算に対して筆算という手法があります。小学校で習うほどよく知られたものです。

もう一つ、**ノーションリミット(Notion Limit, 観念の限界)** もあります。ワークウェアの定義の項で述べましたが、観念とは自分にしかわからないものでした。これを伝えるためには言語化を行う（概念にする）必要があるのでした。観念には他者に伝えることができないという限界があります。

あと一つだけ紹介させてください。 **フォーマルテストリミット(Formal Test Limit, フォーマルな検証の限界)** も存在します。前項でも述べたとおり、従来の検証は権威性を保証するための作業であり、それゆえに重たいものでした。たとえ重たくても、それ以外に方法が無いから行うしかありませんでした。また、この重さゆえに、私達はビジーかつ厳格になりがちで、権威性をともなわないものを否定したりスルーしたりしがちです。この傾向もフォーマルテストリミットによって引き起こされているものです。

このように私達は色んな限界に阻まれています。限界を突破することで、より前に進むことができます。

本題に入りますが、ワークウェア（というよりそれを扱う営みを表すワークウェア・エンジニアリング）は限界突破の手段として重宝します。

### ワークウェアが突破する限界
まずノーションリミットを突破できます。たとえば個人的なモヤモヤを誰にも話せず、自覚すらできないで抱え込むのではなく、紛いなりにも言語化して誰かに伝えることができます。伝えればフィードバックももらえますから、さらに改善していくことができます。ワークウェアという言語化されたものがあるので、それをいじくりまわすという形で進んでいけます。

次にブレインリミットも突破できます。ワークウェアは概念であり、名前もついたものですから、何らかのものを示しています。いちいち「私達は限界というものに阻まれていて、その一つに頭の処理能力的な限界がある」といわなくても「ブレインリミット」というだけで済むようになります。前者より後者の方が頭に優しいですよね。

加えて、（詳細は後の章で扱いますが）ワークウェア・エンジニアリングでは書くことを重視します。ワークウェアをつくる際は、頭の中だけで頑張ってひらめかせるものではなく、書いたものを見ながら発想を広げたり、それをまとめたりといった形で「書いたもの」ベースで思考していくことが多いです。このような思考方法を **第三の脳(3rd Brain, サードブレイン)** と呼ばせてください(※1)。

第三の脳はまさにブレインリミットを壊せるものです。ワークウェア・エンジニアリングにより第三の脳を使うことに慣れていきますので、これはつまりブレインリミットを壊していけるようになるとも言えます。

最後に、フォーマルテストリミットについても、もちろん打ち崩せます。「カジュアルな検証」を行えるからです。従来の権威性検証だけではなく、その前段階でカジュアルに検証を行うことで、前者のスピードも早まります。また前者だけではリーチできなかった対象も検証できるようになります。速さも広さもアップするのです。

以上、ワークウェア・エンジニアリングによる直接的な限界突破はこの3つですが、ワークウェア自体には無限の可能性があります。要は **限界突破を行うためのワークウェアをつくればいい** のです。

一つ例を挙げましょう。筆者は、口頭やチャットだと同時に一つの話題しか扱えないことを **話題の拘束** と呼んでいます。この拘束はボトルネックとなっており、扱える話題が非常に限定されてしまっています（中長期的な議論がおざなりになったり政治が発生したり）。突破するにはどうしたらいいでしょうか。話題ごとの書き込み場所を確保した上で、全員がそれらを守って書き込んでいければよいです。このようなあり方を、話題（トピック）に指向しているということで **トピック指向** と呼んでいます。では、トピック指向を実現するためには、どのようなコミュニケーションを設計すればいいでしょうか――といった風に思考を広げていくことができます。ここで「話題の拘束」と「トピック指向」は（筆者がつくった）ワークウェアです。この調子で進めていけば、本当に話題の拘束を突破する何かを生み出せるかもしれません（※2）。

- ※
    - 1 自分なりの「書く場所」を整備する営みには色んな人が色んな名前をつけています。たとえば[独学大全](ref#4)では「外部足場」と呼んでいます。しかし、第三の脳はこれらと同義ではありません。
        - 第三の脳は、書く場所というよりも、書いたものをメンテナンスしながら思考していく営みを指します。単に書いて残すのではなく、その書いたものを肉付けしていくのです。
        - 文章や絵やプログラミングなどものづくりをされる方は（いまさら名前をつけるまでもなく）馴染みある考え方だと思います。まあ才能がある方は「頭の中でつくって」「手を動かす時はそれを出力するだけ」のストロングスタイルだったりしますが……。
        - あるいは、ブレインストーミングなどアイデア出しをされる方は「観点を見ながら何かひらめかせる」ことをやられているはずです。
        - どちらも第三の脳的だといえます。ワークウェア・エンジニアリングでは、この第三の脳という営みを重視するため、（主にものづくりやアイデア出しに無縁な方のために）こうして説明をしているのです。
    - 2 ちなみに筆者は真剣です。少し紹介すると「ビジネスウィキ」という概念を確立しようとしています。LINE のようなただのチャットに対して、Slack や Teams のような「ビジネスチャット」がありますが、ウィキにも同じ進化が可能だと思うのです。現状最も近いツールは [Scrapbox](https://scrapbox.io/) ですが、まだ足りません（そもそも Scrapbox の設計思想でもないでしょう）。筆者の探求は続きます……。

# まとめ
- ワークウェアとは
    - 仕事で使う概念的な道具（やり方・あり方・考え方・捉え方など）
    - 成立の3条件は「名前がついていること」「観念ではなく概念であること」「実用的であること」
    - 言語化され、名前もついていて、使う（言及する・生かす・試すなど）ことができるもの
- ワークウェア・エンジニアリング（WWE）とは
    - ワークウェアを扱うための方法論
    - 扱うとはつくる、いじる、使う、探すなどすべてを含む
    - 専門家だけではなく、誰でもワークウェアを扱えるようにするための工学
- ワークウェアの立ち位置
    - 第四の潮流（知的生産の技術 → 仕事術・自己啓発 → ライフハック → ワークウェア）
    - ハードウェアでもソフトウェアでもないもの（物質的なハードウェア、電子的なソフトウェア、概念的なワークウェア）
    - 発明でも芸術でも哲学でもないもの（実用的だが実物の無い世界）
- ワークウェアの価値
    - 仮説検証を素早く行えること（カジュアルな検証、権威からの解放）
    - ソロワークの視座（チームワークとソロワークの両輪、ソロワークへのフォーカス）
    - 限界突破（頭の処理能力、観念、フォーカルな検証といった限界の突破。またその他の限界を突破するワークウェアもつくれる）

# === Chapter-3 ワークウェアを眺める ===

本章ではワークウェアの分類をいくつか述べつつ、各分類項目の性質やそこに含まれるワークウェアの例を挙げていきます。

# ワークウェアの利用レベル

## 利用レベル
ワークウェアは「どのように扱うものか」という切り口で分けることができます。

| レベル | 分類名 | 動詞で | 必要な能力 |
| ---- | ---- | ---- | ---- |
| 1 | Name      | 言及する | 語彙力、教養、連想などが必要 |
| 2 | Principle | 心がける | 応用が必要 |
| 3 | Protocol  | まわす   | 能動的使用が必要 |
| 4 | Tool      | 想像する | 想像や思考実験が必要 |

## 各レベルの概要
レベル1、Name は **言及するワークウェア** です。現象、傾向、性質、特徴、心理といったものがこれにあたります。あるいは別の言い方をすると命名、あだ名、たとえなども当てはまります。名付けることで（その名前が指す概念の）理解が容易になったり、言及してコミュニケーションを行ったりできます。

レベル2、Principle は **心がけるワークウェア** です。原理、原則、格言、モットー、心構えといったものがこれにあたります。道具は使うものですが、心がけるワークウェアを使うのはかんたんではありません。というのも、然るべきときに忘れず思い浮かべて、かつ何らかの適切な行動を（たいていはその場ですぐに）起こさなければならないからです。このような行いをワークウェア・エンジニアリングでは **応用** と呼びます。

レベル3、Protocol は **まわす（回す）ワークウェア** です。プロセス、メソッド、手順、プラクティス、ルールなどが当てはまります。まわすワークウェアを使うためには、それについて理解した上で、自らそのとおりに頭手足を動かす必要があります。また、どれだけ厳密に従うかはワークウェアと状況次第です。たとえば手順には厳密なものもあればいいかげんなものもありますし、厳密な手順であっても状況が許せばサボったり端折ったりすることもあるでしょう。いずれにせよ、基本的には面倒くさい営みになります。このような行いを、能動的に行動して使っていくということで **能動的使用** と呼びます。

レベル4、Tool は **想像するワークウェア** です。ソフトウェア、装置、器具などの「道具」としてつくれそうなレベルで言語化が行われたものになります。企画やアイデアほど雑ではありませんし、設計ほど詳細で緻密なものではありません。感覚的に言えば両者を足して割ったようなものです。素人でも想像ができ、しかし専門家が生かせるというバランスです。このようなワークウェアを扱うには、想像を働かせたり、思考実験を行ったりします。実際につくるということはしません（それはただのハードウェアやソフトウェアであり、ワークウェア・エンジニアリングの範囲を超えます）。

## Lv1 言及するワークウェア
レベル1、言及するワークウェアとして、たとえば以下があります。

- 蛙化現象
    - [Z世代が選ぶ流行語ランキング](https://www.tokyo-np.co.jp/article/264372) にも入った言葉で、好意を抱いている相手への熱が急速に冷めること
    - 元ネタはグリム童話の「かえるの王さま」とされる（人間になったりカエルになったりする王子への姫さまの態度）
    - 「名前のついていない」あるある現象に、既存作品の作品名や人物名などを充てる形のネーミングです
- 礼儀1.0
    - 自分を犠牲にして相手のために時間を使うというメンタルモデル
    - 元ネタは[GOROmanさんのツイート](https://twitter.com/GOROman/status/984872963655680000)と思われる
    - 版管理における版の付け方、特に性質がガラリと一新されるメジャーなアップデートは1.x → 2.x と一の位を更新していきますが、ここから取ってきていると思われます（Web2.0などこの手のネーミングもよく見かけます）
- [コミュニケーション1.0](https://scrapbox.io/workware/%E3%82%B3%E3%83%9F%E3%83%A5%E3%83%8B%E3%82%B1%E3%83%BC%E3%82%B7%E3%83%A7%E3%83%B31.0)
    - 筆者のワークウェアで、常に特定の相手を想定するというコミュニケーションのメンタルモデル
    - これも上記と同じネーミングを使って名付けました

### あるある性
例を見てもわかるとおり、言及するワークウェアはたいてい「ある程度の人が薄々勘付いていること」が対象になります。もっと言えばあるある～と共感できるほど「よく知られていて」、かつ知られているということは「わかりやすい」ものであるとも言えます。このような性質を **あるある性** と呼ばせてください。

あるある性の高いものは言及する機会が多いです、なのに名前がついてないので、じゃあつけようというわけですね。SNS やメディアから身内まで、この手の営みは至るところで発生します。きわめて自然で身近な営みです。特に現代は[「最近クラスで何が流行ってる？」→「無い」と返ってくる](https://togetter.com/li/2229386)ほど多様性の時代でもあり、SNS など発信手段も豊富であるため、各集団各界隈の「あるある」が発生しやすく、発信されやすいと思います。別の言い方をすると、「共感」というワードもよく目にすると思います。あるあるの時代、と言っても大げさではないでしょう。

ワークウェア・エンジニアリングは、この自然な営みを（言及するワークウェアという形で）もっと積極的に使っていくことを勧めるものだ、とも言えます。あるあるの時代だからこそ、需要もあります。

## Lv2 心がけるワークウェア
次にレベル2、心がけるワークウェアの例をいくつか挙げます。

- 早寝早起き
    - 早く寝ることと、早く起きること（特に「深夜まで起きない」「早朝に起きる」「この二つを両立できる睡眠時間」のニュアンスがあると思います）
    - 健康的な生活をおくる原則として言わずと知れた言葉ですが、これを「使う」のは難しいでしょう
    - 早く寝るためには？早く起きるためには？自分の体質は？など、考えること・やることはたくさんあります
- VUCA
    - 先の読めない世の中を表現した言葉で、Volatility（変動性）、Uncertainty（不確実性）、Complexity（複雑性）、Ambiguity（曖昧性）の略
    - これも同樣に「使う」のは難しいと思います
    - 先が読めないから何？どうすればいいの？組織はともかく個人にできることなんてあるの？など疑問は尽きません
- 返報性の原理
    - 施しを受けた際にお返しをしなければならないと感じる心理
    - これも「使う」となるとピンと来ませんが、参考にすることはできそうです
    - 試食（タダで食べちゃったし、ちょっとなら買おうか……）やドア・イン・ザ・フェイス（最初断っちゃったからその程度ならまあ契約してもいいかな……）など商業的にはよく使われます

### 言及するワークウェアと心がけるワークウェアの違い
ここで「レベル1の『言及するワークウェア』と何が違うの？」と思われた方がいらっしゃるでしょう。たとえば VUCA は、先の読めない世の中になってきているな～、との肌感覚は言われるまでもなく多くの人が認識していることであり、それを誰かが名付けて広めた（あるいは広まった）だけだからレベル1ではないか、とも言えそうです(※1)。

そのとおりで、**言及するワークウェアと心がけるワークウェアに本質的な差はありません**。基本的に言及できるものは何かしら心がけることができますし、そもそも心がけるものには名前がついており言及できることが多いです（名前をつけずに内々で曖昧に捉えているもの、たとえば感覚や直感といったものもありますが、これらはワークウェアではありません）。

明確な違いはありませんが「あるある性の高さ」と「心がけやすさ」から判断できます。

**言及するワークウェアはあるある性が高く、かつ心がけにくいもの** です。先でも挙げた「蛙化現象」は、そういう性質があるんだ、ふーんとはなりますが、心がけて普段から生かしていくというのは難しいと思います。あるいは恋を用いた対人的ビジネスをされている方なら生かせるかもしれません（熱心なお客様の目を覚ませるため、あるいは人懐っこく惚れやすい後輩や新人に一線を越せさせないために、あえて蛙化現象が起こりやすい行為を行うなど）が、例外的です。

一方、心がけるワークウェアは、まず心がけやすさがあります。たとえば返報性の原理は、ビジネスの文脈では当たり前に使われており、Wikipedia を見ても試食やドア・イン・ザ・フェイスといった具体例があります。何らかの研究や実践に熱心な方が頑張って確立して、名前もつけて広めた（あるいは広まった）というケースが多いです。要はしっかりとした下地がつくられており、心がけるためにはどうすればいいかの例も整備されていることが多いのです。ただし内容が専門的である場合も多いため、あるある性があるとは限りません（その専門の人達に限定すれば高いですが）。もう一つ、早寝早起きのように、歴史的に昔から知られている原則や格言、もっと言えばことわざなどは典型的な心がけるワークウェアです。歴史の重みがあります。

- ※
    - 1 VUCAの起源は1990年代のアメリカ陸軍大学校だといわれています。それがいつ、誰によってビジネスの文脈にもたらされたかは不明ですが、[マーケの強化書編集部調べ](https://genesiscom.jp/kwd-vuca/)では、2013年日本能率協会マネジメントセンターから出版されたボブ・ヨハンセン著『未来を創るリーダー10のスキル 不確実性の時代を生き抜く新たな人材の条件』の帯に記載されていたのがはじまりのようです。さて、VUCA がレベル1とレベル2のどちらであるかですが、これを信じるならレベル1かなという印象です。2013年時点では、まだ大衆が「あるある～」と言えるほどの肌感覚は無かったと思います。

## Lv3 まわすワークウェア
レベル3、まわすワークウェアの例を挙げます。

- TODOリスト
    - TODO（やること）を一行一つで並べ、終わったものを終了状態にする（チェックつける・取り消し線を引く等）というタスク管理手法です
    - ただ言及するだけでなく、心がけるだけでもなくて手順が盛り込まれています
- 時間管理マトリクス（アイゼンハワーマトリクス）
    - 書籍『7つの習慣』でも取り上げられたタスクの優先順位付け手法であり、タスクを「緊急かつ重要」「緊急だが重要じゃない」「緊急じゃないが重要」「緊急でも重要でもない」の 4 つに分けた上で各々の対応に則りましょうというものです
    - これもそれなりの手続きが整備されており、行動に起こすことができます（まわせます）
- アイビー・リー・メソッド
    - 今日の終わりに「明日やらないといけないタスクを6つ」書き出し、優先順位順に並び替え、明日はそのとおりに上から一つずつ消化し、日の終わりにはまだ6個まで補充して――のような厳格な順序的 6 スロットベースのタスク管理手法です
    - TODOリストなどと比べると「このとおりにやれ」とのルールが強いです
    - ルールはサボることもできますが、サボると期待される効果が出ません（たとえば取捨選択が進まず優柔不断を脱せないかもしれません）
- [TALKの原則](https://www.mext.go.jp/component/b_menu/shingi/toushin/__icsFiles/afieldfile/2009/04/13/1259190_12.pdf)
    - ショッキングな話題ですがお許しください
    - 自殺予防に関する原則のことで Tell、Ask、Listen、Keep safe の略です
    - 名前は「原則」とあるのでレベル2、心がけるワークウェアかと思いがちですが、内容としては『Tell：言葉に出して心配していることを伝える』、『Ask：「死にたい」という気持ちについて、率直に尋ねる』など手続きが含まれているため、まわすワークウェアと言えます

### 「まわす」とはどういうことか
まわすワークウェアと呼んでいますが、では「まわす」とはどういう意味でしょうか。

これはプロセスをまわすとか、サイクルをまわすといったニュアンスと同じです。有名なワークウェアを挙げると、PDCAサイクル（をまわす）は誰もが一度は聞いたことがあるのでないでしょうか。

改めて説明すると、「まわす」とは単に一回だけ試しに使ってみることから、継続的に使い続けることも含みます。実際にまわし続けるかどうかはさておき、少なくともまわせるポテンシャルはあります。

いずれにせよ、「そのワークウェアが言っているとおりに」動くものです。単に言及するだけ、あるいは心がけるだけとは違います。私達はワークウェアが言っていることに従い、自ら頭や手足を動かします。手作業とか手動とかいった言い方をすることもあります。どうやればいいかはわかるが、実際に動くのは私達自身というわけです。これは面倒くさい営みであり、基本的には能動的な行動が必要です。能動的使用と呼ぶのでしたね。

このような営みを表現するために「まわす」という言葉を使ってみました。「使う」の方がわかりやすいですが、これは次のレベル4で使いたいので採用していません。

## Lv4 想像するワークウェア
レベル4、想像するワークウェアの例を挙げます。既存の例を探すのは困難なので、筆者によるものを挙げさせていただきます。

- [Facilitation As Code](https://scrapbox.io/workware/Facilitation_As_Code)
    - 会議のファシリテーション（進め方とフォロー）をコードで表現し、実際のファシリテーションもプログラムに任せるというあり方です
    - これにより、以下のメリットが生まれます
        - 会議の進め方・フォローの仕方という暗黙知を形式知化できる
        - 有益なファシリテーションを共有できる
        - ファシリテーションの修正＝コードの修正、となるため、ファシリテーションを形式的に捉える文化が醸成される → ファシリテーションの質が上がる（なあなあやテキトーができない）
        - ファシリテーターという役目がなくなり、その人も議論にさんかできる
- [Potluck SNS](https://scrapbox.io/workware/Potluck_SNS)
    - ブックマークベースドコミュニケーションを行うオープンな、あるいはクローズドなSNSです
    - スペースと呼ばれる空間にメンバーが入っており、ここにメンバー達がブックマークを流します
        - 皆でブックマークを共有したり、それに反応したりする世界観です
        - スタンプは推せますが、テキストは一切書けません
    - スペースは会社単位でクローズドに作ってもいいですし、趣味や業界などオープンなコミュニティとしてつくることもできます
    - ブックマークはまず誰にも見えない個人領域に保存されます、これを自分が属するスペースに流します
        - どのスペースに何のブクマを流すかは選べます
        - 面倒な方向けに自動で流すこともできます
- [カジュアリマインド](https://scrapbox.io/workware/%E3%82%AB%E3%82%B8%E3%83%A5%E3%82%A2%E3%83%AA%E3%83%9E%E3%82%A4%E3%83%B3%E3%83%89)
    - 1日10～20回以上仕込むことを前提としたリマインダーです
        - カジュアルにリマインドすることからこのように名付けています
    - タスクリストでもカレンダーでも捉えられない、微妙に時間帯がつきまとうタスク（スティッキータスク）を扱います
        - 例: 朝のうちにやるべきタスク、いつやってもいいけど今日は昼くらいにやりたいタスク etc
    - 最大の特徴はリマインド後の先送りがかんたんに行えることです
        - たとえば「2分後にまたリマインドして」「20分後にお願い」「200分後によろしく」の選択肢から選ぶことができます（2-20-200）
        - もちろん「今日はもうなくていいや」のようなキャンセルもできます
        - 「大体これくらい後にやりたいなー」という直感的な先送りを反映できるということです

いかがでしょうか。専門家でない方でも「面白そう」「良さそう」と感じられたのではないでしょうか。かつ、専門家の方（たとえばエンジニアの方）は、大体どうやってつくればいいかのイメージが湧いたのではないでしょうか。そうでないとしたらごめんなさい。筆者の表現力が未熟ゆえのことです。

ちなみに、ちゃんと読んでない方は、一度ちゃんと読んで、想像なり思考実験なり頭を働かせてみることをおすすめします。想像するワークウェアはまさにそうやって使うものだからです。文章という味気ない情報（たいていは情報密度も高い）を理解し、想像を働かせる必要があります。10行の箇条書きに10分以上、あるいはそれ以上かけることもありえます。ただ、それでもわからなかったとか、別に良さそうには感じなかったとか、そもそも読む気起きなかった等あるのでしたらごめんなさい。

### 素人が想像できて専門家が実装できる、というバランス
想像するワークウェアには（それなりの前提知識を持つ）素人が想像できることと、実装（開発など形にすること）を行える専門家が実装できることの両輪が求められます。これを **IAIP(Imaginable by Amateur, Implementable by Professional)** と呼びます。

素人に寄りすぎると、ただの妄想となってしまい実現性が乏しいです。逆に設計書など専門家に寄りすぎると素人に理解できませんし、文章量も増えたり論理性や厳密性が重視されたりして「良さそうな概念にありつく」「カジュアルに検証する」という本質から遠のきます。

### バーバルプロトタイプ
想像するワークウェアはプロトタイプの一種とみなせます。しかし従来のものとは違い、具体的なソフトウェアをつくったり、綿密なドキュメントをつくったりはしませんし、デザインプロトタイプやファンクショナルプロトタイプやコンテクスチュアルプロトタイプと呼ばれる「ユーザー向けのプロトタイプ」とも違います。MVP のようにお金をもらうことを目指すわけでもありませんし、SFプロトタイピングのように物語と空想に全面的に振っているわけでもありません。

想像するワークウェアもワークウェアであり、言語によって表現されたものにすぎません。そういう意味では言語的なプロトタイプとでも言えます。ワークウェア・エンジニアリングでは **バーバルプロトタイプ(Verbal Prototype)** と呼ぶこともあります。「想像するワークウェア」と同義ですが、プロトタイプ（試してもらう）というニュアンスを出したい場合に「バーバルプロトタイプ」を使うとわかりやすいでしょう。

さて、このバーバルプロトタイプですが、言語的なプロトタイプというだけではほとんど情報量がありませんので、いくつかニュアンスを挙げておきます。

- IAIP の担保
- 面白さ
    - 目を引く名前
    - 既存の概念を少しアレンジしました、という見せ方
    - わくわくする世界観や未来像の描写 etc
- 情報を並べる意識
    - 箇条書きで表現するのが適しているというイメージ
- 絵や図や表よりも言語
- 形式は自由だが、読みやすさは大事

# ワークウェアの権威レベル

## 権威レベル
前章にて「権威」という言葉を取り上げましたが、ワークウェアはこの権威の程度（権威性の程度）でも分類できます。

| レベル | 分類名 | 権威性の感触 | 解説 |
| ---- | ---- | ---- | ---- |
| 1 | ソロ | 無い | 自分が扱っているだけ |
| 2 | ファミリー | あるかも | 一人以上に通じて扱ってもらえた |
| 3 | クラスター | 少しはある | 特定の集団単位で当たり前に扱われている |
| 4 | コミュニティ | それなりに高い | 権威の発露。特定の界隈では当たり前に使われている、経営者やアーリーアダプターその他意識の高い人は知っている |
| 5 | スタンダード | かなり高い | 権威の確立。標準と呼ばれるレベルで浸透している、ビジネスの文脈では一般的に扱われている |

## 各レベルの概要
レベル1、ソロはワークウェアを自分一人で扱っている段階です。単につくっただけなのもそうですし、自分のために自分でつくって使い続けているのもそうです。自分にしか通じないので、 **権威性は無いに等しい** でしょう。しかし、ワークウェアではあるので、他者に伝えることはできます。

レベル2、ファミリーは他者一人以上に通じて扱ってもらえている段階です。一人でも数人でも十数人でも構いませんが、とにかく自分以外の他者に通じたこと、そして何らかの行動をしてもらえた（一度でも使ってもらえた、ノートにメモされた、他の人と話すときに取り上げたもらった、本人に何らかのフィードバックをくれた等）が必要です。通じただけでは意味がないことに注意してください。このレベル2でもまだまだ権威は無いに等しいですが、他者に通じたことで **ありそう（権威性を上げられそう）** という感触は持てます。

レベル3、クラスターはチームやプロジェクトといった集団単位で当たり前に使われている段階です。チーム独自の取り組みは世の中、数え切れないほどあると思いますが、そういったものはレベル3のワークウェアである事が多いです。実際、メンバーに伝えるためにある程度の言語化も行われているからです。一方で、言語化が行われず OJT や実践を通じて各自が悟っていく、あるいは出来る人を見て盗んでいくといった形での洗練・継承もありふれており、こちらはワークウェアとは言えません。このようなものを言語化して捉えられると、一気にレベル3のワークウェアとなります。いずれにせよ、レベル3のワークウェアは、特定の集団では通用しているため **権威性は少しはある** と言えます。

レベル4、コミュニティは界隈の単位で当たり前に使われている、ないしは知られている段階です。大半のワークウェアはここまでのボリュームしかありません。具体的には「学術的にはしっかり証明されたが実践的にはさほど価値がない」とか「実用的であると十分知れ渡ったがニッチすぎて活用範囲が限られている」などです。ここまでくると、そのワークウェアでご飯を食べることができることもあります。また、界隈以外の、意識の高い人に伝わることもあります。汎用性もそれなりにあるため、複数の界隈に刺さるポテンシャルもあります。 **権威性はそれなりに高い** と言えるでしょう。

レベル5、スタンダードは標準と呼ばれるレベルで浸透している段階です。といってもビジネスの文脈で、と補足しておきます。この文脈とは離れた存在――たとえば子供や学生を含む被扶養者、自営業、ボランティア、その他特殊な仕事や立場の人などへの浸透は想定していません。このワークウェアにもいくつか種類がありますが、少なくとも知名度の高さを備えています。逆を言えば、有名人の言葉がバズったり、何らかの意図や政治等によりメディアによってブームに仕立て上げられたりしただけでもここに至ることがあります。 **権威性はかなり高い** と言えますが、その中身には注意が必要です。知名度に振り切っていて実用性や学術性が乏しいこともあります。

## Lv1 ソロ・ワークウェア
自分だけが扱っている段階のワークウェアです。筆者がつくったものをいくつか挙げます(※1)。

- [タスク管理の3Pモデル](https://scrapbox.io/workware/%E3%82%BF%E3%82%B9%E3%82%AF%E7%AE%A1%E7%90%86%E3%81%AE3P%E3%83%A2%E3%83%87%E3%83%AB)
    - タスク管理には主に Project、Partner、Personal の 3 種類が存在すると述べたもの
    - それぞれ性質が全く異なっており、人が「タスク管理」という言葉を使っているときは、どれを指しているかを確認する必要があります
- [レイジーオープン](https://scrapbox.io/workware/%E3%83%AC%E3%82%A4%E3%82%B8%E3%83%BC%E3%82%AA%E3%83%BC%E3%83%97%E3%83%B3%E3%81%A8%E3%83%AC%E3%82%A4%E3%82%B8%E3%83%BC%E3%83%95%E3%83%A9%E3%83%83%E3%83%88)

    - いわゆる「オープンな情報共有」は、レイジーオープンでは不十分です

    - 既存の駆動型ワークウェア（●●駆動のこと）を探したり使い分けたりするのはもちろん、新たなものをつくることもできます
    - たとえば筆者は[サマリー駆動](https://scrapbox.io/workware/%E3%82%B5%E3%83%9E%E3%83%AA%E3%83%BC%E9%A7%86%E5%8B%95)という要所要所で要約をつくりながら検討を重ねていくスタイルをつくりましたが、このようなこともドリブンドリブンの視座を持っているからこそできることです

他にも色々なワークウェアをつくっており、[こちらで公開しています](https://scrapbox.io/workware/)のでよろしければご覧ください。

### ソロ・ワークウェアは他者に通じる必要があるか
ひとまずイエスと答えておきます。

まずワークウェアの成立条件として Conceptual（観念ではなく概念であること） がありました。なんとなくの感覚とかノリとか勘とかいったものはワークウェアではありません。言語化されている必要がありました。

では、その言語化されたものが他者に実際に通じるかというと、イエスとは限りません。言語化が単にわかりづらいかもしれませんし、そのワークウェアが難しすぎて大多数の人には理解できないかもしれません。もちろん「時間をかけたら理解できるが時間をかける気がない」もありえます。通じた場合は次のレベル2、ファミリー・ワークウェアであると言えるでしょうし、逆に通じなければ（あるいはまだその機会まで行ってなければ）ソロのままです。

ちなみに上述した筆者のソロ・ワークウェアの例は、どれもまともな機会を経たわけではないのでソロとしています。が、おそらく全く通じないことはないと思っています（そもそも「言及するワークウェア」はネーミングがある程度良ければ通じます）し、レベル2のファミリーはもちろん、その先の段階に進めるポテンシャルもあると（個人的には）思っています。

### 未来の自分という他人
ソロ・ワークウェアかどうかの一つの目安は、「未来の自分」に通じるかどうかです。

今日言語化しておいたものが明日の自分にも通じるか。あるいは1週間後、1ヶ月後、1年後の自分にも通じるか。「何言ってるかよくわからん」「一昨日の自分、これ何が言いたんだ？」などは論外として、できればいつ見ても同じ解釈になることが望ましいです。あるいは、解釈が変わる場合でも、その変化後のものを言語化して捉え直せれば問題ありません。

このような営みは、言い方を変えると「未来の自分」を一人の他人として扱い、その人には通じるようにする、と言えます。これをワークウェア・エンジニアリングでは **未来の自分という他人** と呼びます。

人によっては苦しく聞こえるかもしれませんが、言語化とはそういうことです。特に一度言語化して終わりではなく、「未来の自分」という他人からのフィードバックを受けつつ微修正、時には大胆に軌道修正することもよくあります。ものづくりをされている方には身近な世界観かもしれませんが、そうではない方にとっても実は結構身近なことです。自分の考えや人に対する印象などは、比較的後から変わりやすいのではないでしょうか。そうはいっても、しばらくすると収束してくるはずです（中には一向に定まらず延々悩むケースもあるでしょうが）。これはソロ・ワークウェアについても同じことです。

## Lv2 ファミリー・ワークウェア
一人以上の他者に通じた、かつ扱ってもらえた段階のワークウェアです。いくつか例を挙げますが、これらもソロ・ワークウェアが基点ですので、筆者のワークウェアを挙げさせていただきます。

- [フリー1on1](https://scrapbox.io/workware/%E3%83%95%E3%83%AA%E3%83%BC1on1)
    - 社員はいつでも誰でも誰とでも 1on1 を設定できる、というもの
    - [GitLab 社の Coffee Chat](https://handbook.gitlab.com/handbook/company/culture/all-remote/informal-communication/#coffee-chats) など事例はありますが、フリー 1on1 はこのような取り組みを一般化したものです
    - 1on1 の取り組みは通常トップダウンまたは合意形成の形で「毎月やりなさい」「週一でやりましょう」などと決めますが、そうではなく、社員各々に完全に任せるということです
- [コミュニケーションスタッキング](https://scrapbox.io/workware/%E3%82%B3%E3%83%9F%E3%83%A5%E3%83%8B%E3%82%B1%E3%83%BC%E3%82%B7%E3%83%A7%E3%83%B3%E3%82%B9%E3%82%BF%E3%83%83%E3%82%AD%E3%83%B3%E3%82%B0)
    - 雑談や「とりあえず同じ場所で過ごす」など、親近感や信頼関係をつくるためだけのコミュニケーションのこと、あるいはそのような営み
    - 人は社会的動物であり、このような営みは必要ですが、一方でこれを安易に仕事に（特に会議に）混ぜてしまっていることがよくあります
        - そのせいで会議はいたずらに時間が長くなりがちですし、効率化も進みづらいです
        - このような現象を述べるために、筆者が名前をつけて捉えたのです。コミュニケーションを積む(Stack)、というネーミングですね
    - しかし「グルーミング」という既存の便利な言葉があるため、現在筆者はグルーミングを使っています
        - 性犯罪としての用法も強い点が微妙で、もう少し良い言葉が欲しいところではあります……
- [公然の多様性](https://scrapbox.io/workware/%E5%85%AC%E7%84%B6%E3%81%AE%E5%A4%9A%E6%A7%98%E6%80%A7)
    - 身体的障害、育児介護、LGBTQや外国人など「典型的な」多様性を指す言葉
    - 多様性の概念は今や当たり前に言及・議論されますが、その内実として扱われているものは非常に限定されており、その範囲を指すものとして定義しました
        - たとえば「人と対面で話すことが大の苦手」「ASDやADHDなどの発達障害」「独身」といったものは多様性としては扱われず、配慮されないことが多いでしょう
        - 別の例として、「ペットが亡くなったので数日喪に服します」「会社は関東ですが私は関東の気候、特に花粉が合わないので関東の外で勤務したいです」「私はHSPであり密度の高いオフィスは騒がしすぎて無理なので原則リモートがいいです、あるいはフリーアドレスではなく遮音性の高い個室がほしいです」などはいかがでしょうか
    - 似た試みとして、[サイボウズでは「目に見える多様性」と表現されています](https://cybozushiki.cybozu.co.jp/articles/m005986.html)

### 行動実績
ファミリー・ワークウェアは単に通じただけでは意味がありません。メモしてもらえたとかフィードバックを一言もらったなど、軽いものでもいいので、（相手が）何らかの行動を起こしている必要があります。

新規事業時によく登場する[リーンスタートアップの MVP](ref#12) の概念もそうですが、「理解しました」「良さそうですね」といった感想に価値はありません。実際にお金を出してもらえたかどうかがすべてです。ワークウェアの場合は「何らかの行動をしてもらえたかどうか」です。もっとも実際に他者の行動を観測できるとは限りませんが、行動まで必要だという点は意識してください。

これを **行動実績（の解除）** と呼びます。ファミリー・ワークウェアとは「行動実績が解除されたソロ・ワークウェア」であるとも言えます。

ここで「じゃあお金を払って見てもらうのはどうか」「極論脅して言わせたのはどうなのか」と思われた方もいらっしゃるかもしれませんが、それらは解除とは言えません。解除というのは、たとえば会話の流れで自然に出したワークウェアに対してその場で議論が進んだとか、あるいは社内でも社外でもいいので投稿などの形で公開しておいたものに反応がついた、といったことです。つまり相手が自発的に行動したことが目安となります。

## Lv3 クラスター・ワークウェア
特定の集団内で当たり前に扱われている段階のワークウェアです。集団というのは一つのチームやプロジェクトでも構いませんし、大きいものだと会社単位もありえます。

1on1に関するワークウェアを例に、いくつか挙げましょう。

- [ザツダン](https://teamwork.cybozu.co.jp/blog/1on1-wada.html)
    - サイボウズ株式会社によるものです
    - 特徴は「面談ではなく雑談」「定期的に行う」「部下ファースト」
    - 何を話すかは部下が決めていいですし、話したくないならスキップしても良いそうです
- [Coffee Chat](https://handbook.gitlab.com/handbook/company/culture/all-remote/informal-communication/#coffee-chats)
    - GitLab Inc. によるものです
    - 特徴は「誰でも誰にでも自由に申し込める」「日常会話から仕事の会話につながる」「普段つるまない人と喋るのも推奨」
    - サイボウズとは違って定期性は定めていません、またその名のとおりコーヒー休憩中のおしゃべりのつもりでといったニュアンスがあります
- [交換日記式1on1](https://note.com/dyokoyamas/n/n5110bad5b468)
    - 株式会社LayerXによるものです
        - 会社全体というよりは一部で使われている程度なので、ファミリー寄りのクラスター・ワークウェアとでも言えるでしょうか
    - 特徴は「事前に話題を洗い出・何ならやり取りもする」「1on1当日は余白持って望める」「情報密度とゆとり」
    - 事前に非同期的にやりとりをする点が新鮮でしょう

見ての通り、その組織のカラーが反映されていることが多いです。あるいは、反映される前段階では「組織内である程度使われている」こともよくあります。LayerXの交換日記式1on1はそうでしょう。この場合、さらに浸透が進めば組織全体にまで広がります（正真正銘のレベル3になる）、もちろん広がりきるとも限りません。いずれにせよ、ファミリー・ワークウェアよりは利用者の広がりがあります。

### 権威ボリューム
ワークウェアもレベル3、クラスターにまで来ると「もっと利用者が広がるのではないか」といった期待がチラつくようになります。しかし現実は無慈悲で、おおよそどこまで広がるかの可能性はワークウェアごとに決まっています。これを **権威ボリューム** と呼びます。

前述した 1on1 に関するワークウェアは、コミュニケーションという一大ジャンルの取り組みですのでボリュームは大きめです。一方で、LayerXの交換日記式1on1は「事前に書く必要がある」「ある程度継続的に各自がやりとりを行う必要がある（非同期性）」など高度なリテラシーが求められるため、ザツダンやCoffee Chatと比べるとボリュームは下がります。筆者の肌感覚では、仮に数百人以上の会社で全社的に採用して、マネージャーが部下全員にフォローを入れるとしても、2人に1人はサボタージュする（全く書かない）でしょう。自然な啓蒙に任せているだけでは、おそらく全社的な採用には至らないと思います。あるいは上長などに提案したとしても、（その提案先が高度なリテラシーを持っていない可能性も高いので）承認されづらいでしょう。とはいえ、数百人程度なら「高度なリテラシーを持った人が多数派」となる可能性もある（何ならそういう人材のみを集めることが出来る段階）ので、全社的採用がとてつもなく難しいというほど絶望的ではないです。

ちなみに、権威ボリュームはワークウェアごとに決まっていると書きましたが、因子としては時代水準もあります。たとえば2015年の段階で、リモートベースでザツダンを提案してもほとんど浸透しなかったでしょう。リモートで話す技術や経験が少ない時代だったからです。一方で、現在2024年では、サイボウズに限らず通用する組織は多いと思います。交換日記式1on1についても、もう少し大衆の「書くリテラシー」が向上したら（あるいは何らかのテクノロジーがそこを補完してくれたら）当たり前に通用するかもしれません。

組織内、特に全社など広めに展開する場合は、この権威ボリュームを意識したいところです。ボリュームの小さなワークウェアを無理に広めようとしても反発を生みます。あるいは広められたとしても形骸化するだけです（※1）。一般的には、口コミ的にどこまで広がるかという形で自然な経過を観察するスタイルが良いとされます。

- ※
    - 1 組織規模が大きくなってくるとそうも言ってられません。効率的にガバナンスを利かせるためにも、（リテラシーがあまり高くない）多数派に寄せたワークウェアを無理やり適用することがあります。いわゆる「ハイブリッドワーク」の「週2出社ルール」などはまさにそうでしょう。これは多数派にとって出社が（面倒くさくはあるかもしれないが）かんたんなことだからです。逆に「出社は月3回までルール」のような適用は行われません（リモートワークに強い社員が多数派の会社ではありえます）。いずれにせよ、ボリュームを無視した展開は、大きな組織ではよくあることです。

## Lv4 コミュニティ・ワークウェア
特定の界隈で当たり前に使われていたり、少なくとも認知されていたりする段階のワークウェアです。例を挙げます。

- GTD(Getting Things Done)
    - 日々のタスクから人生の目標や哲学との整合まで網羅できる重厚な人生管理メソッドです
    - [有償のメソッドですが、20 年以上の歴史](ref#7)があり、国内でブームになったこともあります。聞いたことや見たことがある方も多いのではないでしょうか
    - 日本にも公認ディストリビューターが存在します →[GTD Japan](https://gtd-japan.jp/)
- [バレットジャーナル](ref#8)
    - ノート術と銘打たれていますが、人生管理メソッドとしてのポテンシャルもあります
    - GTD を含め様々な仕事術が上手く取り入れられており、アナログな手帳ベースで運用できるよう整備されています
    - SNS映えとの相性が良く（国内だけでも割と）投稿が活発です
        - [#バレットジャーナル - Instagram | ハッシュタグ](https://www.instagram.com/explore/tags/%E3%83%90%E3%83%AC%E3%83%83%E3%83%88%E3%82%B8%E3%83%A3%E3%83%BC%E3%83%8A%E3%83%AB/)
        - [#バレットジャーナル - 検索 / X](https://twitter.com/search?q=%23%E3%83%90%E3%83%AC%E3%83%83%E3%83%88%E3%82%B8%E3%83%A3%E3%83%BC%E3%83%8A%E3%83%AB&f=live)
- SRE
    - サイト・リライアビリティ・エンジニアリングの略で、IT システムの信頼性を（ソフトウェアエンジニアリングの力で）底上げしようという潮流または工学です
    - Google が 2003 年に提唱したものですが、2024 年現在では（SRE を担当する SRE エンジニアは）IT エンジニアの一種として定着しており、エンジニアであれば知らないと恥ずかしいレベルにまでなっていると思います

### 多様制約
コミュニティの厳密な定義はしませんが、レベル3のクラスター（単一組織）よりは多様でなくてはなりません。これを **（コミュニティ・ワークウェアの）多様制約** と呼びます。たとえば複数のクラスターから集まっていたり、異なる文化（地域、界隈、立場や地位、人種など）から集まっていたりしています。

多様制約のないワークウェアは、ただのクラスター・ワークウェアです。

ちなみに、規模（人口）は大きいとは限りません。上記の例で言えば、規模の大きさは筆者感覚では `SRE >> GTD > バレットジャーナル` でしょうか。また、ここでは取り上げませんが、数百人しかいない小さなコミュニティ内で定着しているワークウェア、なんてものもあるでしょう。

### 成立ルート
コミュニティ・ワークウェアの成立経路は主に三通りあります。

一つはインターネットです。わかりやすい例はライフハックでしょう。インターネットのおかげで様々なライフハックがやり取りされてきました。（最近は下火になっていますが）ライフハックという言葉自体は、インターネットを使う者の大半が知っているはずです。これを **インターネット・ベースド・ルート** と呼びます。

もう一つがガバナンスです。仕事術や自己啓発術は主に書籍の形で展開されますが、これは出版社という権威的組織がガバナンスを利かせていると言えます。昔はこれしか手段がなかったため、コミュニティ・ワークウェアは非常に限られていました。もちろん、各人とも自分なりのノウハウを持っていたり、人によっては言語化を（つまりはワークウェア化）していたと思われますが、それを伝える手段が無かったのです。伝えられるのはガバナンスに頼れるごく一部の人達だけでした。

ガバナンスには別のあり方もあります。業界標準とか委員会とか呼ばれるものがそうです。本格的な界隈にはその界隈を取り仕切る存在がいて、界隈全体への啓蒙や周知を行うことがよくあります。その極端な例が国によるもので、たとえば「GIGAスクール（構想）」は国が下ろしてきたコミュニティ・ワークウェアです。教職員でこの言葉を知らない人はいないでしょう。このような経路を **ガバナンス・ベースド・ルート** と呼びます。

そして最後の一つが口伝です。いわゆる口コミや噂が該当しますが、この経路で成立するコミュニティ・ワークウェアは稀です。というのも、たいていはガバナンス・ベースドだからです。ガバナンスが下ろしてきたワークウェアが各人の口伝で（も）広がっている、というだけの話です。権威的組織によるワークウェアだからこそ広がるので、その辺の個人によるワークウェアが口伝で広がることはほぼありません。

例外は、先にも書いた「本格的な界隈」であり、たとえば研究者間、エンジニア間、職人間、教師間あるいはアスリート間とかいった形でワークウェアが伝わっていき、（コミュニティ・ワークウェアと呼べるレベルにまで）広がることはあります。

ワークウェアとは概念的道具であり、一般人には馴染みにくいものです。雑で瞬発的なやり取りに終始しがちな口頭だとなおさらでしょう。権威がなければ誰も相手にしません(※1)。一方で、本格的な界隈に居る人――いわば専門家やプロフェッショナルな人達は、そのようなものを扱える能力を持っています。そもそも自身の仕事に関するワークウェアであれば、仕事にも直結するため扱ってやろうという気にもなります。言わば口伝経路はプロフェッショナルだからこそ成立する経路です。このような経路は **マウス・ベースド・ルート** と呼びます。マウスとは口のことです。

- ※
    - 1 言及するワークウェアは例外的で、要するにただの名前ですから一般人でも馴染めます。典型例は若者言葉です（そしてそれをメシの種にしたいメディアがキャッチーに取り上げる――というのは言わずと知れた話です）。その若者言葉も、近年は SNS を見てもわかるとおりインターネット・ベースドだと思います。

## Lv5 スタンダード・ワークウェア
標準と呼ばれるレベルで浸透している、あるいは広く知られている段階のワークウェアです。いくつか例を挙げます。

- 心理的安全性
    - チームメンバーに共有される信念で、「チームの中で対人関係におけるリスクをとっても大丈夫だ」というもの
    - [大元は 1999 年の論文](ref#5)ですが、[2015 年には Google による啓蒙](ref#6)が開始され、今ではビジネスにおいて広く知られています
    - 学術的に研究され、実践的にも裏打ちされ、今では知名度もあり、と権威性は総合的に高いです
- 1on1
    - 主に上司と部下で行う1対1のミーティング
    - 発祥はシリコンバレーで、国内では 2012 年ヤフーの導入をきっかけに広まったとされています
    - クラスター・ワークウェアの項でも述べたとおり、多数のアレンジがありますが、1on1自体は今やスタンダードレベルと言えるほど知られていると思います
    - 啓蒙と実践が進んでいるという点で権威性が高いと言えます
- TODOリスト
    - まわすワークウェアの項でも例として挙げましたので、解説は割愛します
    - 解説するまでもなく、非常によく知られているワークウェアだと思います
    - [起源はわかりませんが](ref#11)、現在、知名度の点で見れば権威性は非常に高いです
        - 一方で実用性については、現在では懐疑的な声が強く、様々なアレンジやツールが乱立しているのは周知の通りです
- チームワーク
    - チームという集団単位で働くこと
    - これも改めて解説するまでもないワークウェアでしょう
    - チームの定義、あり方や運用は千差万別（かつ各社各組織の武器であり文化）ですが、チームという概念の有用性自体は現代では疑いようがないと思います
    - 権威性は非常に高く、現代のビジネスでは信仰と呼んでも差し支えないレベルで浸透していると思います
        - たとえば「チームプレイできません」と言う転職者はまず転職できないでしょう
- 窓際族
    - 出世コースから外れて閑職に追いやられた社員
    - 1970年代の言葉とされていますが、「閑職」「出世しない」「暇」といった特徴を備えた社員のあり方を指す言葉として、現代でも広く知られている概念だと思います。
    - 「社内ニート」「働かないおじさん」「静かな退職」など類似の言葉が複数存在しますが、共通点はメディアにおける話題性が高いことです
    - 権威性は知名度の観点では高いと言えますが、学術的にはほとんど扱われていません。また実用性についても筆者の知る限りは皆無だと思います（※1）。主にメディアがインプレッション（閲覧数など注目）を集めるための「言及するワークウェア」と言えるでしょう

見ての通り、一口にスタンダード・ワークウェアと言っても、性質はそれぞれです。心理的安全性のように比較的最近出てきたものもあれば、TODOリストやチームワークのように昔から知られているものもあります。そうかと思えば、窓際族のように単にメディアの便宜のために生み出され、広められた「言及するワークウェア」もあります。

- ※
    - 1 一社員の立場で、ワークライフバランスを担保する手段としては使える、と着眼する個人は見かけます。特に「静かな退職」はその傾向を如実に示した言葉だと思います。最後に、仮に実用性を真面目に考えたら、どうなるでしょうか。今少し考えてみましたが、筆者の頭からは「社員全員というプールと社内の仕事全部というプールを考えて、両者の動的なマッチングを支援する」「ティール組織では動的に仕事をマッチングできる仕組みがある」「いわゆるジョブ型もこのような仕組みを後押しする上で重宝するかもしれない」「[Liquid Career](https://scrapbox.io/workware/Liquid_Career) の概念も使えそう」といったヒントが出ました。

### スタンダード・スケルトン
基本的にスタンダード・ワークウェアの利用レベルは低くなります。言及するワークウェアか、心がけるワークウェアになります。まわすワークウェアは滅多にありませんし、筆者も思いつきません。

よくあるのは以下のあり方です。

- スタンダード・ワークウェア自体は低レベル(レベル1 or 2)
- それを踏まえて、じゃあ具体的にどうまわしていくかを各組織各界隈がつくりあげる（つまりレベル3のクラスターあるいはレベル4のコミュニティ・ワークウェアとしてつくる）

1on1にしても心理的安全性にしてもそうです。組織や界隈が違えば、取り組みも違ってきます。「1on1≒雑談」の組織もあれば「1on1≒キャリアの話」や「1on1≒仕事上の相談」な組織もありますし、「心理的安全性≒雑談を増やそう」もあれば「心理的安全性≒ビジョン第一にすれば利己性が減るので健全になる」もあります。

このような現象を **スタンダード・スケルトン** と呼びます。スタンダード・ワークウェアはあくまで骨にすぎず、どのように肉付けするかが各組織各界隈で異なってくるのです。

### 疑わしい権威
一方、窓際族のように、特定の組織や界隈が利益を得るために啓蒙された「言及するワークウェア」もあります。このようなワークウェアは、共感や議論は生みますし、広がりやすいため知名度も上がりやすいですが、一方で学術性や実用性はおざなりになりがちです。

権威性が高いと言えば高いのですが、言わば知名度や話題性に振り切っています。このような権威性のあり方を **疑わしい権威(Suspect Authority)** と呼びます。

スタンダード・ワークウェアは、一見するとスタンダードだから価値がありそうと捉えがちですが、疑わしい権威でしかないものもよくあります。一方で、共感しやすく、人を集めやすいため、上手く使えばビジネスに繋げやすいです。メディアのようにインプレッションを集めるのはもちろん、「共感の先の需要」、潜在的なニーズを満たしたり課題を解消したりできる余地もあるでしょう。

# === Chapter-4 ユースケース ===

本章ではユースケースと称して、どのようなときにワークウェアが役立つか――特にどのようなときにどんなワークウェアをつくってどう役立たせるかの例を示します。

# 利用レベルから考える
利用レベルとして以下がありました。

- Lv1: 言及するワークウェア
- Lv2: 心がけるワークウェア
- Lv3: まわすワークウェア
- Lv4: 想像するワークウェア

どういうときにどのレベルのワークウェアを使えばいいかを述べていきます。

## 共感したい・させたい
言及するワークウェアが使えます。

特定の部分を上手く切り取って、上手い名前をつけることで容易に共感を誘うことができます。Twitter など SNS でもよく見られる光景です。

## 議論したい
言及するワークウェアと想像するワークウェアが使えます。

名前をつけて言及することは議論の出発点です。ここは合っている、そこは違う、このあたりがまだ捉えられていない、といったように、言及するワークウェアという「対象」をつくることで、その対象をベースに話を広げていくことができます。当たり前のことを言っているように聞こえますが、意外と難しいことです。名前をつけずに曖昧に語ることはかんたんですが、対象が曖昧なので議論もしづらいです。一方、言及するワークウェアをつくるのは（適当に語るよりは）むずかしいですが、名前がつく分、対象がしっかりと定まります。

より大胆に言えば、**名前とは仮説です(Name is hypothesis)**。名前をつける（言及するワークウェアをつくる）ことは、仮説をつくることに等しいのです。この仮説を揉むというわけです。もちろん、きちんとした検証ではないですが、それでも意味はあります、その前段階の思考や検討を十分に進めることができます。カジュアルな検証と呼ぶのでしたね。

議論についてはもう一つ、想像するワークウェアも使えます。こちらは名前よりももう少ししっかりとつくりこまれた仮説と言えます。情報量が多く、インプットの時間はかかります（1つの想像するワークウェアをひととおり理解するだけでも10分以上かかることはざらにあります）が、その分、揉みがいもあります。

想像するワークウェアは複数のワークウェアから構成されます（しなくてもいいですがした方がワークウェア・エンジニアリングっぽいです＝やりやすいです）が、「こっちはたぶんこういうワークウェアにした方がいい」「今はAというワークウェアがあるけど、BとCに細分化した方がいいんじゃないか？」といった形の議論ができます。ワークウェアを語彙としたコミュニケーションが行えるのです。ワークウェア・ベースド・コミュニケーションと呼ぶのでしたね。

このように言及にせよ想像にせよ、ワークウェアという単位をベースにすることで議論が進みます。

## 自己開示したい
言及するワークウェアが使えます。

現代はチームワークの時代であり、またプロジェクトエコノミーなど対人的に流動的な時代でもあります。信頼関係をいかにして構築していくかは課題ですが、自己開示はよく知られた手法でしょう。

この自己開示にもワークウェアが使えます。要は、自分のことを端的に表現できる名前を考えればいいのです。名前を伝え、その意味も伝えることで、名前というシンボルと、意味という中身が同時に伝わります。これは単に雑に中身を伝えるだけよりもはるかに情報量が多く、また覚えやすいことです（現実的なこと言うと、名前のセンス次第ではあるのですが）。また、長々とした内容も、上手い名前に集約できれば、伝えるのもラクチンです。

たとえば筆者は自分を「リアクティブ陰キャ」と称することがあります。陰キャというネットスラングがあるので、使える文脈は限られるのですが、これは「話しかけられたら答える（受身・反応的）」「自分からは話しかけられない」点を強調した陰キャを指します。筆者のワークウェアです。思い当たる人も多いですし、リアクティブというネーミングも（エンジニアや一部のビジネスマンなどカタカナに馴染んでる界隈では）イメージしやすいため伝わりやすいです。リアクティブ陰キャです、の一言で結構な情報量を与えることができますし、中々忘れにくい言葉ではないでしょうか。

## 反省したい
言及するワークウェア、心がけるワークウェア、まわすワークウェアが使えます。

成長や改善には反省が必要です。というと仰々しいですが、仕事やキャリア上の真面目なものはもちろん、日常生活をちょっと楽にしたいとか、個人的に直したいところがあるのをなんとかしたいとかいった身近な場面も当てはまります。

さて、反省とは何でしょうか。細かい議論をするつもりはありませんが、自覚と対策と運用だと思います。まず自覚については、言及するワークウェアをつくって（つまり名前をつけることで）捉えれば可能です。対策については、「次からは～～に気をつける」といったような心がけを掲げることになると思います、つまりは心がけるワークウェアです。ただし、心がけただけでは行動が難しいのでしたね（応用と呼びました）。具体的に行動に起こしやすいようなやり方なり仕組みなりを考えた方が良いでしょう、これはまわすワークウェアの形でつくれます。

筆者の例で言えば、「会議をよくすっぽかすのでなんとかしたい」と思っていました。反省もしています。が、ただ頭の片隅に置いておくだけまた繰り返してしまうでしょう(※1)。筆者は次のように捉えました。

- 自覚
    - 『シングルタスク脳』
    - 一度に一つのことしかできない性質。おそらく脳の特性
    - 作業Aに集中している間に会議Xが来てしまうと、当然Xは意識されない → 忘れる
    - Xが来る前に集中をやめるか、Xが近づいていると強く割り込むなどの必要がありそう
- 対策
    - 『リマインド』
    - Xが近づいてきたら強く割り込みをする……これを行えるツールがちょうどある
    - リマインダーである
- 運用
    - 『ルーチンタスク』
        - (定期的に実行されるタスク、という概念のことをルーチンタスクと呼びます)
    - 会社スマホに以下を書いたラベルを貼り付ける
        - 「今日の予定を眺めて、各予定の開始時刻-10分の時刻で会社スマホ(iPhone)のリマインダーをセットせよ」

これで筆者は会議のすっぽかしがなくなりました。毎日就業時に会社スマホを見たときに「ああ、今日の会議の分のリマインダーをセットせねば」と気付けるからです。

こう見ると大げさに見えますが、慣れればどうということはありません。何よりワークウェアとして一つずつ確実にとらえているため、確実に前進していくことができます。

- ※
    - 1 中には「いやそれくらい当たり前に気をつけられるでしょ」と思われる方もいらっしゃるかもしれません or それが多数派かもしれません。が、当たり前に気をつけるためには、適切な場面で適切に想起して行動に起こすことが要求されます。ワークウェア・エンジニアリングでは「応用」と呼びますが、これは人によっては難しいことです。また同じ人でも、分野によっては（特に初心者・初学者となる領域では）難しいです。ワークウェアをうまくつくれば・使えば、この困難にも対応していけます。

## 自分を変えたい
言及するワークウェアが使えます。

世の中には小手先の対応ではどうにもならないことがあります。性格は変わらないと言いますが、そのレベルの何かを変えないとどうにもならない場面もあるかと思います。そもそもそういった自覚さえできないこともあります。その結果、漠然と「自分を変えたい」といった欲求や希望が表出してきます。

どうすれば自分を変えられるでしょうか。有名なのは[人間が変わる方法は3つしかない](https://news.yahoo.co.jp/expert/articles/81c903a87ad39ef7f3fcbb70acec4da49b34712c) の格言でしょう。曰く、時間配分と住む場所と付き合う人を変える、とのことですが、これは要するに環境を変えるということです。

では、それなりに問題ない環境で、でももう少しどうにかしたい、といった場合はどうしたらいいでしょうか。ギャンブルのつもりで新しい環境に飛び込むしかないのでしょうか。あるいは[計画的偶発性理論](ref#12)が言うように探索的行動を増やす（元はキャリアの文脈ですが筆者はもっと一般的に「チャンスと出会える確率を上げる行動をするのが良い」と理解しています）のが良いでしょうか。

実はワークウェア・エンジニアリングも使えます。自分を変えるためのワークウェアをつくって使えばいいのです……と言っても、奥が深いので、一つだけ述べます。スタンスを変えます。

- 1 自分が今どういうスタンス（状況や立場や価値観や文化 etc）にいるかを自覚する
- 2 別のスタンスを知る
- 3 新しいスタンスを採用する

このスタンスは、まさに言及するワークウェアで捉えることができます。

前項でも「シングルタスク脳」という例を出しました。別の例を上げると、私達が普段当たり前に浸っている日本の企業文化も捉えることができます。自分でワークウェアをつくってもいいですが、大変なので、まずは既存のものを使うといいでしょう。[異文化理解力](ref#13)はおすすめです（グローバルなビジネスマン向けですが日本で過ごす人が日本の文化を自覚するのにも重宝します）。

このようにして捉えて自覚できたら、次の行動にも繋げられます。特に「他にどんなスタンスがあるだろう」という形で思考できます。『自分を変えたい』とは何とも漠然で意味不明な難題ですが、スタンスというワークウェアをベースにすれば、もう少し堅実に進んでいけるのです。

## 定着させたい
言及するワークウェアやまわすワークウェアが使えます。

新しく知ったり学んだりした知識やこれから始めたい習慣など、新たに定着させたい事物はよくあります。これらもワークウェアでカバーできます。

まず知識については、言及するワークウェアが使えます。よく「自分の言葉でノートに書く」といった勉強法を聞きますが、似たようなものです。特に自分にとって馴染みのある言葉を結びつける（たとえる、こじつける、関連付けるなど）と、思い出しやすくなります。自分で考えたのだ・自分がつくったのだという愛着もあります。

もちろん、その分野の知識を正確に理解し覚えねばならないシチュエーションはよくありますが、何もそれだけではありません。ちょっとした雑談や情報収集、あるいは仕事を多く抱えるマネージャーなど「正直勉強まではしたくないけどざっくりとは理解しておきたい」「何もしないと忘れちゃうのはもったいない、でも勉強まではしたくないなー」といった塩梅はあるはずです。そんなときに、さっさとワークウェアをつくったり、あるいは既存のワークウェアと結びつけてしまえばいいのです。自分用ですから、自分にとってわかりさえすればいいです。極論「プログラミングの第一カテゴリー」「吉良野すたの対人四分類」など、他人が聞いても意味がわからないネーミングでも構いません。特に「この辺とこの辺は同じカテゴリだよなー」と感覚的にくくって、そこに適当な名前をつけていくやり方は（他者には通じないですが）非常に便利です。余談ですが、このように自分の、自分による、自分のためだけのワークウェアをつくる営みを **オブフィング(OBFing: Of me, By me, For me)** と呼びます。

次に習慣については、主にまわすワークウェアが使えます。心がけるだけでは中々行動に繋がらないので、具体的な手順をつくることと、それらにいかにして従わせるかという仕掛けをつくることが必要なのですが、ここで様々なワークウェアが使えます。

前項では筆者の会議すっぽかし対策を取り上げましたが、「シングルタスク脳」「リマインド」「ルーチンタスク」といったワークウェアを使いました。特に、毎日朝一に使う会社スマホに付箋を貼り付けるという形で「忘れずに目に入れる」ことを担保し、結果、今では毎朝今日のすべての予定に対するリマインダーをセットする習慣が身につきました。

ちなみに、このあり方を一般化すると[動線](https://scrapbox.io/workware/%E5%8B%95%E7%B7%9A)というワークウェアで説明できます。毎日通りがかる場所（動線）に何か置いておくと、毎日通るだけに高確率で気付けるはずだ、というものです。動線というワークウェアを意識することで「私にはどんな動線があるのか」「各動線を通る頻度やタイミングはどのくらいか」といった検討ができます。さっきからベースベースとしつこいですが、（ここでは動線という）ワークウェアをベースにして検討を進められる、もっと言えば前進していけるのです。

## 改善したい
どのワークウェアも使えます。

何かと慌ただしい昨今、業務改善や新規事業について考える（ないしは考えさせられている）方も多いのではないでしょうか。そうでなくとも、個人的な事柄についてもあれやこれを改善したいと考えることもあると思います。

このような改善にもワークウェアが使えます。

現状を捉えるには、言及するワークウェアが使えるでしょう。捉え方が不正確だったり間違っていたりすることはもちろんありますが、それは実際に名前をつけて捉えたからこそわかったことです。ここができないと、もやもやしたまま不満を抱えるだけになってしまいます。あるいは愚痴の形で吐き出すのに終始します。生産性はありません。一歩踏み込んで、言及するワークウェアをつくってみる（自分なりに特定の部分を切り取って名前をつける）ことが第一歩です。

どう改善していくかの検討と実践には、心がけるワークウェアやまわすワークウェアが使えます。これも言及と同樣、ワークウェアという形でラフな仮説をつくり、これに従って行動を行うことで情報を得ていきます。自分でつくってもいいですし、仕事術やライフハックなど既存のものを使ってもいいでしょう。あるいは他者に相談して比較したり、あたりをつけてもらうこともできます。

意外と重要なのが、心がけるワークウェアです。特に「この改善とどう向き合うか」という向き合い方や「なぜ改善したいのか」「しなきゃいけないのか」といった背景は、ぜひとも言語化しておきたいです。よくビジョンやマインドは大事と言いますが、私達は人間であり、本質的に怠け者なので、納得感が無いと動けません。要は納得感が得られるような、あるいは納得できないと判断するに足るような心がけるワークウェアをつくりたいです。自分はこういうことを心がけたい人間なのだとか、この組織やこの仕事ではこういう心がけが求められているのだとかいったことを「心がけるワークウェア」の形で捉えるのです。前者と後者を比べれば、優柔不断を抑えられます。さっさと行動したり、逆にさっさとやめたり、これだけは我慢してみるかといった判断もしやすいでしょう。

# === Chapter-5 原理・原則 ===

本章ではワークウェア・エンジニアリング（WWE）の原理、原則、哲学や思想といったものを扱っていきます。

# 権威からの解放

## 権威
WWE においては、他者を信頼させる威力――特にそれを強く備えたものを指します。

## 権威性
権威の強さを権威性と呼びます。

## 権威性の3P
権威性は以下からなります。

- 学術性(Proof)
- 実用性(Practicality)
- 知名度(Popularity)

学術性は、科学的に担保される度合いです。研究や論文によって示されます。

実用性は、実践的に担保される度合いです。実践や道具によって示されます。

知名度は、認知的に担保される度合いです。啓蒙や宣伝によって示されます。

## 権威からの解放
WWE では権威性を絶対視しません。個人による根拠のない思いつきを許容します。そうしてつくられたワークウェアもまた権威にとらわれずに扱います――たとえば批評したり、試したり、議論したり、一部を参考にして新たなワークウェアをつくったりします。これを  **権威からの解放(FFA, Freedom From Authority)** と呼びます。

権威からの解放は、権威に全く頼らないことは意味しません。むしろ権威は重要ですし、最終的には必要なことです。WWE は権威に頼る前段階にフォーカスしています。勘や直感や感性の重要性が語られることがありますが、WWE も同樣です。個人個人のそれらをもっと尊重して、各人が自分なりにワークウェアを扱っていくことを意図しています。そうすることで相当な情報が発散します。発散した情報からは本質を蒸留できることがあります。こうして本質を手繰り寄せていき、ある程度のところまで見えてから権威に頼るのです。

私達は誰もが個性的な存在であり、個性に最も広くかつ深くリーチできるのは自分自身だけです。WWE は権威からの解放によって、これを後押しします。WWE は個人的かつ創造的な営みです。権威にとらわれているだけでは見えてこない事物を引き寄せることができますし、各人とも自分なりに関わることで当事者性（自分事か）を養っていきます。当事者性は主体性を構成するものです。つまり権威からの解放によって、各人は主体的にもなれます。

# ワークウェア・ベースド・コミュニケーション

## ワークウェア・ベースド・コミュニケーション
ワークウェアを用いたコミュニケーションを指します。

WWE ではワークウェアの形で言語化を行い、これを用いてコミュニケーションを行います。アウトプットをレビューするスタイルはクリエイターにはよく知られていますが、同樣です。アウトプット（成果物）というほどしっかりつくりこむ必要はありませんし、レビュー（批評）というほど入念に、あるいは形式的に評価する必要もありません。日常的なコミュニケーションの延長程度でも問題ありません。ただ、ワークウェアという語彙を使う点が違うだけです。

## カジュアルな検証
ワークウェア・ベースド・コミュニケーションにより、どうでもいいワークウェアから良さそうなワークウェア、あるいはまだワークウェア化できていないが重要そうな本質といったものが見えてきます。このような雑な営みを検証と呼ぶことには抵抗があるかもしれませんが、WWE では権威からの解放があります。検証です。とはいえ、検証という言葉を堂々と使うと語弊もありますから、**カジュアルな検証** と呼びます。

## 本質
カジュアルな検証は、カジュアルゆえに明確なゴールもありません。適当なところで中断してください。どこまで続けるかはワークウェア・エンジニアの腕の見せ所ですが、権威の誘惑に負けず、現時点でひねり出しきれるまでひねり出すのが理想です。またアイデアがそうであるように、寝かせることも有効です。この性質上、カジュアルな検証は瞬発的なものではなく、少なくとも短期的、できれば中長期的な営みであるべきです。そのためには、カジュアルの名のとおり、制約をできるだけ減らすことが求められます。計画やタスクは典型的なアンチパターンです。

一つの目安は **本質(Essence)** です。たまに（あるいは稀に）「これだ！」と見えてくることがあります。それが本質です。一見すると、権威から解放されたカジュアルな検証ゆえに無価値に思えるかもしれませんが、そうでもありません。権威から解放し、カジュアルに検証しているとはいえ、私達はすでに多くの権威に触れており、その影響を受けています。明らかなインプット不足を除けば、心配は要りません。そもそも権威の担保は、後から行えば良いのです。WWE で行うことではありません。

ともかく、本質が単一あるいは複数見えてくると、次第に限界が見えてきます。「これで良さそう」「これ以上は出そうにない」といった段階に入ります。これがカジュアルな検証のゴールとなる一目安です。ちなみに、本質が一つもないのにそうなった場合、あるいはもう少し見出せそうに感じる場合は、頑張って本質を見つけてください。発散と収束でいう収束のような営みです。

## 蒸留
アイデア出しや知的生産の文脈では発散と収束がよく使われます。WWE も似ていますが、少し違います。

WWE では **発散と蒸留(Spread and Distill)** と呼びます。収束ではなく蒸留です。収束とは発散されたものを律儀にすべて回収して何かしら生かすニュアンスですが、蒸留はもっとカジュアルです。すべて生かさなくても構いませんし、発散されたものの中にはなかったもの（たとえばその場の思いつき）を取り入れても構いません。発散されたものは「使ってもいいヒント」にすぎず、また使ってもいいヒントはそれらだけではないのです。何を使ってもいいし、使わなくてもいい。強い言い方をすると、蒸留とは手段を選ばずひねり出すということです。**ダシにする** と表現することもあります。

# チームワークとソロワーク

## 仕事
WWE における **仕事(Work)** とは、所属企業や顧客から給料を得て行うビジネスワークのみを指すものではありません。日常生活上の家事、趣味やボランティアの活動といったパーソナルワークも含みます。

WWE はビジネスワークにおける活用も目指しているので、WWE において仕事と言及したときに、主にビジネスワークが想定されていることは多いです。それでもパーソナルワークを排除しているわけではないことに注意してください。

そもそもビジネスとパーソナルが両輪です。パーソナルが上手くいくからこそビジネスも上手く行えるのですし、ビジネスが上手くいくからこそパーソナルも上手くしていけます。WWE ではビジネスとパーソナルを区別せず、どちらも扱います。大胆に言えば、業務時間中にパーソナル側のワークウェアをつくることさえ想定します。逆に業務外の私生活中にビジネス側のワークウェアをつくることもありえます。現状ではルール上、許容されないことが多いでしょうが、それでも WWE 自体はビジネスとパーソナルをいたずらに区切るものではありませんし、どちらかのみを想定するものでもありません。

## チームワーク
仕事は基本的にチームで行うものであり、したがってチームワークが要求されます。チームワークとは本番試合のようなものです。場や空気や秩序といったものがあり、常にそれらに従うことが求められます。従わない者は排斥されます。WWE では **チームは試合(Team is Game)** と表現します。

フリーランスやひとりプロジェクト、あるいは日常的な文脈で言えば一人暮らしなど、一人で仕事を行う場合もチームです。メンバーが一人であるというだけで、チームであることに変わりはありません。チームという言葉でピンと来ないなら、プロジェクトという言葉で捉えても構いませんが、WWE ではチームという言葉を使います（WWE における「チーム」は「メンバーがひとりのチーム」も想定します）。

## ソロワーク
本番試合に対して練習やトレーニングといったものも存在します。チームワークに対しても存在し、これを **ソロワーク** と呼びます。

ソロワークは極めて個人的な営みであり、仕事（ビジネスワークもパーソナルワークも）を上手くやるための個人的な活動全般を指します。スキルを磨いたり、道具を新調したり、将来について考えたり、モヤモヤを吐き出して言語化したり、日々のタスク管理や体調管理を行ったりなど、あらゆるものを含みます。

## 兵隊
チームワークにのみ従事するのは、練習やトレーニングをせず本番試合ばかりしているようなものです。試合を消化することはできますが、それだけです。これを **兵隊(Soldier)** と呼びます。

兵隊的なあり方が求められる仕事やシチュエーションはありますが、弊害も多いです。常に兵隊だと息切れしますし、私生活もおざなりになりがちです。また VUCA な時代に対応しづらくなり、中長期的には首が絞まります。そして、特に経営者や管理者など労働者を使う側としては、従順な兵隊の方が都合が良いですから、兵隊は純粋に使い潰されやすい存在でもあります。このような兵隊的なあり方を食い止めるには、本番試合以外の営み、つまりはソロワークが必要です。

## シナジー
ソロワーク自体は WWE とは限りません。自己完結しているからです。たとえば他者に通じない観念を持っているだけでもソロワークは可能です。

WWE ではもう一歩踏み込みます。ワークウェア（概念）をつくって、これをチームに持ち込むのです。そうしてワークウェア・ベースド・コミュニケーションを行うことで、ひとりひとりの強さや多様な視点を取り入れていくことができます。結果的にチームも強くなります。これを WWE では **シナジー(Synergy)** と呼びます。

## 孤独と余裕
ソロワークには孤独が必要です。孤独とは誰にも何にも邪魔されず一人でいる様を指しますが、ソロワークでは個人的な事柄を扱いますし、発散と蒸留を行うためには自分のペースの尊重や集中の担保も求められます。このようなことは外部の邪魔があっては行えません。孤独でなくてはなりません。別の言い方をすると、ひとりで過ごせるだけの力――メンタルモデルやスキルやスタンスといったものが要求されます。クリエイターや職人、あるいは私情を安易に見せられない立場の方には慣れた営為ですが、一般的にはそうではありません。WWE では孤独に慣れるための方法論（どうしても適応できない方向けの代替案含む）も扱います。

また、孤独には **スラック（余裕, Slack）** が必要です。たとえば毎日一時間以上の、ひとりの時間を確保することができるでしょうか。そのためには物理的な一時間の確保はもちろん、その一時間の最中に邪魔されないことや、その前後（特に前）で疲弊しすぎていないことなども必要です。スラックは意識的に確保しなければなりません。特にチームワーク至上では個人のスラックよりもチームで過ごすことが重視され、スラックを確保しづらいきらいがあります。兵隊の世界観でもスラックでは意図的に潰されがちです。他にもスマホや SNS 等により日々注意を奪われている（スラックを食い潰されている）ことは周知のとおりです。スラックは、私達が思っている以上に意識的に確保していかねばなりません。WWE でもスラックの扱い方は重視します。
