# 所感<a href="sta.md"><img src="https://gyazo.com/a0a22d2fc5cf4fb2525db091fb66594b.png" alt="sta" width="16"/></a>
- AoR(Area of Responsibility)が画期的である
    - 私は[タスクソース]として蒸留させていただきましたが、タスクとしては処理せず「基本的に眺めるだけ」で「そこから必要に応じてタスクを生み出す」ようなものがあるのです。この存在を改めて突きつけてくれたのは、私にとってはAoRでした<a href="sta.md"><img src="https://gyazo.com/a0a22d2fc5cf4fb2525db091fb66594b.png" alt="sta" width="16"/></a>
🌎️[タスクソース｜タスク管理を噛み砕く](https://zenn.dev/sta/books/taskmanagement-kamikudaku/viewer/source)
- ゴールの概念がある
    - 達成したい事柄で、プロジェクトに紐づける
<img src="https://gyazo.com/32ff3cbe229c5adae04eda78aae7c5d0.png" />

- 哲学も包含されている
    - 「プロジェクトなきゴールは夢でしかない」「ゴールなきプロジェクトは趣味でしかない」
    - 無理に[SSoT(信頼できる唯一の情報源)]にせず、複数のツールを使ってもいいが、必ずPARAの構造を徹底する
    - サブ階層は4段まで。これ以上は人間工学的に望ましくない。他のツールも4段を採用している
        - 例1: evernote
            - application
                - stack
                    - notebook
                        - note
        - 例2: OneNote
            - notebook
                - section
                    - section group
                        - page
    - etc
