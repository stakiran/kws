# Transparent Hotline
# コンプライアンス・ホットラインの透明性を確保するという考え方、またそのようなホットラインのあり方。
# 特に通報内容とその対処内容および傾向を可視化するもの。

# 背景
- ホットラインの現状
    - [ホットライン]ソリューションの類型

| Type | 通報窓口の委託 | 通報内容調査の委託 | 通報者への進捗・結果報告 | 例 |
| --- | --- | --- | --- | --- |
| 1 | Y | n | Y | NAVEX |
| 2 | Y | Y | n | リスクホットライン® |
| 3 | Y | n | n | DQヘルプライン |

        - 通報窓口。自社内ではなく第三者機関に設けて、そこに通報されたものを自社に届けてもらう
        - 通報調査。通報内容を自社で調べるのか、第三者機関が調べるのか、第三者機関主導で自社にも協力が飛んでくるのかの3パターン
        - フォローアップ。通報者本人への進捗や結果の共有
    - 通報経路
        - 内部通報。自社社員が自社を通報
        - 外部通報。A社以外の者がA社を通報
- いずれにせよ透明性がなく、通報が実際どのように対処されているかが不明である

# 概要
- ケース:
    - 一組の「通報内容とその対処内容」のこと
    - Transparent Hotlineとは、ケースの透明性とも言える
- 透明にするには？
    - 個別。ケースをマスク化した上で公開する
    - 全体。ケースの統計や傾向を公開する
    - 論理。個別や全体のロジック、アルゴリズム、その他仕組みの解説を公開する
- 透明レベル:
    - どこまで透明性を出せるかをn段階に分けて格付けする
- ホットラインのジレンマ:
    - 透明性を上げようとケースの内容の公開に踏み込むこととなり、以下が脅かされる
        - 通報者の安全性
        - 通報先企業の評判（レピュテーション） → 企業価値の変動に直結する
    - ジレンマに負けず、脅かされない形での公開を実現するには、を模索するべきである<a href="sta.md"><img src="https://gyazo.com/a0a22d2fc5cf4fb2525db091fb66594b.png" alt="sta" width="16"/></a>

# マスケーション・フラグ
- 概要
    - マスク化フラグのこと
    - ケースごとにつく設定の一つで、デフォルトはオフ
    - オンにすると透明化の対象になる
- 意義
    - 透明化したいケースをコントロールしやすくなる
- 活用例
    - 1: 通報者本人が「Transparent Hotlineの対象にする」にチェックを入れた場合に、オンにする
    - 2: フォローアップ時に、希望すればオンにできる
# <a href="sta.md"><img src="https://gyazo.com/a0a22d2fc5cf4fb2525db091fb66594b.png" alt="sta" width="16"/></a>
- もちろん安全に公開されることの保障も必須である。1:だとやりづらいが、2:だと（マスク化の説明をすれば）納得してもらいやすいだろう。ただし、無知な通報者に付け込む形になるのも防がねばならないという意味では難易度が高い
- 理想を言えば、フラグには頼らず、すべてのケースにもれなく適用できる「汎用的なマスク化」を開発すればいいのだろう。たとえば（通報者はもちろん）通報日時や通報先部門など「特定に繋がる情報」をマスク化する、くらいはすぐに思いつく。無論、マスク化しすぎて、公開する情報自体が薄くなりすぎると意味がない
