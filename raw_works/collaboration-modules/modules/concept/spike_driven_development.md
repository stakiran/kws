# Spike Driven Development
**スパイク駆動開発（Spike Driven Development）** とは、スパイクを繰り返すことで探索を広げていく探索手法である。

## スパイクとは
一言で言えば「ラフな実験単位」である。単位感としては「タスク」と同程度だと考えて良い。しかし、スパイクのニュアンスはタスクとはだいぶ異なったものとなる。

スパイクは以下を満たす:

- CLRC（文脈、ログ、結果、カード）を持つ
    - 文脈: スパイクのID（番号など一意の情報）、タイトル、目的、なぜやるか、何のためにやるか、制約や戦略や依存は何かなど
    - ログ: 作業ログ、中間成果物、収集した情報全般
    - 結果: そのスパイクが示す結論。たとえば仮説に対する検証結果、問いに対する答え、結果に至れず中止に至った旨など
    - カード: 文脈、ログ、結果をカード一枚分で端的に整理したもの
- 再利用性
    - スパイクは言及や利用が可能な単位であり、スパイクをもとに新たなスパイクを行える
- 担当者
    - スパイクの担当者は常に一人であり、分担や分業や途中交代はありえない
- 状態
    - スパイクはタスクと同様、状態（オープンとクローズとディスカード）を持っている
    - 閉じる操作としてクローズとディスカードがあり、どちらも同じだがディスカードだと再利用価値がない旨を付与する
    - 閉じたスパイクは以後進めてはならないが、CLRC の更新は可能である
    - 一度閉じたスパイクを再度オープンすることはできない
- タイムボックス
    - スパイクにはタイムボックスを設定しなければならず、タイムボックス経過後は自動でクローズする

チームがカスタマイズ可能なパラメーターが色々考えられる。細かい部分はカスタマイズすればよい。主な例を挙げる:

- CRLC の形式
    - スパイクの ID 形式やカードの形式も含む
- 特殊なスパイクの定義
    - たとえば「スパイクのリンク集」を表すスパイクを許すか等
- タイムボックスの選択肢
- スパイク駆動開発をいつ打ち切るか、何回繰り返すかといった活動の運用の仕方

## スパイク・ベースド・コミュニケーション
スパイク駆動開発では、チームメンバー各々がスパイクをつくっていくことになる。スパイクは共通言語であり、たとえば「スパイク 23 と 27 により～～と言えるから」といった会話が可能である。この考え方を **スパイク・ベースド・コミュニケーション** と呼ぶ。

したがってスパイクは利用可能な形、少なくとも説明可能な形に整えなくてはならず、表現と説明の責任はスパイク担当者に委ねられる。またスパイクを扱えるツールの整備、メンバーのツール習熟、1-spike 1-page に從って蓄積するトピック指向的な考え方への適応も必須となる。責任を果たさない者やツールを扱えない者、考え方に適応できない者は有害であり、速やかに順応してもらうか、さもなければ追放しなければならない。

## アンチパターン
アンチパターンを行うな、という言い方をする。

### 共有成果物は抱えるな
チームとして何らかの製品または成果物を抱えることはない。なぜならスパイク駆動開発は探索手法であって、製品開発手法ではないからだ。しかし、製品開発につながる活動としては使える。Four-jects でいう Preject に相当する。

スパイク駆動開発における唯一の成果物は、スパイクの集まりである。

### マネジメントを行うな
スパイク駆動開発では、各メンバーは粛々とスパイクをつくっていくだけである。したがってプロジェクトマネジメントその他チームワークでほぼ必ず発生するマネジメントの活動は起こり得ない。スパイク・ベースド・コミュニケーションである。スパイクで語るべきである。

### スパイク駆動で動けないメンバーを許容するな
スパイク駆動開発は、各メンバーに高度な自律性と主体性を要求する。スパイクを読み書きする行動を行えない者は有害であり、このようなメンバーがひとりいるだけで他メンバーが阻害されてしまうため許容してはならない。

オンボーディングはしてもいいし、少なくともコンテンツとしては必要だろうが、端的で良い。既存のスパイクを一通り終える期間（スパイク・キャッチアップ）はスパイクの量と難易度次第だが、スパイク・ベースド・コミュニケーション自体は、1日分（8時間分）もあれば可能なはずである。1日分経ってもできない者はそもそも実力不足だと考えて良い。

### 権限格差をつけるな
たとえば 5 人のチームで、そのうち 1 人のマネージャーだけがスパイク用ツールへのフルコントロールを持っている、といった格差はつけるべきではない。

オーナーや Admin レベルのクリティカルな操作は代表者ひとりで良いが、それ以外のあらゆる操作は全メンバーが行えなくてはならない。もし運用上、不可能な場合は、権限を持つ代表者が責任を持って迅速に応えなければならない。これを **Sarvant Admin** と呼ぶ。たとえば生成 AI と連携するためのエクスポートや外部連携まわりも、各メンバーが自由に行えなければならないし、運用上不可能であるなら、権限を持つ代表者がいつでも迅速に代行しなければならない。

Sarvant Admin のパターンとして以下がある:

- メンバー。Sarvant Admin 自身もメンバーになる。代行活動がある分、スパイクの時間が減るし、それ以上にリクエストにすぐ応える必要があるため集中しづらい
- アウトオブメンバー。Sarvant Admin 自身はスパイク駆動開発を行わない。ただし Read Only としてスパイクを読むことはできる。無論、Sarvant Admin として、リクエストには迅速に応える

Sarvant Admin はスパイク駆動開発のボトルネックになるため、極めて重要である。

### CLRC のわかりづらいメンバーを許容するな
スパイクの表現――CLRC を読んでも理解できないし、指摘しても改善しないケースがある。これではスパイク・ベースド・コミュニケーションが成立しないため、そのようなメンバーは追放しなければならない。

別解として CLRC、特に結果とカードを生成 AI につくらせる方法がある。必ず生成 AI がつくったものを使うようにするのである。これなら人間が表現する必要がないため、要追放メンバーでも引き続き貢献できる可能性がある。無論、このような仕組みの構築や、生成 AI の仕事のクオリティ確保（プロンプトや高級モデルの使用など）といった余計な仕事は増える。これ自体はカスタマイズ・パラメーターとして、事前に整備しておくことになる。スパイクとしてカスタマイズすることは許されない。
