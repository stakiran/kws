# Template Based Communication
**Template Based Communication** とは、送り手がテンプレートを送り、受け手がそれを埋める形で答えるコミュニケーション手法である。つまり一往復分のコミュニケーションを扱う。

テンプレートは限定的な状況を解決するべきである。汎用性が高すぎると具体的な行動に落とし込みにくいし、意思決定のコストもかかるため使いづらい。テンプレートが上手く機能しない場合は、上手く機能させるような具体的な項目が出来ていない可能性が高い。そういうわけで、Template Based Communication では状況に応じたテンプレートを複数つくることになるし、同じテンプレートを改版することもありうる。

Template Based Communication において送信するものは、以下のいずれかだけである:

- 1: テンプレート自体とその説明
- 2: テンプレートを埋めたもの

上記以外の余計な front matter、back matter は一切不要である。たとえばチャットベースで Template Based Communication を行う場合、送り手はメンションと 1 のみを書いて送信するし、受け手は 2 のみを返信するはずである。

## 例
状況としては、マネージャーが部下の状況を負担なく知りたいとする。

例1: ky/ky を部下に周知して日次で書いてもらう。ノート上に markdown コードブロックで書いてもらうとする。

```
- 昨日やったこと
    - ...
    - ...
- 今日やること
    - ...
    - ...
```

例2: ky/ky ではなく以下テンプレートを使う。書いてもらタイミングも ky/ky とは違って、その日の終わりになるだろう。

```
- 今日やったこと
    - ...
    - ...
- 明日やること
    - ...
    - ...
```

例3: 例2と同じテンプレートを使うが、フォームの形で提供して、部下らには毎日フォームから送信してもらう

## Framework Based Communication との違い
Template Based Communication は、あくまで限定的なコミュニケーションを、テンプレートの提供とその記入という一往復でまかなうものである。それ以上の運用にはタッチしない。必要なら Framework Based Communication を使って定めればいい。

もちろん定めなくてもいい。この場合、Template Based Communication にどうやって移行するかが課題となるが、以下のやり方がある:

- 1: 合意方式
    - 送り手が Template Based Communication をやりたい旨を受け手に伝え、受け手が承認したら成立する
- 2: 強制方式
    - 送り手が Template Based Communication でやる旨を伝えた後、すぐにテンプレートも送る
    - 合意コストをなくしたい場合に使えるが、双方が Template Based Communication について知っており、かつ必要に応じて運用することがあることまで知っていることが前提となる。この前提がない場合、受け手は身動きが取れなくなるか、送り手への信頼をなくす（突然ドライな振る舞いをしてきたように見える）
